Section 01

Commvault Architecture

The three-tier model that powers every Commvault deployment

Commvault operates on a three-tier architecture: a central management server, one or more data-movement nodes, and the client machines whose data needs protection. Understanding how these tiers interact is the foundation for every configuration and troubleshooting task a backup administrator will encounter.

Tier 1

CommServe (CS)

The administrative brain of the environment. Coordinates all backup, restore, and management activities. Houses the SQL Server-backed CommServe database. Strongly recommended on Windows Server.

Tier 2

Media Agent (MA)

The data mover. Receives backup data from clients and writes it to storage (disk, tape, or cloud). Communicates results and metadata back to the CommServe. Can run on Windows or Linux.

Tier 3

Client / Agent

Any server whose data is being protected. Can be agent-based (file system, database, etc.) or agentless (VMware vSphere, Hyper-V). The installed package is called an iDataAgent (iDA).

Deployment Models

Commvault supports all major deployment patterns, making it one of the most versatile enterprise backup platforms in the market.

üè¢ On-Premise

All infrastructure resides in the customer's own data center. CommServe, Media Agents, and clients are physical or virtual servers within the corporate network.

‚òÅÔ∏è Cloud-Only

All components are hosted on a cloud provider (Azure, AWS, GCP). The architecture and operations remain identical ‚Äî only the underlying infrastructure changes.

üîÄ Hybrid

Some workloads run on-premise, others in the cloud. Commvault supports unified management of both from a single CommServe.

ü§ñ Agentless Backup

Used for VMware and Hyper-V environments. No agent is installed inside the virtual machine; instead a Virtual Server Agent (VSA) on a proxy server handles data capture.

üìù Trainer Note ‚Äî Agent vs. Agentless

For VMware, the VSA proxy can be a dedicated VM or physical server that has network connectivity to the ESXi hosts. For Hyper-V, the VSA is installed directly on the Hyper-V node. In both cases, no agent is installed inside the guest VM.

Backup Types Reference

  • File System Backup ‚Äî protects files, folders, and directory trees on Windows or Linux servers
  • Database Backup ‚Äî application-aware protection for Oracle, SQL Server, MySQL, PostgreSQL, etc.
  • Virtual Server Backup ‚Äî agentless protection for VMware vSphere and Microsoft Hyper-V environments
  • Cloud Workload Backup ‚Äî native integration with Azure, AWS, and GCP services

Section Summary

  • CommServe = management brain (Tier 1), always Windows in most environments
  • Media Agent = data mover (Tier 2), Windows or Linux
  • Client / Agent = the protected machine (Tier 3), agent-based or agentless
  • Commvault supports on-prem, cloud, and hybrid deployments with identical operational procedures

Lab Environment Setup

Provisioning infrastructure on Azure Cloud and VMware Workstation

Before deploying Commvault software, infrastructure must be provisioned. This section covers two approaches: building a lab environment in Microsoft Azure and building a fully local lab with VMware Workstation Pro. The Commvault installation process is identical on both.

Option A ‚Äî Microsoft Azure Cloud Lab

Azure provides a free trial subscription with $200 USD of credits valid for 30 days ‚Äî sufficient to deploy and practice with a complete Commvault environment at no immediate cost.

Virtual Machine Sizing Guide

CommServe VM

  • OS: Windows Server 2022
  • vCPU: 4 cores
  • RAM: 16 GB minimum
  • Disk: 127 GB (OS) + dedicated data disk recommended
  • Port: RDP 3389 open

Media Agent VM

  • OS: Windows or Linux
  • vCPU: 2 cores
  • RAM: 8 GB
  • Disk: 127 GB
  • Port: RDP 3389 or SSH 22

Client VMs (x2)

  • Windows Client: 2 vCPU / 4 GB / W2019
  • Linux Client: 2 vCPU / 8 GB / Ubuntu 22
  • Disk: 64 GB
  • Ports: RDP 3389 / SSH 22

Azure VM Deployment Checklist

  1. Create Resource Group

    Group all lab VMs under a single resource group (e.g., commvault-lab-rg) for easy management and cleanup.

  2. Deploy VMs in the Same VNet & Subnet

    All VMs must share the same Virtual Network and subnet so they can communicate using private IP addresses.

  3. Set Static Private IP Addresses

    Navigate to: VM ‚Üí Networking ‚Üí Network Interface ‚Üí IP Configuration ‚Üí Change to Static. This prevents IP changes on reboot ‚Äî critical for Commvault hostname/IP resolution.

  4. Disable Windows Firewall (Lab Only)

    For lab simplicity, disable the Windows Defender Firewall on all VMs. In production, configure specific firewall rules instead.

  5. Verify Connectivity

    From each VM, ping the private IPs of all other VMs. Successful ping replies confirm network connectivity is working.

‚ö†Ô∏è Azure Cost Warning

Azure bills per hour of runtime. Always stop VMs when not in use. A CommServe VM (4 vCPU, 16 GB) costs approximately ‚Çπ40/hour. Leaving all VMs running overnight can consume a significant portion of your trial credits.

üìå Private vs. Public IP in Azure

When connecting from your laptop, use the public IP via RDP/SSH. Inside the Azure environment, all VM-to-VM communication uses private IPs only. The public IP is not visible from within the VM (use ipconfig to see only the private IP).

Option B ‚Äî VMware Workstation Pro (Home Lab)

VMware Workstation Pro is free for personal use and available at access.broadcom.com after free registration. Minimum requirement: 16 GB RAM on the host machine.

Key Configuration Steps

  • Select Bridge Network mode ‚Äî this allows VMs to communicate with each other and with your host machine using real IPs on your LAN
  • Install VMware Tools inside each Windows VM immediately after OS installation ‚Äî enables file copy between host and VM
  • Assign static IPs via Network Settings ‚Üí IPv4 ‚Üí Manual; use IPs in the same subnet as your home router (e.g., 192.168.0.200, .201, .202)
  • Enable ICMP (ping) inbound firewall rule or disable firewall entirely for lab use
  • Mount the Windows Server ISO via VM Settings ‚Üí CD/DVD to install the operating system

Package Download & Versioning

Obtaining Commvault software and understanding release strategy

Commvault software is downloaded from the official portal. Understanding versioning strategy is essential because most enterprise customers intentionally remain one release behind the latest version.

Version Strategy

üéØ N-1 Principle

Most production environments run the version one behind the latest release. If 11.36 is current, production runs 11.34. This avoids exposure to bugs in newly released versions.

üìÖ LTS ‚Äî Long-Term Support

Version 11.32 is the current Long-Term Support release. Commvault guarantees approximately 5 years of patches and support. This is the most common version in enterprise environments.

üí° Release Cadence

Commvault releases new feature versions twice per year (bi-annually). They also publish hotfixes and service packs for supported versions on a more frequent basis. Always check the Commvault documentation for the support lifecycle of any version you are deploying.

Download Methods

There are two methods to obtain Commvault packages. Access requires a Commvault Support Account (provided by your employer or Commvault representative).

Method 1 ‚Äî Full Media Kit

  1. Navigate to the portal

    Go to store.commvault.com (redirects to cloud.commvault.com). Log in with Commvault credentials.

  2. Select version and platform

    Media Kit ‚Üí Version 11.32 ‚Üí Search "Full". Choose the correct variant: Windows x64 (for CommServe/Windows MA), Unix/Linux x64 (for Linux clients/MA).

  3. Download the package (~10‚Äì15 GB)

    The full package contains all possible agent types and installers bundled together.

Method 2 ‚Äî Download Manager (Selective)

  1. Download the Download Manager executable

    A small ~30 MB launcher from cloud.commvault.com.

  2. Run Download Manager and select packages

    Choose only the packages needed (e.g., Linux x64 File System only), reducing download size significantly.

  3. Specify download location

CommServe Installation & Configuration

Installing the Commvault management server from scratch

The CommServe package is installed using the setup.exe wizard included in the full media kit. As part of this installation, SQL Server is automatically installed and configured ‚Äî you do not need to pre-install it.

‚ö†Ô∏è Drive Selection ‚Äî Critical

In Azure VMs, the D: drive is temporary storage ‚Äî all data is wiped on reboot. Always use the C: drive (or provision a dedicated data disk) for CommServe database and installation paths. In VMware home labs, adding a dedicated virtual disk (e.g., E:) and using it for data directories is strongly recommended by Commvault.

Installation Wizard Walkthrough

  1. Extract the package and launch Setup

    Right-click setup.exe ‚Üí Run as Administrator. Select language and accept the license agreement.

  2. Select "Install package on this computer" and create a new CommCell

    Choose Create a new CommCell. Select All In One or specific roles as needed.

  3. Installation Path

    Use a drive that is NOT the OS drive where possible. In Azure lab, use C: (only available drive). In VMware lab, add a dedicated virtual disk first.

  4. Data Directory Path

    Set to a dedicated non-OS drive (E: or D: if not Azure). This is where DDB and Index Cache hosted if Commserve is given all in one(MediaAgent) role.

  5. SQL Server Installation Path

    Default: C:\Program Files\Microsoft SQL Server\. Accept default or redirect to data drive.

  6. CommServe Database Path

    Ideally on a dedicated drive, separate from the OS and SQL Server binaries.

  7. Disaster Recovery (DR) Backup Path

    Location where DR metadata files are stored. Use a local path for lab; use a network path in production for true disaster recovery.

  8. Set Client Hostname

    When prompted, enter the private IP address of the CommServe VM as the hostname (e.g., 10.0.0.4). This is required when DNS is not configured to resolve hostnames.

  9. Set Admin Credentials

    Create the primary admin account (username: admin). Record this password ‚Äî it is used to log into the Command Center and Console.

Third-Party Software Installed Automatically

  • Microsoft SQL Server ‚Äî hosts the CommServe database (CommServe DB)
  • IIS (Internet Information Services) ‚Äî serves the Command Center web interface
  • Microsoft .NET Framework ‚Äî required by CommServe services
  • VC++ Redistributable ‚Äî C++ runtime libraries

Post-Installation Verification

After installation, verify the environment is healthy before proceeding to Media Agent configuration.

Storage Libraries & Storage Policies

Defining where backup data lives and how long it's retained

Disk Library

A disk library is the logical representation of a storage destination in Commvault. It is backed by physical disk space on the Media Agent server (or a network share). The library is made up of one or more mount paths ‚Äî folder locations where backup data is actually written.

Storage Types in Commvault

üíΩ Disk Library

Most common. Backup data written to local or NAS disk attached to the Media Agent. Fast, easy to configure.

üìº Tape Library

Physical tape drives/autoloaders. Preferred by highly secure environments (government, compliance-heavy sectors). Slow access but low cost for large archives.

‚òÅÔ∏è Cloud Library

Direct-to-cloud backup (Azure Blob, AWS S3, GCP Storage). Elastic capacity, no hardware to manage.

Creating a Disk Library

  1. Console Browser ‚Üí Storage Resources ‚Üí Libraries ‚Üí Right-click ‚Üí Add ‚Üí Disk Library

    Assign a friendly name (e.g., DiskLib_MA01).

  2. Select the Media Agent

    Choose the MA server that will host this library.

  3. Add Mount Path

    Provide a folder path on the MA server (non-temporary drive). Example: C:\CVMountPath1\. Avoid D: on Azure VMs (temporary).

  4. Validate Mount Path

    Right-click mount path ‚Üí Check Status (verifies online/offline) ‚Üí Check Throughput (measures actual write speed). Values above 100 MB/s are excellent.

üí° Mount Path Best Practice

Add multiple mount paths from different drives when available (e.g., E: and F:). This distributes I/O load across spindles, improving throughput and providing redundancy. In a lab with only one drive, multiple mount paths on the same drive are acceptable.

Storage Policies

A storage policy is a logical construct that defines three key parameters for backup data:

üìå What to back up

Associates with sub-clients/backup sets that define the source data (files, databases, VMs).

üìå Where to store it

Points to a specific disk, tape, or cloud library via a storage pool.

üìå How long to keep it

Retention rules ‚Äî e.g., 30 days or 1 month. After expiry, data is automatically removed from storage.

Deduplication vs. Non-Deduplication

Deduplication is a data reduction technique that eliminates redundant blocks before writing to storage. It operates at the block level, comparing content hashes rather than filenames.

üî¥ Non-Dedup Policy

Every backup job is written in full, regardless of content similarity. Two identical 124 MB files are stored as 248 MB of backup data. Simpler, lower CPU overhead, faster restores.

üü¢ Dedup Policy

Unique blocks are written once; duplicate blocks are referenced via pointers in the Deduplication Database (DDB). Two identical 124 MB files are stored as ~124 MB. Significant storage savings for environments with repetitive data.

üìù Deduplication Database (DDB) Partition

The DDB must be hosted on a fast, dedicated disk partition on the Media Agent. A separate SSD or dedicated volume is strongly recommended in production. Never place the DDB on a temporary drive (Azure D: drive).

Storage Policy Summary

  • Disk Library = physical storage location managed by a Media Agent
  • Mount Path = actual folder on disk where backup data is written
  • Storage Pool = groups one or more libraries; can be dedup or non-dedup
  • Storage Policy = defines what, where, and how long; links clients to storage pools
  • Retention = configured in days (e.g., 7, 30 days); after expiry, data is pruned

Agent Installation Methods

Push install and interactive install for Windows and Linux clients

Commvault agents (iDataAgents) can be installed using two primary methods. Understanding both is important because the appropriate method depends on network access, firewall configuration, and client operating system.

üîÑ Push Install (Remote)

Initiated from the CommCell Console or Command Center. CommServe pushes the package from its Software Cache to the target machine over the network. Requires open firewall ports and valid credentials. Fastest for bulk deployments.

üíª Local/Interactive Install

Package is manually copied to the target machine and the setup wizard is run locally. Used when firewall ports are closed between CommServe and client, or for isolated environments. Also called "interactive install" or "manual install."

Push Install ‚Äî Linux File System Agent

  1. Verify Linux package in Software Cache

    Console Console ‚Üí Tools ‚Üí Add/Remove ‚Üí Cache Details ‚Üí confirm Unix/Linux x64 is present.

  2. Command Center ‚Üí Manage ‚Üí Servers ‚Üí Add Server

    Enter private IP of the Linux client, select OS type as Unix/Linux.

  3. Enter Linux credentials

    Username (e.g., cvadmin) and password. For Linux, no dot-slash prefix is needed ‚Äî just the plain username. Ensure the user has sudo/root-level access.

  4. Select package: File System

    Commvault will also auto-select dependent packages.

  5. Click Install and monitor the job

    Navigate to Jobs ‚Üí Active Jobs to watch progress. Installation typically takes 10‚Äì15 minutes.

Push Install ‚Äî Windows File System Agent (Command Center)

  1. Command Center ‚Üí Manage ‚Üí Servers ‚Üí Add Server

    Enter private IP of the Windows client, select OS type as Windows.

  2. Provide Windows admin credentials

    Format: .\administrator (local account) or DOMAIN\username (domain). Enter the Windows VM's admin password.

  3. Select package: File System

    Click Install and monitor the resulting installation job.

Local / Interactive Install ‚Äî Windows (Console Wizard)

  1. Copy the full media kit zip to the client machine

    Use RDP file copy (drag-and-drop) or WinSCP. Extract the zip before running setup.

  2. Run setup.exe as Administrator

    Wizard starts ‚Äî select Install package on this computer ‚Üí Advanced Selection ‚Üí select only File System.

  3. Enter client hostname (use IP)

    Set the hostname field to the client's private IP address (e.g., 10.0.0.7).

  4. Enter CommServe IP address

    The wizard asks which CommServe to register with. Enter the CommServe private IP (e.g., 10.0.0.4).

  5. Authenticate with CommServe console credentials

    ‚ö†Ô∏è These are the CommCell Console login credentials (e.g., admin + password), NOT the Windows server credentials.

  6. Complete the wizard and verify registration

    Go back to CommCell Console ‚Üí Client Computers ‚Üí View ‚Üí Refresh. The new client should appear.

Also, from a new Client/MediaAgent, the backup admin can UNC from the Client/MediaAgent to the CommServe's Software Cache location, then right click, and run Setup.exe as an Admin instead of downloading or copying the installer to the Client/MediaAgent.

Local Install ‚Äî Linux (WinSCP + Interactive)

  1. Download the Linux base package

    From cloud.commvault.com ‚Äî download the Unix/Linux variant.

  2. Transfer to Linux client using WinSCP

    WinSCP is a free SFTP client for Windows. Connect using the Linux client's public IP (or private IP if on same network), username, and password. Transfer the package to /tmp/.

  3. Connect to Linux via PuTTY and extract

    Open PuTTY, enter the Linux public IP, and log in. Use sudo su - to become root. Navigate to /tmp and extract the package.

  4. Run the Commvault installer script

    Follow the interactive prompts to install the File System agent and register with the CommServe.

üí° WinSCP vs PuTTY

WinSCP = GUI SFTP client used to transfer files between Windows and Linux. PuTTY = SSH terminal client used to execute commands on a Linux server. Both are free to download. Use WinSCP first to upload the package, then PuTTY to install it.

Day-to-Day Backup Admin Operations

Roles, responsibilities, and daily workflow for a Commvault administrator

Understanding daily operational responsibilities is essential for both job interviews and for planning your learning path. The backup admin's scope spans incident response, configuration management, user administration, and capacity planning.

Daily Workflow

  • üìß
    Email Triage & Communication
    Review incoming requests and alerts from customers, application owners (DBA teams, Windows admins, Linux admins, cloud teams), and internal backup team. Respond to SLA inquiries, backup failure notices, and configuration requests.
  • üî¥
    Incident Management
    Immediate response to broken functionality: failed backup jobs, restore failures, CommServe downtime, Media Agent offline, performance degradation. Incidents require rapid investigation and resolution.
  • üìù
    Service Request (SR) Fulfillment
    Planned configuration work requested by customers or other teams: configure new backup jobs (file system, DB, VM), report on retention status, add storage, create user accounts, adjust backup schedules.
  • üîß
    Change Request (CR) Implementation
    Scheduled maintenance: Commvault patching/upgrades (monthly or bi-annual), infrastructure changes. Requires documented action plan, rollback plan, and scheduled downtime window (typically weekends).
  • üë•
    Daily Team Review Meeting
    Internal standup with backup team lead and manager. Discuss open incidents, pending SRs, upcoming changes, action items, and any immediate-attention issues.
  • üõ°Ô∏è
    Security & Compliance Administration
    Ensure encryption is enabled for backup data, ransomware protection features (WORM, immutability) are configured, and audit requirements are met. Internal and external audits will verify these configurations.
  • üë§
    User Administration
    Create and manage user accounts in the CommCell Console/Command Center. Apply role-based access control (RBAC) ‚Äî new users (especially junior staff) should have restricted roles, not admin-level access.
  • üíæ
    Library & Storage Management
    Monitor disk utilization in libraries, add mount paths when capacity is low, manage tape media rotation (if applicable), and manage cloud storage bucket settings.

Ticketing System Integration

üéì Exam / Interview Tip ‚Äî Patching

Section 11

Scenario-Based Interview Q&A

Common technical questions with detailed answers for L1/L2 Commvault roles

Explain the three-tier Commvault architecture and the role of each component.
Commvault uses a three-tier model. The CommServe (Tier 1) is the administrative hub ‚Äî it coordinates all jobs, stores metadata in a SQL Server database, and manages licensing and configuration. The Media Agent (Tier 2) is the data mover ‚Äî it receives backup streams from clients and writes data to storage (disk, tape, cloud). The Client/Agent (Tier 3) is the source machine whose data is being protected. An iDataAgent installed on the client communicates with the CommServe for job coordination and sends data to the Media Agent. In agentless scenarios (VMware/Hyper-V), no agent is on the target ‚Äî a Virtual Server Agent proxy handles data capture.
What is the difference between a push install and a local (interactive) install?
A push install (also called remote install) is initiated from the CommCell Console or Command Center. The CommServe pushes the Commvault package from its Software Cache to the target machine over the network using administrative credentials. It requires specific firewall ports (8400, 8403, and 135/139/445 for Windows or 22 for Linux). A local install (interactive install) requires manually copying the package to the target machine and running the setup wizard on that machine. It's used when firewall restrictions prevent push installation. The local install wizard asks for CommServe connection details and CommCell Console credentials to register the client.
What are the mandatory firewall ports for Commvault, and where must they be open?
The mandatory base ports are 8400 and 8403 (TCP). These must be open on ALL machines where any Commvault component is installed ‚Äî CommServe, Media Agents, and all client machines. For push installation of Linux clients, port 22 (SSH) must be open on the client. For push installation of Windows clients, ports 135, 139, and 445 must be open on the client. Port 8401 is only required for specific tunnel communication scenarios.
Is a server reboot required after installing a Commvault package?
No ‚Äî a reboot is generally not required after Commvault package installation. Services start automatically after the installation wizard completes. A reboot may be required in exceptional cases (e.g., pending Windows OS updates, or specific kernel-level driver installations on Linux), but this is the exception, not the rule.
What is the Software Cache, and why is it important?
The Software Cache is a local repository on the CommServe that stores all Commvault installation packages (Windows, Linux, Solaris, etc.). It is essential for remote/push installations ‚Äî when you install a package on a remote client from the CommCell Console, the CommServe pulls the appropriate package from the cache and pushes it to the target machine. Without a populated cache, push installations will fail. The cache can be built by copying packages manually to the CommServe and running a "Copy Software" job, or by using the "Download and Sync Cache" feature if the CommServe has internet access.
Explain deduplication and when you would use a dedup storage policy vs. a non-dedup policy.
Deduplication is a data reduction technique that writes only unique blocks to storage. The Deduplication Database (DDB) maintains a hash index of all written blocks; when a duplicate block is encountered, only a pointer is written instead of the full block. This significantly reduces storage consumption in environments with high data repetition (e.g., multiple VMs with the same OS, regular full backups of slowly changing file systems). A non-dedup policy writes all data in full, regardless of duplicate content ‚Äî it's simpler and imposes less CPU overhead on the Media Agent, making it better for high-throughput environments or data that is already compressed/encrypted. Use dedup when storage savings outweigh the CPU cost of the DDB lookups.
What is the difference between CommCell Console (Java) and Command Center?
The CommCell Console is the legacy Java-based desktop application. It provides deep access to all configuration options and is present in deployments on Commvault v11.32 and earlier. The Command Center is the modern, browser-based web interface accessible at https://CommServe-IP/commandcenter. For new deployments on v11.34 and later, Command Center is the only supported interface. Commvault has deprecated CommCell Console for new environments. As a backup admin, you need familiarity with both because existing customers running older versions or upgrading still use the Java console.
You have been asked to configure a new file system backup for a Windows server. Walk me through the process.
First, I verify that port 8400 and 8403 are open on the target server and confirm network connectivity with a ping test. In Command Center, I navigate to Manage ‚Üí Servers ‚Üí Add Server, enter the server's IP address, select Windows as the OS, and provide the administrator credentials. I select the File System package and click Install, monitoring the installation job to completion. Once the agent is installed, I create a Storage Policy (Plan) if one doesn't exist ‚Äî defining the destination disk library and retention period. Then I navigate to the client's File System agent in Command Center, configure the backup set and subclient to include the desired folder paths (e.g., D:\Data), associate the storage policy, and schedule the backup. Finally, I run a manual backup job to validate the configuration before handing it over.
What is the difference between agent-based and agentless backup in Commvault?
Agent-based backup requires installing a Commvault iDataAgent (package) inside the target machine ‚Äî this applies to file system servers, database servers, physical machines, etc. The agent communicates directly with the CommServe and Media Agent. Agentless backup is used for virtual machines in VMware or Hyper-V environments ‚Äî no agent is installed inside the guest VM. Instead, a Virtual Server Agent (VSA) is installed on a proxy server (for VMware) or directly on the Hyper-V node. The VSA communicates with the hypervisor API to snapshot and back up the VM at the disk level. Agentless backup is simpler to manage at scale (no agents to maintain per VM) but provides less granular control compared to agent-based approaches.
Explain the role of a Disk Library and a Mount Path in Commvault.
A Disk Library is the logical storage entity in Commvault that represents a collection of disk-based storage. It is associated with a specific Media Agent and tells Commvault "this is where you can write backup data on this MA." A Mount Path is a physical folder path on the Media Agent's disk that is configured within a Disk Library ‚Äî it is the actual destination where backup data blocks are written. A single Disk Library can have multiple mount paths across different drives, providing storage pooling and I/O distribution. Commvault writes to mount paths in a round-robin fashion when multiple are configured. Before using a library, you should validate mount paths to confirm they are online and measure their write throughput.
