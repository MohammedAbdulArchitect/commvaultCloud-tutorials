Section 02

CommServe LiveSync Architecture

SQL Server log shipping replication ‚Äî the technology powering CommServe HA/DR

CommServe LiveSync is Commvault's native HA/DR solution for the CommServe component. Unlike block-level replication (EMC SRDF, Veritas VVR) or array-level replication, LiveSync operates at the application layer ‚Äî specifically leveraging Microsoft SQL Server's built-in replication technology.

Replication Technology Deep Dive

üîµ Block-Level Replication (Storage)

  • EMC SRDF: Synchronous block-level replication at SAN level
  • Veritas VVR: Volume-level replication between storage arrays
  • Replicates entire disk blocks ‚Äî no application awareness
  • Hardware-dependent, managed by storage team

üü† Application-Level Replication (LiveSync)

  • SQL Server Log Shipping: Replicates committed transaction logs
  • Application-aware ‚Äî replicates only CommServe database changes
  • Software-managed by Commvault, independent of storage hardware
  • Frequency: Every 15 minutes (default, configurable)

üîë Critical LiveSync Insight

LiveSync uses SQL Server Log Shipping as its underlying replication mechanism across the 11.x product line. This replicates only committed transactions, making it asynchronous ‚Äî not a synchronous real-time mirror. The default replication interval is 15 minutes. This means a maximum of 15 minutes of potential data loss (job metadata) exists in an unplanned failure scenario.

LiveSync Replication Cycle ‚Äî Process Flow

The 15-minute replication cycle is a continuous automated loop managed by Instance 002 on both nodes. Understanding each phase clarifies what data is at risk in an unplanned failure.

üîÑ SQL Log Shipping ‚Äî 15-Minute Replication Cycle
1
CommServe Engine Writes Transactions Production ¬∑ Instance 001
All backup job activity ‚Äî job status updates, schedule changes, client registrations ‚Äî are committed as SQL transactions to the CommServe database on the active node.
‚ñº
2
SQL Transaction Log Generated SQL Server ¬∑ Active Node
SQL Server continuously writes committed transactions to the transaction log file (.ldf). This log captures every database change since the last shipping cycle.
‚ñº
3
Log Backup Job Triggers Every 15 Minutes
Instance 002 (Failover Assistant) on the production node triggers a SQL log backup job on the configured schedule. The log is exported to a shared staging path accessible by both nodes.
‚ñº
4
Log File Shipped to DR Node Network Transfer
The exported transaction log file is transferred across the network (LAN or WAN) to the DR/Standby CommServe node. Bandwidth and latency here directly affect replication success.
‚ñº
5
Log Applied to DR Databases DR Node ¬∑ Instance 002
Instance 002 on the DR node restores (applies) the received log file to the CommServe databases in standby/recovery mode. Databases on the DR node are now synchronised to the point-in-time of the last log backup.
‚ñº
‚ö†
RPO Gap ‚Äî Potential Data Loss Window Risk Zone
Any transactions committed on the active node after the last successful log ship but before the next cycle are not yet on the DR node. In an unplanned failure at the worst moment in the cycle, up to 15 minutes of job metadata may be lost.

Instance 001 vs Instance 002 ‚Äî Understanding the Dual Process Manager

‚öôÔ∏è Instance 001 ‚Äî CommServe Process Manager

  • Primary CommServe services: CommServe Engine, Messaging Queue, Workflow, Web Console
  • Services RUNNING on production, STOPPED on DR (normal state)
  • This is where all backup and restore operations are orchestrated
  • Installed at: Content Store\ (default path)

üîÑ Instance 002 ‚Äî Failover Assistant

  • SQL client instance used exclusively for log shipping and failover control
  • Services ALWAYS RUNNING on BOTH nodes (production and DR)
  • Installed when the CommServe Failover package is selected
  • Installed at: Content Store 2\ (separate path)
  • Contains the Failover Assistant UI for initiating all failover operations
üí° Naming Conventions Used in Production

Different organisations use different naming conventions for the same components. All of these refer to the same entities:

  • Primary CommServe = Production CommServe = Active Node
  • Standby CommServe = Passive Node = DR CommServe = Failover Node
  • Instance 002 = Failover Assistant = SQL Client = CommServe_SQL (common renaming)

Automatic vs Manual Failover

ü§ñ Automatic Failover (Rarely Used)

  • Requires 3 Media Agents designated as Monitoring Nodes
  • Monitoring nodes continuously poll the production CommServe
  • If all 3 nodes lose connectivity, DR node auto-starts services
  • Risk: Potential data loss up to the last replication window (15 min)
  • Split-brain risk during transient network outages (panic reboot scenario)
  • Used by less than 5% of customers due to data loss risk

üë§ Manual Failover (Industry Standard)

  • Backup administrator makes the decision and initiates failover manually
  • Administrator detects failure via monitoring alerts (email/SNMP)
  • Full control over timing ‚Äî no risk of premature failover
  • Can run a final replication before failover to minimise data loss
  • The High Availability Cluster (HAC) package must be installed on all instances
  • Preferred by 95%+ of enterprise customers

Section 03

CommServe LiveSync ‚Äî Lab Build Guide

Step-by-step installation of production and DR CommServe nodes with the 11.32‚Üí11.36 upgrade path

üéì Why Install 11.32 First, Then Upgrade to 11.36?

Commvault 11.36 restricts new installations to Command Center only ‚Äî the legacy Java-based CommCell Console is not available on a fresh 11.36 install. However, when upgrading from 11.32 to 11.36, both the CommCell Console and Command Center remain accessible. This matters because several advanced features (cycle-based retention changes, storage policy management) are only available in the CommCell Console in certain versions. The strategy: install 11.32 first, then upgrade to 11.36.

Pre-Installation Checklist ‚Äî Windows VM Setup

  1. Create Virtual Machine

    Minimum: 4 vCPU cores, 16 GB RAM, 50 GB OS disk. Select Windows Server 2022 (supported for 11.32 and 11.36). Use bridged networking for internet access from the VM.

  2. Configure Static IP Address

    Navigate to: Network & Sharing Center ‚Üí NIC Properties ‚Üí TCP/IPv4 ‚Üí Use the following IP address. Assign a static IP to prevent DHCP changes after reboot. Production CommServe: 192.168.10.200 | DR CommServe: 192.168.10.210

  3. Set Hostname (Optional but Recommended)

    Go to Server Manager ‚Üí Local Server ‚Üí Computer Name. Assign a meaningful name (e.g., COMMSERVE-PROD) before installation.

  4. Disable Windows Firewall (Lab Only)

    Server Manager ‚Üí Local Server ‚Üí Windows Defender Firewall ‚Üí Domain / Private / Public ‚Üí Off. In production: open TCP ports 8400 and 8403 in the enterprise firewall instead.

  5. Disable IE Enhanced Security

    Server Manager ‚Üí Local Server ‚Üí IE Enhanced Security ‚Üí Off for Administrators. Enables normal browser access to Command Center.

  6. Enable Remote Desktop

    Server Manager ‚Üí Local Server ‚Üí Remote Desktop ‚Üí Allow remote connections.

  7. Create a Dedicated Installation Drive

    Disk Management ‚Üí right-click unallocated space ‚Üí New Simple Volume. Create an E: drive (50 GB) for Commvault software. Best practice: never install on the C: (OS) drive.

  8. Download Commvault Software

    Access store.commvault.com ‚Üí Media Kit ‚Üí 11.32 LTS ‚Üí Full Media Kit (Windows x64, ~10 GB). Extract to the E: drive on the CommServe VM.

‚ö†Ô∏è DR Backup Path ‚Äî Always Use Network Share

During CommServe installation, you are prompted for a Disaster Recovery Backup Path. Commvault strictly recommends a network UNC path (e.g., \\fileserver\CommvaultDR\), not a local drive. If the CommServe crashes and cannot boot, a local-path DR backup is inaccessible. A network share remains accessible from any live machine for recovery. This can be changed post-installation via: Control Panel ‚Üí DR Backup Destination.

Installing the CommServe Package (11.32)

  1. Run Setup.exe from the extracted media kit

    Select: Install on this computer ‚Üí Create a New CommServe ‚Üí Server Component (not All-in-One, unless single-node environment).

  2. Installation Path

    Change from default C: to E:\Program Files\Commvault\ for dedicated drive separation.

  3. Database Path

    Specify the CommServe database path (E: drive) and the Disaster Recovery backup path (network share preferred).

  4. Admin Credentials

    Default username: admin (cannot be changed). Set a strong password and register a valid email address ‚Äî required for password recovery and license management.

  5. Wait for Completion

    Installation includes SQL Server engine, all Commvault packages, and database creation. Typical duration: 60‚Äì90 minutes.

  6. Verify Installation

    Start ‚Üí Commvault ‚Üí Process Manager ‚Üí verify all services are green. Access Command Center at https://[CommServe-IP]/commandcenter.

Upgrading to 11.36

Once 11.32 is confirmed operational, upgrade to 11.36 using the full media kit for 11.36:

  1. Pre-upgrade: Stop Active Jobs

    In production: Suspend all active backup jobs, disable schedulers, run a CommServe DR backup. In lab: not mandatory.

  2. Run 11.36 Setup.exe

    The installer detects the existing 11.32 installation automatically and presents the Upgrade Feature Release option. Select it.

  3. Skip Full Database Maintenance

    Unless directed by Commvault support, do not select this option ‚Äî it extends the upgrade window by several additional hours without operational benefit in most cases.

  4. Wait for Completion

    Typical duration: 60‚Äì90 minutes. Post-upgrade version will show as 11.36.xx in the Process Manager.

Two methods exist for populating the software cache:

Installing the Failover Package on Production CommServe

  1. Run Setup.exe from the media kit on the production CommServe

    Select: Install on this computer ‚Üí Advanced Selection ‚Üí locate and check CommServe Failover package.

  2. Display Name

    Assign a unique display name (e.g., CommServe_SQL) ‚Äî do NOT use the same name as the CommServe itself. The IP/hostname will remain the same (same server).

  3. Verify Instance 002 Created

    Start ‚Üí Commvault ‚Üí you now see two Process Managers: Commvault Process Manager (Instance 001) and Commvault Process Manager - Failover Assistant (Instance 002).

‚ö†Ô∏è High Availability Cluster (HAC) Package ‚Äî Mandatory for Automatic Failover

If the Server Component installation option was chosen instead of All-in-One, the High Availability Cluster (HAC) package is not installed automatically. This package is required for automatic failover to function. To add it post-installation: CommCell Console ‚Üí select CommServe client ‚Üí All Tasks ‚Üí Add/Remove Software ‚Üí Install Software ‚Üí find High Availability Computing in the Tools section. Install on all four instances: Production Instance 001, Production Instance 002, DR Instance 001, DR Instance 002.

Building the DR CommServe Node

  1. Create DR Virtual Machine

    Same OS version and resource profile as production (Windows Server 2022, 4+ vCPU, 16 GB RAM). Assign hostname COMMSERVE-DR and static IP: 192.168.10.210.

  2. Apply Same Prerequisites

    Disable firewall (lab), enable RDP, assign static IP, create dedicated drive.

  3. Run Setup.exe ‚Üí Select Failover Package Only

    Navigate to: Install on this computer ‚Üí Join Existing CommServe (or Advanced Selection) ‚Üí select CommServe Failover. The wizard automatically installs both Instance 002 and Instance 001 ‚Äî including SQL Server engine ‚Äî to create a mirror of the production CommServe.

  4. Provide CommServe Credentials

    When prompted, enter the credentials for the production CommServe console (admin password), not the local server credentials.

  5. Verify DR Instances

    On the DR node, Instance 001 services will be stopped by default ‚Äî this is expected and correct. Instance 002 services run continuously to manage replication.

Enabling LiveSync Replication

  1. Create a Non-Deduplicated Storage Pool

    Required as a prerequisite: Command Center ‚Üí Storage ‚Üí Add Disk Storage ‚Üí disable deduplication toggle. This pool is used specifically for LiveSync job data.

  2. Create a LiveSync Plan

    Command Center ‚Üí Manage ‚Üí Plans ‚Üí Server Backup Plan. Add a copy pointing to the non-deduplicated disk storage. Define full and incremental schedules.

  3. Enable CommServe LiveSync

    Command Center ‚Üí Manage ‚Üí System ‚Üí CommServe LiveSync ‚Üí Enable Replication. Configure: Automatic Failover (leave disabled), Replication Frequency (default: 15 minutes), Plan (select the LiveSync plan created above).

  4. Verify Sync Status

    Failover Assistant ‚Üí Instance 002 ‚Üí observe Last Sync Time changing from Not Available to a current timestamp. Status should show green for both nodes.

Production Failover vs Maintenance Failover ‚Äî Key Difference

üîß Maintenance Failover

  • DR console operates in Read Only mode
  • Only restore requests can be processed
  • Activity Control panel shows all options disabled/unchecked
  • Use case: CommServe OS patching where backups can wait but restores must remain available
  • Appropriate for short 2‚Äì3 hour change windows

üîÑ Production Planned Failover

  • DR console operates as a fully functional CommServe
  • Both backups AND restores run normally
  • Activity Control panel shows all options enabled
  • Clients are fully routed through the DR CommServe
  • Use case: Formal DR drill, extended maintenance, datacenter migration

Failback ‚Äî Critical Consideration

üö® Reverse Replication Before Failback

If backup jobs ran on the DR CommServe during a production planned failover (e.g., a DR drill lasting 24+ hours), those job metadata entries exist only on the DR CommServe database ‚Äî they have not yet replicated back to the production CommServe. Before initiating failback, ensure the reverse replication (DR ‚Üí Production) completes successfully. Initiating failback without completed reverse replication will result in those backup jobs being absent from the production CommServe job history.

Linux CommServe Deployment

Rocky Linux 9.5 ‚Äî supported flavors, LVM setup, and installation with cvpkgadd

Commvault 11.36 introduced official support for Rocky Linux as an alternative to Red Hat Enterprise Linux for CommServe deployments. This is particularly relevant in environments standardised on Rocky Linux for cost reasons (Rocky Linux is a free RHEL clone) or in containerised/cloud-native environments.

‚ö†Ô∏è Linux Flavor Compatibility ‚Äî 11.36 Only

CommServe 11.32 does NOT support Rocky Linux ‚Äî only Red Hat Enterprise Linux. Rocky Linux support was introduced in version 11.36. This means the 11.32‚Üí11.36 upgrade path cannot be used for a Linux CommServe on Rocky Linux. Install 11.36 directly. Always verify OS compatibility at documentation.commvault.com under System Requirements for the target version.

Supported Linux Flavors for CommServe (11.36)

Red Hat Enterprise Linux (RHEL)

  • Supported since early Commvault versions
  • RHEL 8.x and RHEL 9.x supported for 11.36
  • Commercial license required ‚Äî standard in large enterprises

Rocky Linux

  • Free, open-source RHEL binary-compatible clone
  • Rocky Linux 9.5 supported from CommServe 11.36+
  • Ideal for lab environments and cost-conscious deployments
  • Download the DVD ISO from rockylinux.org (free, ~11 GB)

LVM Partition Strategy ‚Äî Best Practice

The recommendation for Linux CommServe is to install Commvault on /opt, separated from the root partition. This requires creating an LVM logical volume on a dedicated disk:

# Step 1: Create Physical Volume on new disk
pvcreate /dev/sdb

# Step 2: Create Volume Group
vgcreate vg_commvault /dev/sdb

# Step 3: Create Logical Volume (adjust size as needed)
lvcreate -L 50G -n lv_opt vg_commvault

# Step 4: Format with XFS (recommended for Commvault on Linux)
mkfs.xfs /dev/vg_commvault/lv_opt

# Step 5: Mount permanently ‚Äî add to /etc/fstab
echo '/dev/vg_commvault/lv_opt  /opt  xfs  defaults  0 0' >> /etc/fstab
mount -a

WORM Storage Lock

Write Once Read Many ‚Äî immutable backup storage for compliance and ransomware defence

WORM (Write Once Read Many) is a data storage principle where, once written, data cannot be modified or deleted until a defined retention period has expired. Commvault's WORM Storage Lock feature implements this principle at the storage pool level, providing a last line of defence against data tampering ‚Äî whether by malicious insiders, ransomware attackers, or accidental actions by administrators.

WORM Storage Lock ‚Äî Protection Layers
üóÑÔ∏è
Storage Pool / Disk Library Mount Path
Storage Lock prevents manual modification or deletion of backup objects (chunks/volumes) stored on the mount path. Even a storage admin cannot delete or rename these files while the lock is active. Only the Commvault process can remove data after retention-based aging completes.
üìã
CommServe Database ‚Äî Backup Job Records
Compliance Lock prevents deletion of backup job entries from the CommServe database (the job history). A backup admin with Master-level access cannot delete backup jobs or reduce retention below the configured minimum. Ensures job continuity records cannot be erased.
‚è∞
Retention Enforcement
The WORM lock period is governed by the maximum retention of all storage policies associated with the storage pool. If one policy has 14-day retention and another has 30-day retention, ALL objects are locked for 30 days (the maximum). Retention can be increased but NOT reduced once WORM is enabled.

Storage Lock vs Compliance Lock ‚Äî The Critical Distinction

Storage Lock

Protects the Physical Data

Locks the actual backup data files (chunks/volumes) on the disk library mount path against modification, renaming, or deletion.

  • Prevents a rogue storage admin from deleting the mount path folder
  • Prevents a ransomware attacker (who gained access to the media agent) from encrypting/deleting backup data
  • Cannot be disabled ‚Äî only expired through retention aging
Compliance Lock

Protects the Job Records

Prevents deletion of backup job entries from the CommServe console (Command Center or CommCell Console).

  • Prevents a backup admin from accidentally deleting the wrong client's jobs
  • Prevents retention reduction ‚Äî cannot lower retention below set value
  • Can be disabled using an auth code from Commvault support
  • Enabled automatically when Storage Lock is enabled
‚ö†Ô∏è WORM is Irreversible ‚Äî Plan Retention Carefully First

Once WORM Storage Lock is enabled on a storage pool, the Storage Lock cannot be disabled. The only way to remove it is to decommission the entire storage pool and recreate it. The Compliance Lock can be disabled with a multi-step process requiring an auth code from Commvault Support. Before enabling WORM, review and finalize the retention settings of all plans/storage policies associated with that pool. Remember: the maximum retention across all policies defines the WORM lock duration.

What WORM Prevents and Permits

üö´ Actions BLOCKED by WORM

  • Manually deleting backup objects/volumes from the mount path folder (Storage Lock)
  • Renaming backup data files on the mount path (Storage Lock)
  • Deleting backup jobs from CommCell Console or Command Center (Compliance Lock)
  • Reducing storage policy/plan retention below current value (Compliance Lock)
  • Deleting the mount path folder or disk library (Storage Lock)

‚úÖ Actions PERMITTED with WORM Enabled

  • Commvault process deleting aged-out backup data (after retention expires)
  • Increasing retention (only direction allowed)
  • Creating new files outside the mount path folder on the same disk
  • Running new backup jobs to the WORM-protected storage
  • Performing restores from WORM-protected data
  • Disabling Compliance Lock (with Commvault Support auth code)

Ransomware Protection for Media Agents

OS hardening, ACL enforcement, and disk library data protection against cyber threats

Ransomware protection in Commvault is a media-agent-level security feature that hardens the media agent's operating system against unauthorised access to disk library data. Unlike WORM (which locks data after writing), ransomware protection is an active defence mechanism that prevents malicious processes from accessing or modifying the mount path in the first place.

Why Disk Libraries Are Specifically Vulnerable

The distinction between cloud, tape, and disk libraries is fundamental to understanding why ransomware protection targets disk libraries specifically:

‚òÅÔ∏è Cloud Library

Access requires API calls with authentication tokens (Azure Storage Explorer, AWS S3 Browser). Cannot be directly browsed or modified from the media agent file system ‚Äî inherently protected by cloud access controls.

üìº Tape Library

Physical media ‚Äî requires SCSI/fibre channel communication. Cannot be directly accessed via file system operations. Physical tape ejection provides inherent air-gap protection.

üíæ Disk Library

Mount path is a folder on the media agent's local disk or NAS share. Directly accessible via Windows Explorer or Linux shell. If an attacker gains OS-level access to the media agent, they can directly read, modify, delete, or encrypt backup data files.

üîë Core Concept: Ransomware Protection Scope

Ransomware Protection is designed exclusively for disk library mount paths. It works by applying file system-level ACLs (Access Control Lists) and role-based security policies to the mount path folder. Only the Commvault Media Agent service account is granted write access. All other processes ‚Äî including local administrator accounts ‚Äî are restricted to read-only or no access. This prevents a compromised OS or malicious process from encrypting or deleting backup data even with administrator credentials.

üåç Real-World Attack Scenario: Why Both Features Are Used Together

A sophisticated ransomware attacker compromises a backup administrator's domain account. With this account, they gain RDP access to the media agent server (Windows). They attempt to encrypt all backup data in the disk library mount path.

With only Ransomware Protection: File system ACLs block the encryption process ‚Äî backup data is protected.
With only WORM: The attacker could overwrite individual files (WORM prevents deletion, not all overwrite scenarios on some filesystems).
With both Ransomware Protection + WORM: Maximum defence ‚Äî ACLs block file access, and WORM ensures any data that does exist cannot be legally deleted until retention expires. Together, they provide defence-in-depth for backup data.

Scenario-Based Interview Questions

Real-world implementation and design questions for L3 Commvault roles

What is CommServe LiveSync and what replication technology does it use?
CommServe LiveSync is Commvault's native HA/DR solution for the CommServe component. Unlike block-level storage replication (EMC SRDF, Veritas VVR), LiveSync operates at the application layer by leveraging Microsoft SQL Server Log Shipping to replicate all CommServe databases from the active node to the passive node. The default replication interval is 15 minutes, meaning replication is asynchronous ‚Äî it is not a real-time continuous mirror. The purpose is to minimise CommServe downtime by allowing an immediate failover to a pre-configured passive node rather than rebuilding the CommServe from scratch using the DR backup files.
Why would you recommend manual failover over automatic failover for most enterprise customers?
Automatic failover introduces a significant risk of false triggering ‚Äî for example, a virtual machine panic reboot recovers within 60 seconds, but if the monitoring nodes detect the outage first, the services fail over to the DR node. When the production CommServe recovers, there is now a potential split-brain situation, and up to 15 minutes of job metadata from the last replication cycle is missing on the DR node. Additionally, a full failover and failback cycle takes approximately 30 minutes each ‚Äî 60 minutes total ‚Äî during which all backup jobs fail regardless. For these reasons, 95%+ of enterprise customers use manual failover, relying on monitoring alerts (email/SNMP) to notify the backup admin of an unplanned outage, who then manually initiates failover with full knowledge of the situation.
What is the difference between production maintenance failover and production planned failover?
Both stop services on the production CommServe, but differ in what is permitted on the DR node. Production Maintenance Failover places the DR CommCell Console in Read Only mode ‚Äî only restore requests can be processed, and no scheduled backups run. The Activity Control panel shows all options disabled. This is used for CommServe OS patching where restores must remain available during the change window. Production Planned Failover is a complete failover where the DR CommServe assumes full production CommServe responsibilities ‚Äî both backups and restores run normally, Activity Control is fully enabled, and all clients route their backup traffic to the DR CommServe. This is used for formal DR drills or extended maintenance requiring full backup continuity.
A customer says their backups are "immutable". How do you verify this in Commvault and what technology underpins it?
Navigate to Command Center ‚Üí Storage ‚Üí select the disk or cloud storage pool ‚Üí Configuration tab ‚Üí look for the WORM section. If the WORM Storage Lock toggle is enabled, the storage is immutable. The underlying technology is Commvault's WORM Storage Lock feature, which applies two protection layers: (1) Storage Lock ‚Äî applies write-protection at the file system level to the disk library mount path, preventing modification, deletion, or renaming of backup data objects until the retention period (defined by the maximum retention across all associated storage policies) expires; and (2) Compliance Lock ‚Äî prevents backup job deletion from the CommServe console and prevents retention reduction. Together these ensure backup data cannot be tampered with by humans or malicious processes for the defined retention period.
Why is ransomware protection applied to media agents specifically, and why does it target disk libraries only?
Ransomware protection is applied at the media agent level because the media agent is the component with direct file system access to disk library mount paths. An attacker who gains OS-level access to a media agent can directly read, modify, or encrypt backup data files stored in the mount path folder ‚Äî no special tools required. Cloud libraries require API authentication (Azure Storage Explorer, S3 Browser) and cannot be directly accessed from the media agent file system. Tape libraries use SCSI/fibre channel protocols with physical media, not file system access. Ransomware protection works by applying strict ACLs to the mount path ‚Äî only the Commvault service account retains write permission. All other processes, including local administrator accounts, are restricted. From version 11.28, this protection is enabled by default on Windows media agents; Linux media agents require manual enablement via CLI commands.
What are the two main pre-requisites before enabling WORM Storage Lock on a storage pool?
Two critical prerequisites must be addressed. First, review and finalise the retention settings of all storage plans/policies associated with the storage pool. Once WORM is enabled, retention cannot be reduced ‚Äî only increased. The WORM lock duration is governed by the maximum retention across all policies pointing to the pool. Accidentally setting WORM on a pool with a 1-year retention policy means all data in that pool is locked for a minimum of 1 year. Second, understand that WORM Storage Lock cannot be disabled after enablement ‚Äî the only path is decommissioning the entire storage pool. The Compliance Lock component can be disabled (with Commvault Support auth code), but the Storage Lock is permanent until each data object's individual retention period expires.
Why would you install Commvault 11.32 first instead of directly installing 11.36 in a new environment?
Commvault 11.36, when installed fresh on a new server, restricts users to Command Center only ‚Äî the legacy CommCell Console (Java-based) is not accessible. However, when upgrading from 11.32 to 11.36, both consoles remain available because the upgrade path preserves the console access. The CommCell Console is important because several features ‚Äî including advanced storage policy management, cycle-based retention configuration, and certain scheduling options ‚Äî are either only available or significantly more accessible in the CommCell Console. For new customers comfortable exclusively with Command Center, direct 11.36 installation is acceptable. For environments requiring CommCell Console access (which is common in large enterprises with complex configurations), the 11.32 ‚Üí 11.36 upgrade path is the recommended approach.
Explain the CommServe Disaster Recovery (DR) backup best practice and why a local path is insufficient.
The CommServe DR backup generates SQL database dump files of all CommServe databases (CommServe.dmp and supporting databases) at regular intervals to the configured DR Backup Destination path. Best practice is to use a network UNC share path (e.g., \\fileserver\CommvaultDR\) rather than a local drive path. The reason: if the CommServe server suffers a catastrophic failure (OS corruption, hardware failure, cannot boot), a local-path DR backup is inaccessible ‚Äî you cannot mount the drive or access the files without recovering the server first. A network share path remains accessible from any other live server on the network. This means you can immediately begin the CommServe recovery process on a new machine using the DR files, significantly reducing RTO. Additionally, DR backup operates in two phases: Phase 1 writes dump files to the metadata destination (network share), and Phase 2 copies these files to the CommServe DR Storage Policy library, which can be disk, cloud, or tape.
üéØ Model Interview Answer ‚Äî LiveSync Architecture

Structure your answer: "CommServe LiveSync is Commvault's HA/DR solution for the CommServe itself. It uses SQL Server Log Shipping ‚Äî an application-level replication technology ‚Äî to replicate all CommServe databases from the active node to the passive node every 15 minutes. This is an asynchronous replication, meaning there can be up to 15 minutes of potential metadata loss in an unplanned failure. The architecture requires a Failover Assistant (Instance 002) to be installed on both nodes ‚Äî this is the SQL client instance that manages the replication process. Instance 001 is the main CommServe service process, which runs on production and remains stopped on the passive node until a failover is performed. Commvault supports three deployment models: production-only, production plus standby in the same DC (HA), and production plus standby in the same DC plus a DR node in a remote DC (full HA/DR). All configurations are Active-Passive ‚Äî Active-Active is not supported."
