Overview & Prerequisites

Understanding Microsoft 365 as a SaaS Platform and Commvault's Protection Strategy

Microsoft 365 (formerly Office 365) is a cloud-based SaaS platform that provides Exchange Online, Microsoft Teams, OneDrive for Business, and SharePoint Online — all without requiring on-premises infrastructure. Because this is a SaaS solution, traditional agent-based backup strategies do not apply. Commvault uses a dedicated Access Node model with API-based authentication to protect these workloads.

What Microsoft 365 Provides

📧

Exchange Online

Cloud-hosted mailbox service. Each licensed user gets a dedicated mailbox with up to 50 GB of storage, accessible via Outlook Web (outlook.office.com).

💬

Microsoft Teams

Collaboration platform with channels, chats, and meetings. Teams content is backed up per channel and can be restored to its original location or as HTML files.

🗂️

OneDrive for Business

Individual file storage — 1 TB per user by default. Files can be restored granularly or in bulk. Backup agent type: Cloud Apps.

🌐

SharePoint Online

Team sites for document collaboration and intranet content. Requires a certificate-based authentication and supports both document and full-site restore.

Key Concepts Before You Start

Access Node

A Windows server with the Office 365 package installed. It acts as the bridge between Commvault and Microsoft's cloud APIs. Linux is not supported.

Forever Incremental Jobs

All Microsoft 365 backup jobs run as incremental — even the first run. There is no concept of a full backup. The first job captures all existing data.

Archive Jobs (Exchange)

Exchange Online and on-premises Exchange backup jobs are called Archive Jobs, not Backup Jobs. Teams, OneDrive, and SharePoint show as Backup jobs.

Two Plans Required

You need both a Server Plan (storage, frequency, schedule) and a Microsoft 365 Plan (retention for mailboxes). The M365 plan retention takes precedence.

Step-by-Step: Creating the M365 Environment

  1. Subscribe to Microsoft 365 Business Plan: Visit microsoft365.com and choose a plan that includes Exchange Online, Teams, OneDrive, and SharePoint. For lab or PoC use, the Business Basic plan is sufficient — it includes 1 TB OneDrive per user and hosted Exchange.
  2. Create an Admin Account & Tenant: During signup, provide a company name (e.g., techcorp). Your tenant name is auto-generated. Microsoft assigns a domain like [email protected]. This first account is automatically the Global Administrator.
  3. Access the Admin Center: Log in at admin.microsoft.com. Multi-Factor Authentication (MFA) is required on first login. Use Microsoft Authenticator or Google Authenticator to complete the setup.
  4. Verify Global Administrator Role: In the Admin Center under Active Users, click the user and confirm the role is Global Administrator. This role is mandatory for the initial Commvault configuration.
  5. Verify Exchange Online License: In the user's Licenses and Apps tab, confirm that Exchange Online is assigned. Without this license, the mailbox backup configuration will fail.
  6. Disable Multi-Factor Authentication (temporary — Express method only) If using the Express configuration method, navigate to the MFA settings for the service account and disable MFA temporarily during the initial Commvault configuration. Re-enable MFA after configuration is complete. This step is not required when using the Custom (recommended) method — the Custom method uses Azure App Registration with scoped permissions and does not require MFA to be disabled.
  7. Verify Mailbox in Exchange Admin Center: Go to admin.exchange.microsoft.com → Mailboxes. Confirm a mailbox was automatically created for your user. Mailboxes are created automatically when Exchange Online license is assigned.
  8. Verify Mailbox Access via Outlook: Browse to outlook.office.com and log in with the admin credentials. Confirm email can be sent and received.

You can reuse a single Azure App Registration for multiple Microsoft 365 workloads (Exchange, Teams, OneDrive). Simply add the required permissions for each workload to the same app. This reduces administrative overhead and simplifies secret rotation management.

Exchange Online Backup & Restore

Configuring Archive Jobs, Plans, and Mailbox Recovery

Commvault protects Exchange Online through Archive Jobs. The term "Archive" refers to the job type label used for all Exchange workloads — it does not automatically mean production mailbox data is moved or deleted. In a standard backup configuration, Commvault copies mailbox data to the storage library without removing it from the production mailbox. Whether data is actually moved and replaced with stubs (archiving) depends on the specific archiving policy configured — consult Commvault documentation for your version to confirm the behaviour for your use case. When stubbed data is accessed by a user, a recall job runs automatically to restore it to the production mailbox.

Understanding Archive Jobs vs. Backup Jobs

📦

Archive Job (Exchange Online & On-Prem Exchange)

The job type label used for all Exchange Online and on-premises Exchange workloads. In a standard backup configuration, mailbox data is copied to the storage library without deletion. Whether data is moved and stubbed depends on the archiving policy configured. All Exchange jobs run as Archive — there is no standard "backup" type for Exchange.

💾

Backup Job (Teams, OneDrive, SharePoint)

Standard backup job type. Copies data to the storage library without removing it from the source. Always incremental — first job captures all data, subsequent jobs capture changes only.

Creating the Microsoft 365 Plan

The Microsoft 365 Plan defines mailbox retention — how long archived mailbox data is kept in the Commvault storage library. This plan is distinct from the Server Plan.

  1. Navigate to Plans In Command Center: Manage → Plans → Create Microsoft 365 Plan
  2. Define Plan Name Use a meaningful name such as M365-ExchangeRetention-1Yr
  3. Set Retention Period Define how many days/months/years archived mailbox data is retained. Avoid setting to "Indefinitely" unless you have sufficient cloud or tape storage — it will cause uncontrolled storage growth.
  4. Optional: Archive Mailbox Settings Enable backup of archived mailboxes if required by your organization's compliance policy.
⚠️
Common Production Mistake: Indefinite Retention

Many organizations set Indefinite Retention in the Microsoft 365 Plan and then wonder why storage keeps growing despite having a short retention in the Server Plan. The M365 Plan retention overrides the Server Plan for mailbox data. Always set a definite retention value — use cloud or tape library if long-term retention is required for compliance.

Configuring the Exchange Online App in Command Center

  1. Navigate to Office 365 In Command Center: Protect → Office 365 → Configure Now → Exchange Online
  2. Provide App Name Enter a user-friendly name (e.g., ExchangeOnline-PoC)
  3. Select Server Plan Choose the Server Plan that defines storage location and backup frequency
  4. Configure Index Server Create or select an Index Server. The Index Server stores mailbox indexes enabling granular item-level search and restore. Designate the CommServe as the Index Server if no separate server is available.
  5. Select Access Node Select the Windows server where the Office 365 package is installed
  6. Enter Azure App Credentials Provide: Application ID, Application Secret (Value), Directory (Tenant) ID
  7. Verify Connection Click Verify Connection. Resolve any permission errors before proceeding.

Restore Options — Exchange Online

📬

Restore Mailbox (In-Place)

Restore the entire mailbox back to its original Microsoft 365 location. Supports overwrite option. Suitable for catastrophic mailbox loss.

✉️

Restore Specific Messages

Browse the indexed mailbox and restore individual emails. Useful for accidental deletion recovery — restores to original inbox location.

📁

Restore as PST File

Export mailbox or selected messages to a PST file. Useful for legal holds, e-discovery, or importing into other mail clients.

💡
Restore Decision Is the Administrator's Call

As the Commvault admin, you execute the restore as specified by the Office 365 Administrator. Clarify before restoring whether they need: in-place restore, out-of-place restore, specific message recovery, or PST export. Do not assume the restore type.

Section Summary — Exchange Online

  • All Exchange Online backup jobs are Archive jobs — not standard backup jobs
  • Job phases are Archive and Archive Index — no Scan or Backup phase
  • The Microsoft 365 Plan controls retention for archived mailbox data — takes precedence over Server Plan
  • Indefinite retention causes uncontrolled storage growth — always set a definite value
  • Discovery status "Not Available" usually indicates missing API permissions or an incorrect client secret
  • Always verify connection after initial configuration before running backups

Microsoft Teams Backup & Restore

Protecting Team Channels and Posts with Cloud Apps Agent

Microsoft Teams backup in Commvault captures team channels, posts, and files. Unlike Exchange Online, Teams uses the Cloud Apps agent and the job type is shown as a standard Backup (not Archive). Each Teams group is backed up as a channel unit.

Creating Teams in Microsoft 365

Before backing up Teams, create the team groups in the Microsoft 365 Admin Center or through the Teams client:

  • Admin Center → Teams & Groups → Create a new Team
  • Provide a team name, description, privacy setting (Public/Private), and add members
  • Each team has channels — these are the backup units in Commvault
  • The admin portal is accessible at admin.teams.microsoft.com
  • The Teams web client is accessible at teams.microsoft.com

Teams — Azure App Registration Differences

📝
Teams Requires a Redirect URI

Unlike Exchange Online (where the Redirect URI field can be left blank), the Teams app registration requires a Redirect URI. Enter the Command Center URL: https://[CommServeIPorHostname]/commandcenter. Use the IP address if hostname resolution might not be reliable.

Teams Backup Configuration

  1. Register Azure App for Teams Follow the same App Registration process, but ensure all Teams-specific permissions are added and admin consent is granted.
  2. Configure Teams App in Command Center Protect → Office 365 → Add App → Teams. Provide app name, storage plan, index server, and access node.
  3. Enter Azure Credentials Provide Application ID, Directory ID, and Application Secret (Value — not Secret ID).
  4. Verify Connection Configuration tab → Verify Connection. Resolve any missing permissions (common: ChannelSettings.ReadWrite.All).
  5. Add Content Browse and discover team channels. Select the channels to protect and assign the Microsoft 365 Plan.
  6. Run Backup The job type will display as Backup, and the agent will show as Cloud Apps.

Restore Options — Teams

🔄

Restore Channel (In-Place)

Restores the entire team channel back to the original Teams tenant. Recreates the channel structure. Posts can optionally be restored as posts in the Posts tab (Option 1) or as HTML files under the Files tab (Option 2).

📄

Posts Restored as HTML Files (Default)

Default behavior (Option 2): Posts are restored as HTML files in the channel's Files tab rather than appearing as live posts. To restore posts as live channel messages, explicitly select Option 1 before submitting the restore job.

From Commvault version 11.36 onwards, deleting a Teams application from the Command Center triggers an approval email to an administrator. OneDrive for Business Backup & Restore

Per-User File Storage Protection Using Existing App Registration

OneDrive for Business provides 1 TB of personal cloud storage per licensed user. Commvault backs up OneDrive using the Cloud Apps agent, and the same Azure App Registration used for Teams can be reused — you simply add the required OneDrive permissions to the existing app.

OneDrive Configuration Steps

  1. Add OneDrive Permissions to Existing App In the Azure portal, navigate to your existing app registration. Under API Permissions, add Files.ReadWrite.All from Microsoft Graph (Application permissions). Grant admin consent.
  2. Configure OneDrive App in Command Center Protect → Office 365 → Add App → OneDrive for Business. Provide app name, storage plan, index server, and access node.
  3. Enter Azure App Credentials Reuse the same Application ID, Directory ID, and Application Secret from the existing app registration.
  4. Verify Connection Check configuration tab → Verify Connection. All permissions should show green.
  5. Add Content — Select Users Choose All Users or specific users whose OneDrive should be protected. Each user's OneDrive is backed up as a separate content item.
  6. Assign Microsoft 365 Plan Select the plan that defines retention for OneDrive data.
  7. Run Backup Job type = Backup (Incremental). Agent = Cloud Apps.

Restore Options — OneDrive

👤

Restore User (Entire OneDrive)

Restores all files for a specific user's OneDrive back to its original location. Useful for bulk recovery after accidental deletion or ransomware.

📄

Restore Specific Files

Browse the indexed OneDrive content and select individual files or folders to restore. Supports restore to original location or an alternate file system path.

Backup Type Display — Important Clarification

All Microsoft 365 backup jobs display as Incremental in the job details, even the very first run. The first job captures all existing data (equivalent to a full backup) but is still labeled as Incremental. This is by design — Microsoft 365 workloads follow a Forever Incremental model.

💡
Filtering Large OneDrive Content

Commvault supports content filters for OneDrive backups, allowing you to exclude files by size, type, or modification date. This is useful when protecting many users with large OneDrive allocations. Consult the Commvault documentation for the specific filter syntax for your version.

SharePoint Online Backup & Restore

Certificate-Based Authentication for Site Collection Protection

SharePoint Online is a web-based platform for team collaboration, document management, and intranet sites. Commvault's SharePoint Online backup is unique because it requires certificate-based authentication in addition to the standard app credentials — making it the most involved of all the Microsoft 365 workloads to configure.

SharePoint Concepts

  • Team Sites: Collaboration spaces connected to Microsoft 365 Groups. Suitable for internal team use.
  • Communication Sites: Broadcast-style sites for publishing content to a broad audience.
  • Site Collections: A hierarchy of sites — each site collection has a root site and can have subsites.
  • Tenant Admin Site URL: Format is https://[tenantname]-admin.sharepoint.com

  • .CER file — Public key only. Upload this to the Azure App Registration under Certificates.
  • .PFX file — Contains both public and private key (password protected). Upload this to Commvault Command Center during SharePoint app configuration.
  • Delete the .CER file from the access node after uploading to Azure. Keep only the .PFX file for Commvault.

SharePoint App Configuration in Command Center

  1. Navigate to Add SharePoint Online App Protect → Office 365 → Add App → SharePoint Online
  2. Enter Tenant Admin Site URL Retrieve from the SharePoint Admin Center URL: https://[tenant]-admin.sharepoint.com
  3. Enter Azure App Credentials Application ID, Directory ID, and Application Secret (same as other workloads)
  4. Upload PFX Certificate Browse and upload the .PFX file generated in Step 3 above

  • Assign a Microsoft 365 Plan for retention

Restore Options — SharePoint Online

Restore Document

Commvault Workflow Automation

Creating Custom Workflows to Automate Repetitive and Complex Operations

Commvault Workflows are visual, configurable automation scripts that allow administrators to standardize repetitive tasks, enable self-service, reduce human errors, and enforce compliance without requiring end users to understand the Commvault interface. Workflows can be created for complex multi-step processes and exposed as simple single-click operations.

Why Use Workflows?

🔁

Automate Repetitive Tasks

MSP environments with hundreds of customers need the same onboarding steps for each client. A workflow can automate tenant creation, plan assignment, and user setup — replacing manual work that would otherwise require dedicated staff for each customer.

🧑‍💻

Self-Service for Application Teams

DBA teams, VMware admins, and application owners can run ad-hoc backups, restores, or readiness checks without Commvault training. You provide a workflow with limited scope — they click and execute.

🛡️

Reduce Human Errors

Standardized workflows enforce correct configuration every time. For example, always assigning a specific retention plan to new tenants without giving the user a choice of other plans.

🔗

Third-Party Integration

Workflows can include REST API calls to integrate Commvault with ServiceNow ticketing, cloud platforms, and custom portals — enabling automated alert escalation and ticket creation.

Predefined vs Custom Workflows

📦 Predefined Workflows

  • Provided by Commvault out-of-the-box
  • Available in Developer Tools → Workflows within Command Center
  • Also downloadable from store.commvault.com
  • Example: Enable Warm Storage Mode, Decommission Client, etc.
  • Can be used as-is or as templates for custom workflows

🛠️ Custom Workflows

  • Created from scratch in the Workflow Designer
  • Drag-and-drop activity-based design
  • Can include logic, conditions, loops, and email notifications
  • Tailored to specific organizational requirements
  • Deployed to CommServe and accessible in Command Center

Workflow Designer Overview

The Workflow Designer in Command Center (Developer Tools → Workflows → New Workflow) provides a visual canvas with three main areas:

Activities Panel (Left)

Predefined activities organized by category: Control Flow, Operations, Utilities, Q Commands. Drag activities onto the canvas to build the workflow.

Design Canvas (Center)

The work area where activities are placed and connected via transitions (arrows). Each arrow is a transition that can have conditions.

Properties Panel (Bottom)

Shows configuration options for the selected activity — display name, inputs, outputs, continue-on-error settings, and maximum restarts.

Building the Workflow — Step by Step

  1. Create New Workflow Developer Tools → Workflows → New Workflow. The Start activity is automatically added. In the Properties panel, enter the Workflow Name (e.g., CheckReadiness-WF) and description.
  2. Define Input Variable In Inputs, add variable: Name = ClientName, Type = Client Entity. Leave the value blank so Commvault dynamically fetches all accessible clients for the drop-down.
  3. Add Check Readiness Activity From Activities → Operations, drag Check Readiness onto the canvas. Set Display Name, and in the Inputs section select the input source as Workflow Inputs → ClientName. Set Continue on Error = True so the workflow proceeds even on failure to send the notification.
  4. Add Decision Activity From Control Flow, drag Decision onto the canvas. Label it Report Ready?. In the Expression, enter the condition: Exit code of Check Readiness = 0 (success).
  5. Add Success Email Activity From Utilities, drag Send Email onto the canvas. Label it Success Email. Configure: To = Workflow Executor (Username), Subject = Client Readiness Passed: [ClientName], Body = HTML report variable.
  6. Add Failure Email Activity Add another Send Email activity. Label it Failure Email. Configure: To = Workflow Executor, Subject = ALERT: Readiness Check Failed: [ClientName], Body = include Job ID variable so the executor can pass this to the Commvault admin for investigation.
  7. Add Workflow End Activity Add a Workflow End activity at the conclusion of both the Success and Failure paths. This is required — without Workflow End, the job will remain in Pending state and never complete successfully.
  8. Configure Transitions (Connectors) Draw transitions: Start → Input → Check Readiness → Decision. Then Decision (True) → Success Email → Workflow End. Decision (False) → Failure Email → Workflow End.
  9. Save and Deploy Click Save, then Deploy. Select the CommServe as the Workflow Engine. The workflow is now available in Command Center under the Workflows section.

Commvault REST API

API Fundamentals, Authentication, and Postman Integration for Commvault Administrators

Commvault provides a comprehensive REST API that allows programmatic access to all backup operations, job management, client management, and reporting functions. As a Commvault administrator or implementation engineer, REST API knowledge is essential for troubleshooting API integrations, supporting third-party tool integrations, and enabling automation scenarios.

Why REST API Knowledge Matters for Admins

🔍

Troubleshooting API Issues

When a third-party tool reports Commvault API failures, use Postman to reproduce the exact API call. If results are correct, the issue lies with the third-party tool or network — not Commvault.

🔗

Third-Party Integration

Provide ServiceNow, monitoring tools, or custom dashboards with the correct API endpoints to pull backup job statuses, alerts, and storage metrics from Commvault.

🤖

Private Cloud Automation

When customers build private cloud portals to self-manage their backups, you provide the correct API endpoints for operations like running backup, assigning plans, and triggering restores.

📊

Reporting Automation

Pull job summaries, client properties, storage utilization, and SLA data through APIs and integrate with BI tools or custom dashboards for operational visibility.

REST API Core Concepts

🌐 Endpoint

The base URL for all API calls. For modern Commvault: https://[hostname]/commandcenter/api. Port 443 only in v11.x+.

📂 Resource

The path appended to the endpoint identifying the operation. Example: /login, /Client, /Job, /Subclient. The resource specifies what you are acting upon.

⚡ HTTP Method

The action verb. In Commvault: GET (read), POST (create/trigger), PUT (update/modify), DELETE (remove). POST covers most operations, but PUT is used in certain update endpoints — always check the API reference for the correct method.

🏷️ Header

Metadata sent with each request. Defines content type (application/json or application/xml) and includes the auth token after login.

Authentication — Token-Based Login

Every Commvault REST API session starts with a /login request. The CommServe responds with an auth token that must be included in the header of all subsequent requests. The token expires (default: 30 minutes, configurable up to 60+ minutes).

POST — Login Request (JSON)
# Method: POST # Resource: /login # Endpoint: https://[CommServeIP]/commandcenter/api/login # Header: Content-Type: application/json Accept: application/json # Body (JSON): { "username": "cvadmin", "password": "Y3ZhZG1pbjEyMw==", ← Base64 encoded password "timeout": 60 } # Successful Response (200 OK): { "token": "QSDK eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9..." } # Save this token — include in all further request headers
📝
Password Must Be Base64 Encoded

The login API requires the password in Base64 encoded format — not plain text. Use an online encoder (e.g., base64encode.org) or a Postman pre-request script to encode it. Username can be entered as plain text.

HTTP Response Codes

✅ 200 OK

Request was accepted and processed successfully. For a backup trigger POST, 200 means the job was initiated — not that the backup succeeded. Run a separate GET Job call to check job completion status.

⚠️ 401 Unauthorized

Invalid or expired credentials. Either the username/password is wrong during login, or the auth token has expired in subsequent requests. Re-authenticate to get a new token.

❌ 404 Not Found

The resource URL is incorrect, or the resource does not exist. Check the endpoint URL and resource path. Also check if the Client ID or Job ID exists.

🔒 403 Forbidden

The authenticated user does not have permission to perform the requested action. Check the Commvault user's role and associated capabilities.

🎓
Interview Tip: 200 Does NOT Mean Job Success

A common misconception: HTTP 200 from a backup trigger API means the API request was accepted and the job was queued. It does not mean the backup job completed successfully. To verify job completion, you must poll the GET /Job/[JobID] endpoint and check the job status field.

Scenario-Based Interview Q&A

Common Questions on Microsoft 365 Backup, Workflows, and REST API

🎓
Exam Tip

These questions reflect real-world scenarios you will encounter in Commvault implementation and support roles. Review the answers and understand the reasoning — memorizing answers without understanding the concepts will not help in practical scenarios.

Microsoft 365 Backup Questions

Your Exchange Online backup job completed successfully, but the storage utilization on the backup library is not decreasing despite a 1-month retention policy in the Server Plan. What is the likely cause?
The Microsoft 365 Plan assigned to the Exchange Online app has retention set to Indefinitely. The M365 Plan retention takes precedence over the Server Plan retention for Exchange mailboxes. The Server Plan's 1-month retention is not honored for mailbox data. To fix this, edit the M365 Plan and set a definite retention value (e.g., 12 months). Communicate to the customer that cloud or tape storage may be needed if compliance requires long-term retention.
After configuring Exchange Online backup in Commvault, the discovery status shows "Not Available" and displays the error: "Azure apps with insufficient permissions or MFA is enabled." How do you troubleshoot this?
1. Navigate to the Exchange Online app → Configuration → Verify Connection. The error details will specify which permission is missing or if credentials are incorrect. 2. If a specific permission is missing (e.g., Directory.Read.All, Group.Read.All), add it in the Azure portal under the app's API Permissions and re-grant admin consent. 3. If the error is "invalid client secret," regenerate a new client secret in the Azure portal, copy the Value column (not the Secret ID), and update the Commvault configuration. 4. Re-run Verify Connection until all permissions are authorized.
A customer wants to protect their Exchange Online, Teams, and OneDrive with Commvault. Do they need three separate Azure App Registrations?
No. A single Azure App Registration can be reused for all three workloads. The key is to add all the required permissions for each workload (Exchange, Teams, OneDrive) to the same app, and grant admin consent for all. This simplifies management — only one Client Secret to rotate and one App ID to track. Teams requires a Redirect URI in the app registration, so ensure this is configured. SharePoint Online requires a separate certificate-based configuration but can also reuse the same app registration.
Why does the Exchange Online backup job show as an "Archive" type rather than "Backup"? Does this mean data is being deleted from the mailbox?
There are two separate things to understand here — the job type label and the actual operation being performed.
Why "Archive" instead of "Backup"? Commvault uses the label Archive for all Exchange and Exchange Online jobs — this is simply how Commvault classifies the job type for Exchange workloads. It is a fixed label, not a description of what happens to the data.
Does it delete data from the mailbox? No — not by default. In a standard backup configuration, Commvault reads mailbox data and copies it to the storage library. The original emails remain untouched in the production mailbox. Nothing is moved or deleted.
When does data actually get removed? Only if the customer has specifically configured a Commvault archiving policy. In that case, emails meeting the archiving criteria (e.g., older than X days) are moved to the Commvault storage library and replaced with a stub — a lightweight pointer in the mailbox. When a user clicks a stubbed email, a recall job retrieves the original message automatically. This is an active customer choice, not a default behaviour.
In summary: The "Archive" job label does not mean data is deleted. Whether mailbox data is moved and stubbed depends entirely on whether an archiving policy has been configured. If unsure, check the subclient archiving settings or consult Commvault documentation.
During SharePoint Online backup configuration, what is the purpose of the .CER and .PFX certificate files, and where is each used?
The .CER file contains only the public key of the self-signed certificate. It is uploaded to the Azure App Registration under Certificates to establish trust between Commvault and SharePoint Online. The .PFX file contains both the public and private keys (password protected) and is uploaded to the Commvault Command Center during SharePoint Online app configuration. Commvault uses the private key in the PFX to authenticate with SharePoint Online. Both files are generated from the same PowerShell commands run on the Access Node.
A SharePoint site was deleted and a restore was attempted. The restore fails with "URL exists in the recycle bin." How do you resolve this?
Microsoft retains deleted SharePoint sites in its own recycle bin for 93 days before permanent deletion. During this period, you cannot restore via Commvault to the same URL because the URL is considered "in use" by the recycle bin entry. Options: 1) Restore the site from Microsoft's own recycle bin in the SharePoint Admin Center (admin.sharepoint.com → Active sites → Deleted sites) to make the site active, then overwrite it using Commvault. 2) Wait for the 93-day Microsoft retention to expire, then perform the Commvault restore. You cannot manually override this Microsoft behavior.

Workflow Questions

A workflow was built and deployed, but all workflow jobs remain in "Pending" state after the check readiness activity completes. What is the likely cause?
The workflow is missing the Workflow End activity. Without a Workflow End activity at the conclusion of each execution path, Commvault does not know when the workflow is complete and leaves the job in Pending state indefinitely. Add the Workflow End activity as the final step in each branch (success path and failure path), reconnect the transitions, and redeploy the workflow.
An MSP with 200 customers wants to automate tenant onboarding in Commvault. What workflow components would you recommend and what advantages does this provide?
Design a Customer Onboarding workflow with: (1) Input variables for customer name, storage plan, and retention requirement. (2) Activities to create a new CommCell tenant/company, assign a storage plan, create access user accounts, and configure default backup settings. (3) Email notification to the technical account manager with onboarding summary. Advantages: eliminates the need for a dedicated admin per customer, ensures consistent configuration standards across all customers, reduces human errors from manual setup, and reduces onboarding time from hours to minutes.

REST API Questions

A REST API developer reports that the Commvault API for pulling job summaries is returning incorrect data. As the Commvault admin, how do you verify whether the issue is with Commvault or the developer's code?
Open Postman and execute the same API call (same endpoint, same parameters, same auth token) that the developer is using. If Postman returns the correct, expected data, the issue is not with Commvault — it is with the developer's code, network configuration, or how they are parsing the JSON response. Show the Postman results to the developer as evidence and ask them to investigate their implementation. If Postman also returns incorrect data, raise a Commvault support case with the API call details and response body.
What is the difference between an HTTP 200 response to a backup trigger API and a successful backup job? How do you determine if the backup job actually succeeded?
HTTP 200 from a backup trigger POST request means the API request was received and a backup job was initiated — it does not indicate job completion or success. The response body will contain the jobId of the created job. To check if the backup job succeeded, you must run a separate GET /Job/[JobID] call and inspect the job status field in the response. Wait until the status transitions to "Completed" with a success state before confirming a successful backup.
A customer is building a private cloud portal and wants to trigger Commvault backups, assign plans to clients, and check job status through the portal. What is your role as the Commvault admin in this scenario?
As the Commvault admin, your role is to provide the correct API endpoints to the portal development team. This includes: the endpoint URL and authentication method, the POST API for triggering backups with the correct request body format, the POST/PUT API for assigning a plan to a client or subclient, and the GET API for polling job status. Use Postman to test and validate each API before providing it to the developers. You are not expected to write the portal code — that is the developer's responsibility. Verify that each API returns the expected response before handing over to the development team.
