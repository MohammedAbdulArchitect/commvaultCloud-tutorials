What is IntelliSnap?
Commvault's snapshot technology — automating application-consistent snapshots across SAN and cloud storage

Definition and Purpose

IntelliSnap is a Commvault feature that automates the creation of application-consistent snapshots on storage arrays and cloud storage. It bridges the gap between raw storage-level snapshots (which are crash-consistent) and true application-consistent backups by coordinating with the application before the snapshot is triggered.

🎯 The Core Problem IntelliSnap Solves

When a storage team manually triggers a snapshot of a LUN (Logical Unit Number) hosting a running Oracle or SQL Server database, the snapshot is taken without notifying the application. The database may have data in memory, pending transactions not yet written to disk, and open I/O operations. The resulting snapshot is crash-consistent — as if the server lost power. The database may not start cleanly after restore. IntelliSnap solves this by quiescing (freezing) the application first, then triggering the snapshot — guaranteeing application consistency.

Where IntelliSnap Is Used

✓ SAN Storage (NetApp, Hitachi, HP, IBM, Pure Storage...) ✓ Cloud Storage (Azure, AWS, GCP) ✗ Virtualized files (VMDK, VHD) — NOT supported
📘 SAN or Cloud Only

IntelliSnap requires the underlying storage to support snapshot technology — either a physical SAN array (NetApp, Hitachi Vantara, Pure Storage, HP 3PAR, etc.) or a cloud platform (Azure Managed Disks, AWS EBS, GCP Persistent Disks). It cannot be used with virtualized file-based storage like VMware VMDK files on a datastore, or with NFS-backed storage that doesn't support hardware snapshots.

Key Advantages of IntelliSnap

  • Speed: A 100 TB LUN can be snapshotted in minutes — time is independent of data volume. The snapshot creation is near-instantaneous on modern arrays.
  • Application Consistency: Commvault quiesces the application (via agent, VSS, or pre/post scripts) before triggering the snapshot, ensuring the database starts cleanly after restore.
  • Zero Impact on Production: Once the snapshot is created, the backup copy reads from the snapshot — not the live production volume. The production server is completely unaffected during the data transfer phase.
  • Automation: Storage admins previously had to manually create and manage snapshots. IntelliSnap automates the entire lifecycle — creation, catalog registration, retention, backup copy, and expiry.
  • Granular Recovery: Snapshots support guest file restore and application-level recovery (database, mailbox, etc.).

IntelliSnap vs. Streaming Backup — Key Differences

🌊 Streaming Backup (Standard)
  • Data is read from the live production volume and sent over the network
  • Backup size = actual data volume written to cloud/disk library
  • Longer backup window for large datasets
  • Production server experiences I/O load during backup
  • Single copy in one location (library)
  • Job in Commvault: single backup job
📸 IntelliSnap (Snapshot-Based)
  • Snapshot created on storage array in seconds/minutes
  • Only metadata (~15 KB) written to library — actual data stays on array
  • Fast initial protection regardless of dataset size
  • Zero I/O impact on production after snapshot is taken
  • Two copies: snap copy (on array) + backup copy (on library)
  • Jobs: snap backup job + backup copy job spawned automatically

IntelliSnap Backup — Detailed Phase Breakdown
1
Application Quiesce (Freeze)

Commvault instructs the application to quiesce — freeze all I/O and flush in-memory data to disk. How quiesce happens depends on context: if a Commvault agent (SQL, Oracle, Exchange) is installed → agent quiesces the application directly. If no agent (agentless VM) → VSS (Windows) or pre/post scripts (Linux) are used. After quiesce, the application is in a known-good consistent state.

2
API Request to Storage Array — Trigger Snapshot

While the application is quiesced, Commvault sends an API request to the storage array (e.g., NetApp ONTAP API, Azure Disk Snapshot API) to create a snapshot of the target LUN or disk. The storage array creates the snapshot instantly — this is a metadata-level operation on most modern arrays, taking seconds regardless of volume size.

3
Application Un-quiesce (Resume)

As soon as the snapshot is confirmed created, Commvault un-quiesces the application — I/O operations resume immediately. The quiesce window is typically only a few seconds. The production application experiences minimal freeze time.

4
Snapshot Catalog Registration (Metadata Only)

Commvault records the snapshot in its catalog — the CommServe database stores metadata about the snapshot: which volume, on which array, at what time, with which job ID. The actual snapshot data remains on the storage array — it is NOT copied to the library at this stage. The library receives only ~15 KB of metadata. This is why snap copy jobs show a tiny media size (e.g., 15 KB) while primary copy jobs show the full data size (e.g., 1 GB).

5
Backup Copy (Optional but Recommended)

A second child job — the backup copy job — is automatically spawned. It mounts the snapshot on the media agent and copies the actual data from the snapshot (not the live volume) to the cloud library or disk library. This creates a traditional streaming backup from the snapshot, enabling long-term retention beyond what the storage array can hold. Production I/O is unaffected since the backup reads from the snapshot, not the live volume.

What You See in the Job Controller — Two Jobs Per Backup

When you trigger an IntelliSnap backup, you will always see (at minimum) two jobs:

Job 3012 VM Admin Snap Backup — the parent job you triggered. Creates the snapshot on the storage array and writes metadata to the snap copy. Snap Backup
Job 3015 Backup Copy — automatically spawned child job. Reads from the snapshot, copies full data to the cloud library (primary copy). Backup Copy
📘 Operation Type in Job Controller

When IntelliSnap is enabled, the job controller shows the operation type as "VM Admin Snap Backup" instead of the usual "VM Admin Backup." This confirms that IntelliSnap mode is active. If you see plain "VM Admin Backup," IntelliSnap is not enabled on that subclient.

What Appears on the Azure Portal During an IntelliSnap Backup

For Azure VM IntelliSnap backups, you can verify the snapshot was created in the Azure Portal. Navigate to Search → Snapshots. You will see a snapshot entry with the naming format:

Azure Snapshot Naming Convention
# Format: {JobID}_prodvm-os-disk_Commvault_snap 3013_prodvm-os-disk_Commvault_snap # 3013 = the VM-level job ID (child of 3012 parent) # prodvm-os-disk = OS disk name of the backed-up VM # Commvault_snap = Commvault naming suffix # Type shown: snap (distinguishes from backup copy) # One snapshot per disk — if VM has 2 disks, 2 snapshots appear

Production VM Setup — Installing PostgreSQL (Ubuntu 22.04)

The production VM simulates a real-world database workload. PostgreSQL is installed to demonstrate application consistency after IntelliSnap restore.

bash — PostgreSQL Installation and Setup
# Update package repositories first sudo apt-get update sudo apt-get upgrade -y # Install PostgreSQL (version 14 on Ubuntu 22.04) sudo apt-get install -y postgresql postgresql-contrib # Verify service is running systemctl status postgresql # Expected: active (running) # Switch to postgres user and access psql shell sudo -i -u postgres psql # Create a test database and user inside psql CREATE DATABASE testdb; CREATE USER backupuser WITH PASSWORD 'mypassword'; GRANT ALL PRIVILEGES ON DATABASE testdb TO backupuser; \q # Allow remote connections — edit postgresql.conf # Find: #listen_addresses = 'localhost' # Change to: listen_addresses = '*' sudo nano /etc/postgresql/14/main/postgresql.conf # Allow remote connections — edit pg_hba.conf # Find IPv4 local connections line, change 127.0.0.1/32 to 0.0.0.0/0 sudo nano /etc/postgresql/14/main/pg_hba.conf # Restart PostgreSQL after config changes sudo systemctl restart postgresql # Allow PostgreSQL port in OS firewall sudo ufw allow 5432/tcp # Test connection with backupuser psql -h 127.0.0.1 -U backupuser -d testdb # Enter password when prompted
💡 Why Set Up a Real Database?

The purpose is to simulate an actual production environment — a VM with a running database — so that after the IntelliSnap restore, we can verify that PostgreSQL starts cleanly, the service shows "active (running)", and we can connect to the database. This validates that the snapshot was application-consistent. A VM with no application running cannot demonstrate this advantage of IntelliSnap.

Authentication — Backup Contributor vs. Custom Role

For IntelliSnap Azure VM backups, authentication follows the same Managed Identity approach as standard Azure VM backup. However, there is an important lesson from the lab:

⚠ Azure Built-in "Backup Contributor" Role — Insufficient for VM Browsing

The Azure built-in Backup Contributor role was initially assigned to the access node's managed identity. While it appears to cover backup operations, it does not include sufficient permissions to list and browse virtual machines in the subscription from Commvault. When configuring the hypervisor VM Group content, the VM list shows "No items found." The fix: switch from the built-in Backup Contributor role to a Commvault Custom Role using the JSON file provided in Commvault's documentation. The custom role has exactly the right set of permissions — including VM discovery, disk snapshot creation, and backup operations.

Two Managed Identity Methods — System Assigned vs. Custom Role

🔵 System Assigned Managed Identity
+ Built-in Role (e.g., Backup Contributor)
  • Enable System Assigned Identity on the access node VM
  • Assign an Azure built-in role at subscription scope
  • Quick to configure — no JSON editing
  • Risk: built-in roles may lack some permissions (as seen in lab — VM browsing failed)
  • Use: exploratory/lab environments where permission gaps are acceptable
🟣 System Assigned Managed Identity
+ Commvault Custom Role (JSON)
  • Enable System Assigned Identity on the access node VM
  • Create custom role using Commvault's provided JSON file
  • Paste subscription ID into JSON before creating
  • Assign the custom role at subscription scope to the access node
  • Commvault-tested, includes all required permissions
  • Use: production environments — recommended approach
⭐ Always Use Commvault's Custom Role JSON

Commvault's documentation publishes a specific JSON file for each protected workload type (Azure VMs, Azure SQL, etc.). These JSON files contain exactly the permissions tested and validated by Commvault engineering. Using Azure built-in roles or AI-generated JSON can result in missing permissions causing subtle failures — like VM browsing working but snapshot deletion failing, or backup succeeding but restore failing. Always download and use the JSON from the Commvault documentation article "Roles and Permissions Required for Protecting Azure Resources with Commvault."

Snap Copy vs. Backup Copy — Understanding Both
What data each copy holds, where it lives, and how to view jobs from different perspectives

The Two-Copy Architecture

Property Primary Snap Copy Primary Copy (Backup Copy)
What is stored Metadata only (~15 KB) — snapshot catalog entry Full VM/application data (~1 GB or more)
Where data lives Snapshot remains on the storage array (Azure, NetApp, etc.) Full data written to cloud library / disk library
Retention recommendation Short: 1–7 days (snapshot occupies array capacity) Longer: weeks/months (as per customer SLA)
Recovery speed Very fast — restore directly from array snapshot Slower — reads full data from library
Job type Snap backup job (spawns automatically) Backup copy job (spawns automatically from snap job)
View in List Snapshot Right-click Primary Snap copy → List Snapshots: shows snapshot details No snapshots to list — standard backup data

📘 Subclient-Level Browsing Limitation

When viewing backup history from the subclient level, only snap backup jobs are shown by default. To see backup copy jobs, go to Backup History → Advanced → select "Include Backup Copy jobs." At the storage policy copy level (right-click Primary Copy → View Jobs), backup copy jobs are visible by default without needing to select "advanced."

⭐ Retention Planning for Snap Copy

Never set a long retention on the Snap Copy. Snapshots consume storage array capacity — a 100 TB volume snapshot that grows 5% per day will consume 5 TB after one week. Typical best practice: 3–7 days for snap copy retention. For longer retention, rely on the Backup Copy (primary copy) which is written to the library and is more cost-efficient for long-term storage. The snap copy provides fast recovery for recent events; the backup copy provides long-term retention.

Viewing Jobs from Three Different Entry Points

The same IntelliSnap backup job can be viewed from three perspectives in Commvault, each showing slightly different information:

From VM Group / Subclient Shows snap backup jobs only by default. Access backup copy jobs via Backup History → Advanced → Include Backup Copy. Good for day-to-day job monitoring. Default View
From Virtual Machine (Client) Shows backup jobs for that specific VM. To see VMs in Client Computers view, enable: right-click Client Computers → Custom View → Show Virtual Machines (agentless VMs are hidden by default) VM-Level View
From Storage Policy Copy Right-click Primary Snap → View Jobs: shows snap jobs. Right-click Primary Copy → View Jobs: shows backup copy jobs directly — no "Advanced" selection needed. Also shows application size vs. media size comparison. Most Detailed View

IntelliSnap Restore — From Snap Copy and Backup Copy
Restoring from snapshots, staging storage account, and verifying application consistency

Restore from Snap Copy (Fast Recovery)

Azure VM IntelliSnap — In-Place Restore from Snap Copy
1
Navigate to Restore: Storage Policy → Primary Snap → View Jobs

From the Primary Snap copy → right-click → View Jobs to see the snap backup jobs. Select the job to restore from. Alternatively, navigate via VM Group → Backup History → select the snap backup job.

2
Browse and Restore → Full VM → In-Place

Click Browse and Restore. Select the VM (e.g., ProdVM). Select "Recover All Selected." Choose In-Place restore. Select Power On after restore and Unconditionally Overwrite.

3
Select Staging Storage Account (Mandatory for Azure)

Azure VM IntelliSnap restore requires a staging storage account. Commvault uses this to temporarily import the managed disk before recreating the VM. Select an existing Azure storage account from the dropdown. Without this, the restore job will fail with an error about staging path.

4
Monitor Restore — VM Deleted and Recreated

In-place restore deletes the existing VM and recreates it from the snapshot. During this process, the VM in Azure Portal shows as "stopped" or "deallocated." After restore completion, refresh the portal — the VM appears as "Running." Note: the Public IP may remain the same for Azure IntelliSnap restores (unlike streaming restore where it changes).

5
Verify Application Consistency

SSH into the restored VM. Run systemctl status postgresql — status should show active (running). Connect to the database: psql -h 127.0.0.1 -U backupuser -d testdb. If the database connects and responds to queries, the IntelliSnap was application-consistent — the database is in a clean state without needing recovery mode.

💡 Restoring from Backup Copy When Snap Has Expired

After the snap copy retention expires (e.g., 1 day), the snapshot is deleted from the Azure storage array. However, the backup copy in the primary copy is still available. You can restore from the backup copy exactly the same way — Browse and Restore from the Primary Copy. This proves that even when the snapshot is gone, data is still restorable from the library. This dual-copy architecture ensures you always have at least one recovery path available.

Spool Copy & Data Aging
How spool retention works, when snapshots get cleaned up, and how to manually trigger data aging

Understanding Spool Copy Retention

Retention for the Snap Copy can be set in two ways:

📅 Fixed Day Retention (e.g., 1 Day)
  • The snapshot stays on the storage array for exactly 1 day (24 hours)
  • After 1 day, the next data aging job marks it for deletion
  • Data aging runs once every 24 hours by default
  • The snapshot remains visible in List Snapshots until data aging runs
  • Use when: you want a short but defined snap retention window
🔄 Spool Copy (No Fixed Retention)
  • The snapshot is retained on the storage array until the backup copy completes
  • Once the backup copy job finishes writing to the library, the snapshot is immediately marked for expiry
  • The next data aging job deletes it from the array
  • Purpose: keep snapshots only as long as needed for backup copy — then clean up
  • Use when: you only need the snapshot as an intermediate step, not for direct restores
📘 Why Snapshots Don't Disappear Immediately After Expiry

Even after setting the retention to "Spool Copy" and confirming the backup copy completed, the snapshot does not disappear immediately from the Azure Portal or from Commvault's List Snapshots view. This is because Commvault's data aging job only runs once every 24 hours by default. It is the data aging job that actually triggers the deletion of expired snapshots from the storage array. If you want immediate cleanup, you must manually run a data aging job from the CommCell Console (right-click CommServe → Data Aging). After the manual data aging run, the expired snapshots will be cleaned from both the storage array and the Commvault catalog.

Running Data Aging Manually

Commvault Console — Manual Data Aging
# In CommCell Console (Java Console): # 1. Right-click the CommServe (top node) # 2. Select "Data Aging" # 3. This manually triggers the data aging job # What happens during data aging: # - Commvault checks all backup jobs in the database # - Identifies jobs whose retention period has expired # - For snap copies: sends delete request to the storage array API # - Removes the job records from the Commvault catalog # - Azure storage array snapshots are deleted from Azure Portal # After data aging runs successfully: # - Primary Snap copy → Right-click → View Jobs: shows no jobs # - Primary Snap copy → Right-click → List Snapshots: shows no snapshots # - Azure Portal → Snapshots: the Commvault_snap entry is gone # - But Primary Copy still has the backup copy job — restore still possible!
⭐ Key Insight — Snap Gone, Restore Still Possible

After the snap copy is aged out (deleted), the primary snap copy shows no jobs and no snapshots. However, the backup copy job in the primary copy remains. You can still browse and restore from the backup copy — just do it from the Primary Copy view instead of the Primary Snap Copy view. This demonstrates the value of the dual-copy architecture: snap for fast short-term recovery, backup copy for long-term data protection.

Azure SQL Database (PaaS) — Overview
IaaS vs. PaaS SQL, what Azure manages vs. what you manage, and why you still need Commvault

IaaS SQL vs. PaaS SQL — The Critical Distinction

🖥 IaaS — SQL on a Virtual Machine
  • You create a VM, install SQL Server on it
  • You have full OS access (RDP/SSH)
  • You manage the VM, OS patching, SQL patching
  • You can install Commvault SQL agent directly on the VM
  • Commvault connects directly to the SQL instance
  • Examples: Azure VM with SQL, AWS EC2 with SQL Server
☁ PaaS — Azure SQL Database (Managed Service)
  • You only create the SQL database/server — no VM to manage
  • No OS access — Azure manages the underlying VM completely
  • Azure handles OS patching, SQL patching, HA, backups internally
  • You cannot install any agent on the underlying VM
  • Commvault uses an external access node with special permissions
  • Also called: Managed SQL, Azure SQL Managed Instance
📘 Why Use Commvault When Azure Already Backs Up SQL?

Azure SQL provides its own native backup (point-in-time restore, geo-redundant backups). However, many customers also use Commvault for Azure SQL backup because: (1) Enterprise policy requires all data — regardless of platform — to be protected under a single backup tool for unified reporting and compliance. (2) Commvault gives granular control over retention periods, RPO, and recovery options. (3) It allows restoring Azure SQL backups to on-premise SQL servers or different regions. (4) Regulatory requirements may mandate third-party backup verification. The choice is up to the customer — they can use Azure native, Commvault, or both.

Azure SQL Database — Key Resources Created

SQL Server (Logical) The container/namespace for SQL databases. Has a globally unique hostname like sqlserver0615.database.windows.net. The underlying VM is completely hidden from users — you cannot see it in Azure Portal's VM section. Logical Container
SQL Database The actual database within the SQL Server. One SQL Server can host multiple databases. Resource type appears in resource group as "SQL database." Can import a sample AdventureWorks database at creation time. Actual DB
Authentication SQL Authentication (username + password) — these credentials are for the SQL server, not for any VM. The VM where the DB is hosted does not accept these credentials. SQL Credentials Only
Free Tier Limitation Azure SQL free tier (100K vCore seconds, 32 GB RAM) does NOT allow database copy/export operations, which Commvault uses internally during backup. The free tier causes "Pole operation status return failed" (also seen as "Poll operation") errors. Solution: disable the free tier and use the paid compute tier (~$6/month). Free Tier Blocks Backup
Networking / Firewall By default, Azure SQL Server blocks all connections. You must add firewall rules: "Allow Azure services" and/or specific IP rules. At creation time, you can add your client IP automatically. The access node's IP must be whitelisted to allow Commvault to connect on port 1433. Firewall Rules Required
📘 Command Center Only — No Console Console Option

Azure SQL PaaS backup configuration can only be done from the Command Center (web UI). The classic CommCell Console (Java console) does not have an option to configure Azure SQL backup. This is part of Commvault's strategic direction — all new cloud workloads (Kubernetes, Azure SQL, Azure VM, GCP) are being developed for Command Center only. The Java console is being maintained for legacy operations but new features are Command Center exclusive.

Azure SQL Database — Configuration
Access node setup, custom role for SQL, Commvault virtual client creation, and troubleshooting ODBC

Access Node for Azure SQL — Why a Separate Node?

Since the underlying VM of Azure SQL PaaS is managed by Azure and not accessible to users, you cannot install a Commvault agent directly on it. Instead, you deploy a separate access node (proxy) that has permissions to access the SQL database via the network (port 1433). The access node uses its managed identity to authenticate to Azure and to the SQL database.

Troubleshooting — Common Azure SQL Backup Errors

⚠ Error 1: "Database [name] on server is not currently available"

Root Cause: The ODBC driver is not installed on the Windows access node. Commvault's SQL agent cannot connect to Azure SQL Database without the Microsoft ODBC Driver for SQL Server.
Fix: Download and install: (1) Microsoft Visual C++ Redistributable (required dependency), then (2) Microsoft ODBC Driver for SQL Server. Both must be installed before resuming the backup job. After installation, resume the existing job — do not resubmit a new job to avoid cluttering logs.

⚠ Error 2: "Pole operation status return failed"

Root Cause: The Azure SQL Database is on the free tier. The free tier restricts database copy/export operations, which is exactly what Commvault triggers internally during backup (it creates a dump/copy of the database for export).
Note: The exact error wording may appear as "Pole operation" or "Poll operation" depending on your Commvault version — both refer to the same issue.
Fix: In Azure Portal → SQL Database → Compute + Storage → disable the "free database" option and switch to paid compute (approximately $6/month for basic tier). After disabling free tier, kill the failed job and resubmit a new backup job.

⚠ Error 3: Push Install Fails on Windows Access Node

Root Cause: Windows Defender Firewall is blocking remote installation ports (135, 139, 445). Even if NSG ports are open in Azure, the Windows OS-level firewall still blocks these connections.
Fix: (1) Open NSG inbound rules for 135, 139, 445 (temporary). (2) Disable Windows Defender Firewall in Server Manager → Local Server → Defender Firewall → Off for Private, Public, and Domain profiles. After installation completes, re-enable firewall and remove temporary NSG rules. Port 8400 and 8403 must remain open permanently.

What Commvault Does During Azure SQL Backup

Commvault's Azure SQL backup follows a specific internal process that differs from traditional SQL agent backup:

Step 1 — Connect Access node connects to Azure SQL Server on port 1433 using the SQL authentication credentials provided Via ODBC
Step 2 — Export (Copy) Commvault triggers a database copy/export operation internally — similar to a BACPAC export. This is why the free tier blocks it (copy operations are not permitted on free tier). DB Copy/Export
Step 3 — tagfs Transfer After export, Commvault uses its internal "tagfs" technology to transfer the exported database data to the cloud library. Log entry shows: "starting to export... using tagfs" Internal Transfer
Staging Path (Restore) During restore, a staging path (local folder on access node) is required. Commvault restores the database dump to this folder first, then imports it back to Azure SQL. Must provide a valid path on the access node (e.g., C:\CVRestore\) Mandatory for Restore

Upgrade & Maintenance Best Practices
Critical guidance on Commvault version management, maintenance releases, and data integrity validation

The Real Cost of Skipping Maintenance Releases — A Real-World Example

A critical lesson from production experience that every backup admin must internalize:

⚠ Real Incident: Backups Completing Successfully — But VMs Won't Restore

  • Customer was on Commvault 11.32 — a supported version — but had not installed any maintenance releases (running on a very low MR number)
  • VMware backup jobs ran and showed "Completed" status in the Job Controller every day for months
  • When a restore was needed, every restored VM showed a Blue Screen of Death — the VM would not start
  • VMware team and OS team both tried to troubleshoot — no success
  • Commvault support was engaged — Commvault acknowledged a known bug in that old MR version where restore completes but VM boots to blue screen
  • Resolution: upgrade to a newer MR — but existing backups taken on the old version were NOT restorable
  • Customer had to wait until new backups were taken after the MR upgrade to have restorable backups
  • Business impact: complete data protection gap for all existing backup jobs
⭐ The N-1 Rule for Commvault Versions
  • Platform/Feature Release (e.g., 11.32, 11.36, 11.40): These are major releases. Do NOT upgrade to the latest platform release immediately — it may be unstable. Follow the N-1 rule: use one version behind the latest. For example, if 11.40 just released, stay on 11.36 until 11.40 is proven stable.
  • Maintenance Release (MR) (e.g., 11.32.100, 11.32.150): These are hotfixes/patches. Always keep MRs current. There is very little risk in applying the latest MR for your platform release — they contain critical bug fixes like the restore failure described above. An outdated MR can silently cause data loss.
  • OS Compatibility Check: Before any platform upgrade (e.g., 11.32 → 11.36), verify all OS versions in your environment (CommServe OS, Media Agent OS, client OS) are compatible with the new version — check the System Requirements Guide in Commvault documentation.

Data Integrity Best Practices

DDB Verification Jobs If using Commvault software deduplication (DDB), enable and schedule DDB Verification Jobs. These verify the deduplication database is intact and backup data is recoverable. Without verification, silent corruption can go undetected. Enable DDB Verify
Data Verification Schedule If using storage-level deduplication (e.g., HP StoreOnce — Commvault cannot use software dedup with it), enable a Data Verification schedule in Commvault to regularly verify backup data integrity. Alternative to DDB
Media Agent in Same Region as Storage If backing up to public cloud, deploy Media Agent in the same cloud region as your storage account/bucket. This prevents egress charges (data transfer fees for moving data out of a cloud region). Ask Commvault for recommendations on optimal Media Agent placement. Cost Saving
Regular OS/Driver Patching Keep CommServe OS, Media Agent OS, HBA drivers, and network drivers current. Outdated drivers on Media Agents (physical servers) can cause backup failures or data corruption that are difficult to diagnose. Regular Patching
Periodic Restore Testing Schedule periodic restore tests (quarterly at minimum) to verify backups are actually restorable. The scenario described above — backups completing but restores failing — would have been caught immediately if the customer had tested restores periodically. Critical Practice
🎓 Interview/Architect Question

An interviewer may ask: "How do you ensure backup data integrity in your environment?" The expected answer covers: DDB verification jobs, data verification schedules, periodic restore testing, maintaining current MR versions, and monitoring backup job completion with proactive alerts. Simply saying "backups complete every day" is not sufficient — the example above shows that completion does not guarantee restorability.

Interview & Exam — Scenario-Based Q&A
Real-world questions on IntelliSnap, Azure SQL PaaS backup, troubleshooting, and best practices
What is the difference between a streaming backup and an IntelliSnap backup? When would you recommend IntelliSnap to a customer?
A streaming backup reads data directly from the live production volume over the network and writes it entirely to the backup library. A 100 TB database takes hours to stream. During the backup window, the production server experiences I/O load and must remain accessible. An IntelliSnap backup creates an application-consistent snapshot on the storage array in seconds or minutes — size-independent. The actual data stays on the storage array (only ~15 KB metadata goes to the library as the snap copy). A backup copy job then reads from the snapshot (not the live volume), so production is unaffected during data transfer. I would recommend IntelliSnap when: (1) the customer has a large dataset where streaming backup windows are too long, (2) the storage array is SAN-based or cloud-based and supports snapshots, (3) zero production I/O impact is required during backup, or (4) fast recovery from recent events is needed (restore directly from array snapshot in minutes).
After enabling IntelliSnap, you see two jobs in the Job Controller every time a backup runs. What are they and what does each one do?
The two jobs are: (1) Snap Backup Job (e.g., "VM Admin Snap Backup") — this is the job you initiated. It quiesces the application, sends an API request to the storage array to create a snapshot, then writes only the snapshot metadata (~15 KB) to the Snap Copy in the library. (2) Backup Copy Job — automatically spawned by the snap backup job. It mounts the snapshot on the media agent and copies the full VM/application data to the Primary Copy (cloud library). This is a traditional streaming backup, but reading from the snapshot instead of the live volume. Both jobs run automatically when IntelliSnap is enabled and a Snap Copy exists in the storage policy. The snap copy job is always fast; the backup copy job takes longer because it transfers the full data to the library.
You set the snap copy retention to "Spool Copy." The backup copy job completes, but the snapshot is still visible in the Azure Portal. Why, and how do you remove it?
The snapshot is still visible because Commvault's Data Aging job has not run yet. Spool Copy retention means: delete the snapshot as soon as the backup copy completes. However, the actual deletion happens when the data aging job runs — which only executes automatically once every 24 hours. Even though the snap copy is logically expired (backup copy done), it is not physically deleted from Azure until data aging runs. To immediately remove it: in CommCell Console → right-click the CommServe → Data Aging → Run Now. After the manual data aging run, Commvault sends a delete API call to Azure, the snapshot disappears from the Azure Portal, and the Primary Snap copy shows no jobs in List Snapshots. The Primary Copy backup copy job remains unaffected and is still restorable.
You assigned the Azure built-in "Backup Contributor" role to the access node's managed identity. The hypervisor was configured in Commvault, but when adding the VM Group content, the VM list shows "No items found." What is the issue and how do you resolve it?
The Backup Contributor built-in Azure role does not include the permissions needed for Commvault to list and browse virtual machines in the subscription. The role may cover some backup operations but lacks VM discovery permissions. Resolution: switch to a Commvault Custom Role. In Commvault documentation → "Roles and Permissions Required for Protecting Azure Resources with Commvault" → find the JSON template for Azure VM backup. In Azure Portal: Subscriptions → IAM → Add → Create Custom Role → paste the Commvault JSON (with your Subscription ID replacing the placeholder, without curly braces). Create the role, then assign it to the access node VM's managed identity at the subscription scope. After this, the VM Group content browse will show all VMs in the subscription.
What is the difference between IaaS SQL Server backup and Azure SQL (PaaS) backup in Commvault? Why is an access node required for PaaS backup?
With IaaS SQL (SQL Server on a VM), you have full VM access — you can install the Commvault SQL agent directly on the VM, connect to the SQL instance locally, and back up via agent-based backup or agentless VM backup. With Azure SQL PaaS, the underlying VM is completely managed by Azure — you cannot log into it, cannot install any software on it. You only interact with the SQL database endpoint. Therefore, Commvault cannot install an agent on the host. Instead, you deploy a separate access node (a VM you create and control) with the MSSQL agent installed. The access node connects to the Azure SQL Database over port 1433 using SQL credentials and the managed identity, triggers a database export/copy, and transfers the data to the backup library. The access node acts as a proxy — it has agent capabilities and connectivity to both the Azure SQL service and the backup library.
An Azure SQL backup job fails with "database is not currently available." You verify the database is online in the Azure Portal. What is the likely cause and fix?
Despite the database being online in Azure Portal, this error typically indicates that the ODBC driver is not installed on the Windows access node. Commvault's SQL agent uses ODBC to connect to Azure SQL Database. Without the Microsoft ODBC Driver for SQL Server on the access node, the agent cannot establish a connection even though the database is reachable at the network level (port 1433 connectivity may be fine). The fix is to install two components on the Windows access node: (1) Microsoft Visual C++ Redistributable (required dependency), then (2) Microsoft ODBC Driver for SQL Server (the version compatible with your SQL Server edition). Note that Commvault's documentation may not clearly list ODBC as a prerequisite — this is a known gap. After installing, resume the failed job (or resubmit a new one). The backup should proceed past the connection phase and export the database successfully.
A customer's Azure SQL backup fails with "Pole operation status return failed" (also seen as "Poll operation status return failed"). The database is online and ODBC is installed. What is the cause?
This error occurs when the Azure SQL Database is on the free tier. Commvault's backup process internally creates a copy/export of the database (similar to a BACPAC export). The Azure SQL free tier explicitly restricts database copy operations to prevent abuse of the free service. This limitation is not a Commvault bug — it is an Azure platform restriction. The fix: in Azure Portal → SQL Database → Compute + Storage → disable the "Apply free database offer" checkbox → switch to paid compute (basic tier, approximately $6/month). After disabling the free tier, kill the failed job and resubmit a new full backup. The export operation will now complete successfully.
A customer reports that their Commvault backup jobs have been showing "Completed" for 6 months, but when they tried to restore a VM last week, it booted to a Blue Screen of Death every time. What happened and what should they have done to prevent this?
This is a known real-world scenario caused by an unpatched Commvault bug in an outdated Maintenance Release (MR). The customer was on a supported platform version (e.g., 11.32) but had not applied MR updates for an extended period. A bug existed in that MR version where VM restore completes successfully in Commvault but the restored VM is non-bootable. The bug was fixed in a later MR. Because they never applied the MR fix, all backups taken during that period are unrestorable from those jobs. Prevention: (1) Always keep Maintenance Releases current — apply new MRs as soon as they are released and tested. MRs are hotfixes that specifically address bugs like this. (2) Perform periodic restore tests — quarterly test restores of representative VMs would have immediately caught this issue. (3) Monitor Commvault's known issue list and release notes for the MR versions you are running.
