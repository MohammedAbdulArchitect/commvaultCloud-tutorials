Section 01

Logical Architecture

BackupSets, Subclients, Plans, and how they relate to physical components

The physical architecture of Commvault defines the three infrastructure tiers ‚Äî CommServe, Media Agent, and Client. The logical architecture sits on top of this and defines what data gets protected, how it is organised, and when it runs. Understanding this layer is fundamental to configuring and troubleshooting any backup job.

The Logical Hierarchy

üì¶

Backup Set

A logical collection of all data protected by a given agent on a client. For a Windows File System agent, the default Backup Set automatically covers all drives (C:, D:, etc.) unless you narrow it down.

üóÇÔ∏è

Subclient

Defines a specific subset of data within a Backup Set. For example, protecting only the D: drive on a Windows server where critical application data lives ‚Äî while excluding the C: OS drive.

üìã

Plan

Governs a subclient's protection by defining the schedule (how often), storage target (where), and retention (how long). Plans are configured in Command Center.

What a Plan Defines

  • Schedule (RPO) ‚Äî How frequently backups run: Full on Sundays, Incremental Monday‚ÄìSaturday at 9 PM, for example.
  • Storage target ‚Äî Which disk library, cloud bucket, or tape pool the data is written to.
  • Retention ‚Äî How many days backup data is kept before aging out. Example: 7 days, 30 days, 1 year.
  • Storage Policy equivalent ‚Äî Plans in Command Center map to Storage Policy + Schedule Policy in the Java Console.
üìò Plan vs. Storage Policy

Plans are configured in the Command Center (web UI). Storage Policies and Schedule Policies are the equivalent objects configured in the Java Console. When you create a Plan in Command Center, Commvault automatically creates a corresponding Storage Policy and Schedule Policy visible in the Java Console.

Section 02

Storage Types & Disk Libraries

Configuring disk storage with and without deduplication

Before any backup job can run, at least one storage destination must be configured. Commvault supports three primary storage types, each suited to different environments and compliance requirements.

üñ¥

Disk Storage

The most common type. A local or network-attached drive on the Media Agent. Fast and easy to configure. Supports deduplication.

‚òÅÔ∏è

Cloud Storage

Amazon S3, Azure Blob, or GCP Storage. Elastic capacity, pay-per-use. Ideal for offsite retention and long-term archival.

üìº

Tape Storage

Preferred in air-gapped, dark-site, government, and highly regulated environments. Supports compression only ‚Äî no deduplication at the Commvault layer.

Disk Storage ‚Äî Dedicated vs. Shared

‚úÖ Dedicated (Local)

  • Disk is directly connected to a single Media Agent
  • Not shared with any other MA ‚Äî exclusive use
  • Lowest latency, simplest configuration
  • Select Local as the type in Command Center

üîó Shared (Network)

  • Disk is shared across multiple Media Agents
  • Uses NAS or SAN connectivity
  • Requires credentials for the network share path
  • Select Network as the type; provide UNC path + credentials

Understanding Deduplication

Deduplication is one of the most impactful features for reducing backup storage consumption. It operates at the block level ‚Äî identifying and eliminating redundant data chunks before writing to storage.

üí° Deduplication Example

You have two identical 1 GB files on a client with different names. Without dedup: both files are written to storage ‚Üí 2 GB consumed. With dedup: the second file's blocks are recognised as duplicates; only pointer references are written ‚Üí ~1 GB consumed. Savings ‚âà 50% in this case. Real-world savings typically range from 50‚Äì90%.

Deduplication Database (DDB)

Deduplication requires a dedicated DDB partition ‚Äî a separate drive on the Media Agent used exclusively for the deduplication index database. The DDB must never share the same drive as backup data.

  • Separate drive required ‚Äî The DDB partition must reside on a dedicated drive (e.g. F: drive) separate from backup data (E: drive).
  • Multiple DDB partitions ‚Äî Large environments can span the DDB across multiple partitions for performance.
  • DDB = Deduplication Database ‚Äî Short-form used in Commvault documentation and job logs.
  • Folder must be empty ‚Äî The folder selected as a mount path for backup data must be empty before associating it to a library.
‚ö†Ô∏è When NOT to Use Commvault Deduplication

For certain storage types, Commvault recommends disabling its built-in deduplication:

  • HP Catalyst / StoreOnce ‚Äî The storage appliance performs deduplication natively; double-dedup reduces efficiency. Commvault recommends configuring the plan without deduplication in this case.
  • Tape libraries ‚Äî Tape does not support block-level deduplication. Only compression applies. Never configure DDB for tape-only storage pools.

Section 03

Creating Plans & Subclients

Defining schedules, retention, storage, and subclient content

A Plan in Command Center is the central configuration object that ties together when to backup, where to write it, and how long to keep it. One plan can be applied to many subclients across different clients and agents.

Lab: Creating a Server Backup Plan

  1. Navigate to Manage ‚Üí Plans ‚Üí Add Plan ‚Üí Server Backup Plan
  2. Name the plan clearly

    Use a descriptive name tied to its purpose. Example: WinFS-7day-NonDedup or LinuxFS-7day-Dedup.

  3. Set backup frequency (RPO)

    Define the Full backup schedule (e.g. Sunday) and Incremental schedule (e.g. Monday‚ÄìSaturday 9 PM). Exclude Sunday from the incremental schedule to avoid overlap with the Full run.

  4. Assign storage

    Select the disk library created in the previous step. Dedup or non-dedup storage can be assigned to any plan ‚Äî there is no OS-specific restriction.

  5. Set retention

    Define how many days backups are retained on storage. Example: 7 days. Retention interacts with the Cycles setting (covered in the Console Comparison section).

  6. Ignore snapshot options

    The Snapshot / IntelliSnap options only apply if your storage supports snapshot APIs and an IntelliSnap license is active. Leave disabled in standard deployments.

üìò Naming Convention Best Practice

Plan names carry no technical restriction ‚Äî they are identifiers only. Use a consistent naming pattern that makes the plan's purpose immediately clear: [AgentType]-[Retention]-[StorageType]. Example: WinFS-30day-Dedup. This removes ambiguity when supporting multiple environments.

Creating a Subclient & Assigning a Plan

  1. Navigate to Protect ‚Üí File Servers ‚Üí [Target Client]
  2. View the default subclient

    Every agent creates a Default Subclient automatically. It protects all data under the agent's scope unless overridden. You can use it directly or create additional subclients for specific paths.

  3. Add a new subclient (optional)

    Click Add Subclient. Provide a name (e.g. D-Drive-AppData), assign a plan, and define content.

  4. Define content

    Use the Browse option to navigate the client's directory structure. Select specific folders or files to protect. For Linux clients, navigate Unix paths (e.g. /usr/confidential.txt).

  5. Save the subclient

    The subclient is now ready. It will run automatically per the plan's schedule, or can be triggered manually (ad-hoc / interactive backup).

üí° Default vs. Custom Subclient

The Default Subclient protects everything the agent discovers ‚Äî all drives on Windows, or all file systems on Linux. A custom subclient lets you narrow scope to specific paths, which is the common production pattern. Customer teams will tell you exactly which paths require protection.

üìò Index Cache

The Index Cache resides on the Media Agent and stores catalogue metadata about all backed-up files ‚Äî paths, versions, timestamps. This is separate from the CommServe database. It is used for browse and restore operations. Always ensure sufficient disk space on the MA for index cache growth.

Running an Ad-Hoc Backup (Windows)

  1. Go to Protect ‚Üí File Servers ‚Üí [Windows Client] ‚Üí Subclients
  2. Select the subclient and click Backup
  3. Choose backup type ‚Äî Select Full for the first run. Subsequent runs can use Incremental.
  4. Uncheck email notification and click Submit
  5. Monitor via Jobs ‚Äî Navigate to the Jobs section. Click the Job ID to view progress, throughput, and object counts.
üí° Compression vs. Deduplication

Even on a non-dedup storage library, Commvault automatically applies compression. You may see significant savings (e.g. 84%) on the job details screen even without dedup enabled. Compression = CPU-based size reduction. Dedup = block-level elimination of duplicate data. Both can be active simultaneously on dedup-enabled libraries.

Performing an In-Place Restore (Windows)

  1. Navigate to the Windows client in Protect ‚Üí File Servers
  2. Click Restore ‚Äî The latest recovery point (backup time) is shown
  3. Browse the backup ‚Äî Navigate the file tree to select files/folders to restore
  4. Select In-Place restore ‚Äî Destination is the same as the source path
  5. Tick "Unconditionally Overwrite" ‚Äî Required if the file already exists at the destination
  6. Submit and monitor the restore job to completion

Linux File System Backup & Restore

The process is identical to Windows, with two key differences: the content paths follow Unix directory conventions, and elevation to root is required for certain paths.

üéì Interview Tip ‚Äî Backup Process Flow

A common interview question: "Explain the backup process flow for a file system agent." Key points to cover: (1) CommServe initiates the job; (2) the client iDataAgent gathers source data; (3) data is transferred directly to the Media Agent (CommServe never handles data); (4) the MA deduplicates, compresses, encrypts and writes to the library; (5) CommServe updates the metadata database and index cache. Always emphasise that CommServe never touches backup data.

Section 05

Backup Types Explained

Full, Incremental, Differential, Synthetic Full, Log Backups, and the Cycle concept

Full

Backs up all selected data regardless of previous backup state. Largest backup size. Provides the most self-contained recovery point. Typically scheduled weekly (e.g. Sundays).

Incremental

Captures only data that has changed since the last backup (full or incremental). Smallest backup size. Fastest to run. Requires the last full + all incrementals for a complete restore. Typically scheduled daily.

Differential

Captures all data that has changed since the last Full backup. Larger than incremental but only requires the last full + the latest differential for restore. Grows in size each day until the next full runs.

Synthetic Full

Creates a new full backup by consolidating existing full + incrementals on storage ‚Äî without reading from the source client. Reduces backup window impact on the production server. Commvault auto-schedules this as a safety net (every 7 days) if no traditional full has run.

Log Backup

Backs up database transaction logs (Oracle archive logs, SQL Server transaction logs) for point-in-time recovery. Applicable to database agents only ‚Äî not used for file system agents. Supports granular recovery between full/incremental schedules.

Incremental vs. Differential ‚Äî At a Glance

üéì Key Interview Distinction

Incremental: "What changed since the last backup?" (last backup could be a full or another incremental)
Differential: "What changed since the last Full?" (always references the most recent full only)

A differential on Wednesday after a Sunday full will contain all changes from Monday + Tuesday + Wednesday ‚Äî it grows larger each day. An incremental on Wednesday contains only Wednesday's changes.

Understanding Retention Cycles

Commvault retention is governed by two criteria ‚Äî both must be satisfied before a backup job ages out:

  • Retention Days ‚Äî The number of days the backup must be kept (e.g. 7 days)
  • Cycles ‚Äî The number of full-to-full cycles that must complete before old jobs are deleted
üìò What is a Cycle?

A cycle is the period from one Full backup to the next Full backup, including all incrementals in between. With a 7-day retention and 1 cycle configured: even if 7 days have passed, Commvault will not delete the first Full until the second Full (and its following incrementals) have completed. This guarantees that at least one complete recovery cycle is always available on storage.

Note: The Cycles setting is visible only in the Java Console, not in Command Center. Command Center focuses on days-based retention for simplicity.

Data Aging

A system-level job called Data Aging runs daily (by default at 12 PM) and performs the following actions:

  • Scans all backup jobs for jobs that have exceeded retention criteria (days + cycles)
  • Marks expired jobs for deletion from the CommServe database
  • Frees storage space on disk libraries by removing aged backup data
  • This job is auto-created at installation ‚Äî you do not need to configure it manually
üí° Viewing Jobs by Type (Java Console)

From the Java Console top-level: Right-click ‚Üí View ‚Üí Job History to see all backup jobs. Select Data Recovery Operation to see restore jobs. Select Admin Job History ‚Üí Data Aging to verify data aging runs. Always verify that data aging jobs complete successfully on a daily basis.

CommServe Disaster Recovery Backup

Protecting the CommServe database ‚Äî the most critical backup in any Commvault environment

The CommServe database is the single source of truth for your entire backup environment ‚Äî it stores all metadata, job history, client configurations, and policies. If this database is lost without a DR backup, the entire CommCell environment must be rebuilt from scratch. Ensuring CommServe DR backups complete successfully every day is a top priority for every backup administrator.

CommServe DR Backup ‚Äî Process Flow

Phase 1
SQL dump generated ‚Äî DMP files created for all CommServe databases
‚Üí
Phase 2
DMP files cached to metadata staging location (local path on CommServe)
‚Üí
Phase 3
Cache copied to defined metadata export path (e.g. E:\CommserverDR)
‚Üí
Phase 4
(Optional) Copy uploaded to Metallic Cloud (Commvault Cloud)
‚Üí
Phase 5
Final copy written to CommServe DR Storage Policy on disk library

Key Components

  • SQL Server Engine ‚Äî Installed automatically with CommServe. Hosts the CommServe database. Check via Control Panel ‚Üí Programs and Features.
  • CommServe DB (DMP file) ‚Äî Primary database dump. Located at C:\Program Files\Commvault\ContentStore\CommserveDR\. The file named commserve_<hostname>.dmp is the most critical.
  • Supporting databases ‚Äî AppStudio, CacheDB, HistoryDB, WorkflowEngine ‚Äî smaller databases that are also included in the DR dump.
  • Metadata export path ‚Äî A secondary local or network path (e.g. E: drive) where DR dumps are exported. Note this path carefully ‚Äî it is your first recovery option if the C: drive fails.
  • CommServeDR Storage Policy ‚Äî Auto-created when you add the first library. Visible in Java Console ‚Üí Storage Policies. Not visible in Command Center.
  • Metallic Cloud ‚Äî Commvault's own cloud offering. An optional third copy destination for CommServe DR backups.
CommServe DR ‚Äî Default Paths
# Primary DR staging / cache location
C:\Program Files\Commvault\ContentStore\CommserveDR\

# Example metadata export path (user-defined, configure in DR settings)
E:\CommserverDRBackup\

# Verify SQL Server installation in programs list
# Control Panel ‚Üí Programs and Features ‚Üí search "SQL"
# Expected: SQL Server Engine, SQL Management Studio, SQL Native Client
üö® Critical: Monitor DR Backups Daily

CommServe DR backup failure is one of the most serious situations in backup administration. If the CommServe crashes and there is no current DR backup, the entire environment metadata is lost. Always verify that the DR job completed successfully in the previous 24 hours. Alert your manager immediately if DR backups are failing.

üí° Adding a Tape Copy to CommServe DR Policy

For additional protection, add a secondary tape copy to the CommServeDR Storage Policy: In Java Console ‚Üí Storage Policies ‚Üí CommServeDR Policy ‚Üí Right-click ‚Üí All Tasks ‚Üí Create New Copy ‚Üí assign a tape pool. Then run an Aux Copy to replicate DR data to tape. This ensures a copy exists even if both the disk library and CommServe itself are lost.

üéì Interview: CommServe DR Process Flow

"Explain the CommServe DR backup process."

Answer: "The CommServe DR backup runs automatically as a scheduled job. It first generates a SQL dump of all CommServe databases ‚Äî the primary commserve.dmp and supporting databases. These dumps are staged to a local cache path, then exported to a user-defined metadata location. Optionally, a copy is uploaded to Commvault Metallic Cloud. Finally, the DR data is written to the CommServeDR Storage Policy on the configured disk library. This provides three recovery options: local metadata path, cloud copy, and disk library copy."

Cloud Library (AWS S3)

Configuring object storage as a Commvault backup target

Cloud storage integrates seamlessly with Commvault as a backup target. The most common configuration uses Amazon S3 (Simple Storage Service) as an object store. Before configuring a cloud library in Commvault, the cloud storage team must provision the required resources and supply the credentials.

Prerequisites ‚Äî Information Required from Cloud Team

  • Access Key ‚Äî IAM user access key for authenticating to AWS
  • Secret Access Key ‚Äî Paired secret credential (generated once ‚Äî save immediately)
  • Service Host ‚Äî The AWS region endpoint URL, e.g. s3.us-east-1.amazonaws.com
  • S3 Bucket Name ‚Äî The container within S3 where backup objects will be stored
üìò Files vs. Objects

In cloud storage terminology, data stored inside an S3 bucket is called an object ‚Äî not a file. When communicating with cloud support engineers, always use the term objects. A bucket is the container; objects are the items inside it. Commvault writes backup data as objects inside the configured bucket.

Lab: Creating an AWS IAM User & Access Keys

  1. Log in to AWS Console ‚Äî Navigate to aws.amazon.com/console
  2. Search for IAM ‚Äî Identity and Access Management ‚Üí Users ‚Üí Create User
  3. Username ‚Äî Use a descriptive service account name, e.g. commvault-svc-account
  4. Attach permissions ‚Äî Search for and attach the AmazonS3FullAccess managed policy
  5. Create Access Key ‚Äî Under Security Credentials ‚Üí Create Access Key ‚Üí select Third-party service as the use case
  6. Save credentials immediately ‚Äî Copy both the Access Key and Secret Access Key. The Secret Key is shown only once and cannot be retrieved later.
‚ö†Ô∏è Secret Access Key ‚Äî One-Time Display

The Secret Access Key is only displayed at the time of creation. If you close the dialog without saving it, you must generate a new key pair. Always store credentials securely in a password manager or encrypted vault immediately upon creation.

Lab: Creating an S3 Bucket

  1. Search for S3 in the AWS Console
  2. Create Bucket ‚Äî Choose General Purpose. Provide a globally unique bucket name.
  3. Leave access settings default ‚Äî Block all public access should remain enabled
  4. Click Create Bucket ‚Äî The bucket is now ready to receive backup objects

Lab: Configuring Cloud Library in Command Center

  1. Navigate to Storage ‚Üí Cloud ‚Üí Add Cloud Storage
  2. Select vendor type ‚Äî Commvault supports AWS, Azure Blob, GCP, EMC, Oracle Cloud, and others. Select Amazon S3.
  3. Provide name, credentials, service host, and bucket ‚Äî Enter all details supplied by the cloud team
  4. Select Media Agent ‚Äî The MA acts as the gateway for data transfer to the cloud
  5. Save ‚Äî The cloud library is now available for use in plans
üñºÔ∏è

Cloud library configuration ‚Äî Command Center showing Amazon S3 credential entry, service host, and bucket selection

Figure: Cloud storage setup in Command Center ‚Äî the Media Agent acts as the gateway for all data written to the S3 bucket

Cloud Storage Backup Data Structure

When Commvault writes backup data to cloud storage, it creates a structured hierarchy within the bucket:

  • Container-level folders ‚Äî Organised by CommCell ID, storage policy, and copy number
  • Chunk files ‚Äî Backup data is written in fixed-size chunks (typically 64 MB). Each chunk contains compressed, optionally encrypted, and optionally deduplicated data blocks.
  • SFILE containers ‚Äî Commvault's proprietary packaging format for cloud objects. Visible inside the S3 bucket as SFILE named objects.
  • Metadata files ‚Äî Separate objects containing the catalogue and index information for each chunk
üéì Interview: Cloud Library Configuration

"What information do you need from the cloud team to configure a cloud library in Commvault?"

Answer: "Four pieces of information are required: the Access Key and Secret Access Key for authentication; the Service Host (region endpoint URL such as s3.us-east-1.amazonaws.com); and the S3 Bucket Name that has been provisioned for backup storage. In production, the backup admin typically does not create these ‚Äî they are provided by the cloud or infrastructure team."

Tape Library Configuration

Physical and virtual tape library concepts, components, and lab setup

Tape libraries remain relevant in specific enterprise contexts ‚Äî government agencies, financial institutions with air-gap requirements, and dark-site environments where no external network connectivity is permitted. Understanding tape terminology and configuration is an important skill for L2 backup administrators.

Tape Library Components

ü§ñ

Tape Library (Robotic Arm)

The physical enclosure containing multiple tapes and drives. A robotic arm moves tape media between storage slots and drives automatically. Acts as the controller for all media movement operations.

üíø

Tape Drive

The mechanism that reads from and writes to tape media. When a backup job runs, the robotic arm mounts a tape media cartridge into an available drive. A library can have multiple drives for parallel operations.

üè∑Ô∏è

Tape Media (Cartridge)

The physical storage medium ‚Äî a cartridge containing a magnetic tape reel. Each cartridge has a barcode label used by the library management software to identify and track individual tapes.

Additional Tape Terminology

Barcode
Unique identifier label on each tape media cartridge. Scanned by the library to track location and status.
IE Slot (I/E Port)
Import/Export port ‚Äî the entry point for inserting new tapes into or removing tapes from the library without interrupting operations.
Shelf Slot
Storage slot inside the library where tape cartridges reside when not mounted in a drive.
Drive Address
The SCSI or Fibre Channel address identifying each drive in the library to the host operating system.
üí° Tape vs. Disk: Deduplication & Compression

Tape storage does not support Commvault deduplication at the product layer. Only compression applies for tape-based backup data. Never configure a DDB partition for a storage pool that exclusively uses tape. Configure tape pools without the deduplication option enabled.

The Commvault Virtual Tape Library (VTL) emulator is intended for training and lab environments only. Configuration of production tape libraries requires the library to be connected, drivers installed on the MA, and the library discovered by Commvault during the tape configuration wizard.

Hyper-V Agentless Backup & Restore

Architecture, lab setup, VSA agent configuration, and VM backup/restore workflows

Hyper-V is Microsoft's hardware virtualisation platform. Most enterprise environments use Hyper-V or VMware vSphere (together accounting for ~90% of virtualisation deployments). Backup administrators must understand how to protect virtual machines running inside Hyper-V ‚Äî using Commvault's agentless backup method via the Virtual Server Agent (VSA).

What is Agentless Backup?

In agentless VM backup, no Commvault agent is installed inside the guest virtual machine being protected. Instead:

  • The Virtual Server Agent (VSA) is installed on the Hyper-V host server (not inside the VMs)
  • Commvault uses the Hyper-V VSS (Volume Shadow Copy Service) API to take application-consistent snapshots
  • The VSA coordinates with the hypervisor to capture the entire VM ‚Äî including disk, memory state, and application data
  • We call this agentless because the protected workload (the guest VM) has no agent installed inside it
üìò Agentless ‚â† No Agent at All

Despite the term, agentless backup still requires an agent ‚Äî the Virtual Server Agent (VSA) ‚Äî installed on the Hyper-V host. It is agentless from the perspective of the guest VMs being protected. This is an important distinction in interviews and client discussions.

Hyper-V Architecture Overview

üñ•Ô∏è Standalone Hyper-V Node

A single Windows Server with the Hyper-V role enabled. Used in smaller environments. No automatic VM failover capability. Configure Commvault with the server's hostname or IP + local admin credentials.

üîó Hyper-V Failover Cluster

Multiple Hyper-V nodes grouped in a Windows Failover Cluster. Provides automatic VM failover when a node fails. Configure Commvault with the cluster IP + cluster admin credentials ‚Äî not individual node IPs.

Section 11

Interview Q&A ‚Äî Scenario-Based

Real-world questions covering all topics in Part 3

What is a Subclient and how does it differ from a Backup Set?
A Backup Set is the logical collection of all data protected by an agent on a client machine. For a Windows File System agent, the default Backup Set includes all drives. A Subclient defines a specific subset of data within a Backup Set ‚Äî for example, only the D: drive or a specific application folder. Subclients are where you define content, assign a plan, and run backup jobs. In production, custom subclients are configured to protect exactly the data the application or customer team specifies.
What is the difference between Incremental and Differential backups?
Incremental backup captures all data changed since the last backup ‚Äî whether that last backup was a full or another incremental. It produces the smallest backup size and fastest run time, but requires the last full plus all incrementals in sequence for a full restore. Differential backup captures all data changed since the last Full backup. It is larger than an incremental (and grows each day), but simplifies restore ‚Äî you only need the last full plus the single latest differential. Choose incremental for minimal storage consumption; choose differential for faster restore times in RPO-critical environments.
What is a Synthetic Full backup and why would you use it?
A Synthetic Full creates a new full backup by consolidating the most recent full backup and all subsequent incrementals ‚Äî reading from storage rather than from the source client. This means no I/O load on the production server during the operation. Commvault automatically schedules a Synthetic Full every 7 days as a safety net via the schedule policy. It is visible in the Java Console schedule policy but not in Command Center for Commvault v11.32 and earlier. The definition: "A backup that consolidates previous incrementals and the last full to create a new synthetic full, without reading from the source."
Explain retention days and cycles in Commvault.
Retention has two components that both must be satisfied before backup data is aged out. Retention Days is the minimum number of days the data must be kept. Cycles defines how many full-to-full cycles (one full + its incrementals + the next full = one cycle) must complete. Even if 7 retention days have passed, Commvault will not delete backup jobs until the next full backup cycle has completed. This prevents the scenario where your only full backup expires before a new one is written. The Cycles setting is visible in the Java Console Storage Policy properties ‚Äî it is not exposed in Command Center.
Why is the CommServe DR backup critical, and what is its process flow?
The CommServe database contains all metadata for the entire backup environment ‚Äî job history, client configurations, storage policies, schedules, and the backup catalogue. If the CommServe fails without a current DR backup, the environment cannot be recovered. The DR backup process: (1) generates a SQL dump of the CommServe database; (2) caches DMP files to a local staging path inside ContentStore\CommserveDR; (3) exports the dumps to a user-defined metadata path (e.g. E:\CommserverDRBackup); (4) optionally uploads a copy to Metallic Cloud; (5) writes the final copy to the CommServeDR Storage Policy on disk. Backup administrators must monitor DR job completion daily.
What is the difference between Command Center and the Java Console?
The Command Center is the modern web-based interface introduced in Commvault v11.20, mandatory for new deployments from v11.34. It simplifies operations by combining Storage Policy and Schedule Policy into a single Plan object and uses user-friendly terminology (Servers, Plans, Disk Storage). The Java Console (CommCell Console) is the legacy Java thick-client that exposes deeper configuration ‚Äî including Cycles-based retention, CommServeDR Policy management, Aux Copy, and Storage Pools. Both consoles manage the same CommCell; changes in one are reflected in the other. In environments where both are present, follow whichever the organisation standardises on ‚Äî mixing consoles in production is not recommended.
What information do you need to configure a cloud library in Commvault?
Four pieces of information are required from the cloud team: (1) Access Key ‚Äî the IAM user access key for AWS; (2) Secret Access Key ‚Äî paired credential, shown only once upon creation; (3) Service Host ‚Äî the regional S3 endpoint, e.g. s3.us-east-1.amazonaws.com; (4) Bucket Name ‚Äî the S3 container provisioned for backup storage. In Command Center: Storage ‚Üí Cloud ‚Üí Add Cloud Storage. Select the vendor (AWS, Azure, GCP, etc.), enter the credentials, and assign a Media Agent as the gateway. Backup administrators typically do not create cloud resources ‚Äî they configure using details provided by the cloud operations team.
Explain Hyper-V agentless backup. What agent is actually required?
Hyper-V agentless backup means no Commvault agent is installed inside the guest virtual machines being protected. The Virtual Server Agent (VSA) is installed on the Hyper-V host server. It uses Microsoft's VSS API to create application-consistent snapshots of running VMs. For a Failover Cluster environment, Commvault is configured with the cluster IP and cluster credentials ‚Äî this ensures VMs are always discovered and protected regardless of which node they are running on. For standalone Hyper-V Manager setups, the individual node IP and local admin credentials are used. The term "agentless" specifically refers to the absence of an agent inside the protected guest VM.
When would you NOT use Commvault deduplication, and why?
Commvault deduplication should not be configured in two scenarios: (1) HP Catalyst / StoreOnce appliances ‚Äî These storage platforms perform native deduplication inside the appliance itself. Enabling Commvault dedup on top of this creates "double dedup," which reduces efficiency and can cause performance issues. Commvault formally recommends disabling its dedup for Catalyst targets. (2) Tape libraries ‚Äî Tape storage does not support block-level deduplication at the Commvault layer. Only compression applies to tape-based data. Never configure a DDB partition for a tape-only storage pool.
