VMware Application-Aware Backup — What & Why
Application-consistent agentless backups for VMs running databases and critical applications

The Problem: Agentless VM Backup ≠ Application Consistency

When you take a standard agentless (snapshot-level) backup of a VMware VM that is running a database like SQL Server, the backup captures the VM's disk state at a point in time. However, a database engine keeps data in memory and has pending transactions that may not have been flushed to disk at the exact moment of the snapshot. When you restore this VM and try to start the database:

  • The database may fail to start, or start with errors
  • Transaction logs may be inconsistent
  • Data written to memory but not yet committed to disk is lost
  • The database may require recovery procedures before it becomes usable

This is called a crash-consistent backup — the VM is captured as if the power was suddenly cut. It works for the OS but is unreliable for running applications.

📘 Application-Aware Backup — The Solution

Application-Aware Backup captures the VM AND the current state of running applications at the same instant — including data in memory and pending transactions. When restored, the application starts cleanly in a consistent, known-good state. This is called an application-consistent backup. It does not require installing a dedicated database agent inside the VM.

Agent-Based vs. Application-Aware — Critical Comparison

🔌 Agent-Based Backup (e.g., SQL Agent)
  • Install SQL Server agent directly inside each VM
  • Schedule separate SQL backup jobs per VM
  • Full application integration — table-level granular restore
  • Transaction log backups supported (RPO of minutes)
  • Consumes an agent license for every VM
  • High management overhead for 100s of VMs
  • Use when: critical DBs needing log backups or granular restore
🛡 Application-Aware Backup (Agentless)
  • No agent installed inside the VM — enable one option in VM group
  • One schedule covers both VM and application backup
  • Application-consistent — DB starts clean after restore
  • No transaction log backups (only full/incremental VM backups)
  • No additional agent license consumed — only VM license
  • Scales easily to hundreds of VMs
  • Use when: scalable environments, ease of management is priority
⭐ Licensing Advantage

With Application-Aware Backup, Commvault does not consume an agent-based license for the SQL database — only the VM-level license applies. In large environments with hundreds of VMs running SQL Server, this can represent a significant licensing cost saving compared to deploying individual SQL agents in every VM.

Supported Hypervisors for Application-Aware Backup

Application-Aware Backup is not limited to VMware. Commvault supports this feature across multiple virtualization platforms. The underlying procedure is essentially the same — only the hypervisor-specific details differ.

✓ VMware vSphere ✓ Amazon EC2 (AWS) ✓ Microsoft Azure ✓ Microsoft Hyper-V ✓ Nutanix AHV ✓ OpenStack ✓ Oracle VM ✓ Red Hat Virtualization ✓ VMware Cloud
📘 Steps Remain the Same Across Hypervisors

Whether you are configuring Application-Aware Backup on VMware, AWS EC2, or Azure, the Commvault configuration steps are essentially the same. What differs is how each hypervisor platform is connected as a Commvault hypervisor. Once connected, the Application-Aware option is enabled in the VM Group properties — the rest is identical.

VSA proxy requirements, guest VM requirements, and first-run automatic installations

VSA Proxy Requirements

Application-Aware Backup requires a VSA (Virtual Server Agent) proxy, just like standard agentless VM backup. The proxy handles the backup operations and is separate from the VMs being protected.

Guest VM Requirements (VMware)

The virtual machine being protected must meet the following requirements. These are configured on the guest OS side — not by the backup admin directly, but in coordination with the Windows/Linux teams.

💡 First-Time Backup — Automatic Package Push

You do NOT manually install packages inside the guest VM for Windows. When you run the Application-Aware backup for the first time, Commvault automatically pushes the required packages into the guest VM: File System Core, Media Agent binaries, and application plugins (e.g., MSSQL plugin for SQL Server). This automatic installation only happens once — subsequent backups skip this step since packages are already present.

Application-Aware Backup — Job Workflow
How Commvault spawns multiple child jobs and what each one does

What You See in the Job Controller

When you trigger an Application-Aware Backup, a single job spawns multiple child jobs. Understanding this job hierarchy is important for troubleshooting and for explaining to customers what is happening.

Parent Job — VM Backup (e.g., Job 9212)
The main job triggered by your schedule or manual run. This is the only job you initiated. It orchestrates all other jobs. Note: the actual orchestrator job (e.g., 9213) is hidden by Commvault — you won't see it in the Job Controller. Only the visible parent and its children appear.
VM Admin Job
Child Job — Install Software (First Run Only)
Automatically triggered only on the first ever Application-Aware backup run for a given VM group. Commvault pushes File System Core, Media Agent binaries, MSSQL plugin, Storage Accelerator, and VSA-related packages into the guest virtual machine. On all subsequent backups, this job does NOT appear — the packages are already installed.
Install Software — First Run Only
Child Job — VSA App-Aware Workflow
The workflow job that manages the application-aware process. It goes through phases: pre-checks (verifying packages are installed and the application is discovered), application quiesce, VSS snapshot creation, and application backup coordination. You can see the detailed phases listed in the workflow job's status.
VSA App-Aware Workflow
Child Job — Snap Backup (IntelliSnap)
A snapshot backup of the application data (e.g., the SQL Server databases and their drives). Unlike the VM-level streaming backup, the application backup uses IntelliSnap — a snapshot-based technology. The snap engine used is "Virtual Server Application Snap" by default, though hardware snap engines (NetApp, Hitachi, IBM, HP, etc.) can also be configured.
IntelliSnap — Application Snap
📘 Two Parallel Things Happening

Application-Aware Backup does two things simultaneously: (1) a streaming VM backup of the entire virtual machine (OS disk, data disks) — this is the agentless VM-level backup you already know. (2) An IntelliSnap application backup specifically targeting the running application (e.g., SQL Server databases). After restore, you have both — a full VM restore option and a granular application restore option.

Automatic Package Discovery — What Gets Installed

Commvault automatically identifies the application type running in the guest VM during the first backup run. Based on what it discovers, it installs the correct plugins. You never have to specify "this VM has SQL Server" manually — Commvault detects it.

Discovered: SQL Server Commvault installs: Media Agent, MSSQL Plugin, File System, Storage Accelerator, VSA packages — all automatically Auto-Detected
Result in Client Tree After the first run, a new SQL Server client appears in the CommCell Console under the guest VM — created automatically by Commvault, no manual setup required Auto-Created
SQL Client Properties The auto-created SQL client uses IntelliSnap with snap engine "Virtual Server Application Snap" by default IntelliSnap Default
Storage Accelerator Also installed automatically — allows the guest VM (now a Commvault client) to send application backup data directly to cloud library, bypassing Media Agent Auto-Installed

Viewing Backups in the Storage Policy

After an Application-Aware backup completes, you can verify what was captured by checking the storage policy copies:

Primary Snap Copy Shows snapshot backups only. Right-click → List Snapshots to see VSS snapshots taken (e.g., C: drive snapshot for SQL Server hosted on C:) Snap Backups
Primary Copy Shows both the VM backup (streaming) and the SQL Server backup (IntelliSnap) as separate entries — two backups from one triggered job Both Captured

Application-Aware Restore — VM and Database
Two restore paths: full VM restore and granular application (database) restore

Restore Scenarios and Paths

🔴 Scenario A: VM Crashed Completely
  • First restore the full virtual machine (VM-level restore)
  • Once VM is up and running, verify application state
  • If application (SQL) is also corrupted, perform application restore next
  • Order: VM restore first, then application restore if needed
🟢 Scenario B: Only Application/Database Corrupted
  • VM is running fine, only the SQL database has an issue
  • Go directly to database restore — skip VM restore entirely
  • Navigate: Protect → Databases → Select instance → Browse and Restore
  • Select database(s) → In-place restore → overwrite existing

Database-Only Restore — Step-by-Step (SQL Server Example)

Restoring an Accidentally Deleted SQL Database
1
Confirm Scenario: Database Deleted, VM Still Running

In SQL Server Management Studio, verify that the target database (e.g., AdventureWorks) is missing. The VM is functional but the database is gone. A restore request comes to you.

2
Navigate to Commvault: Protect → Databases

In the Command Center or CommCell Console, go to Protect → Databases. Find the SQL Server instance that was auto-created by the Application-Aware backup. Select the specific database to restore.

3
Browse and Select Recovery Point

Click Browse and Restore. Select the latest available backup point (or a specific point in time). Choose the database(s) to restore — for example, the AdventureWorks database only.

4
Select In-Place Restore and Submit

Choose In-Place Restore — this restores to the same SQL instance and same database name, overwriting whatever is there. Submit the job. The restore will recreate the MDF (data file) and LDF (log file) from backup.

5
Handle Storage Accelerator Connectivity Issues (if any)

If the restore job pends or fails initially with a connectivity error, it may be because the Storage Accelerator is trying to connect the client directly to the library — and the guest VM does not have direct connectivity to the cloud library. In this case, disable the Storage Accelerator on the SQL client: go to Client Properties → Additional Settings and disable the Storage Accelerator. Re-run the restore.

6
Verify in SQL Management Studio

After the restore job completes, open SQL Server Management Studio, expand Databases → you should see the AdventureWorks database reappear. Run a query to confirm data integrity (e.g., SELECT TOP 10 * FROM HumanResources.Employee).

⚠ Storage Accelerator and Guest VM Connectivity

When Commvault installs packages in the guest VM during Application-Aware backup, it also installs the Storage Accelerator. By default, the Storage Accelerator will attempt to have the guest VM send restore data directly to the backup library — bypassing the Media Agent. If the guest VM does not have direct connectivity to the cloud library, the restore will fail or pend. In such cases, disable the Storage Accelerator on the SQL client's additional settings and let the Media Agent handle the data flow instead.

Azure Cloud Backup — Overview & Terminology
Azure storage concepts, lab architecture, and what Commvault protects in Azure

What Commvault Protects in Azure

This tutorial covers two Azure protection workloads, both requiring an access node deployed in the Azure cloud:

🗂 Azure Blob Storage (Object Storage)
  • Azure's object storage service — equivalent to AWS S3
  • Stores files, videos, images, unstructured data
  • Protected using the Cloud Apps agent
  • Commvault Virtual Client type: Azure Blob
  • Three authentication methods: Access Key, IAM AD, IAM VM Role
🖥 Azure Virtual Machines
  • Azure IaaS virtual machines (equivalent to AWS EC2)
  • Agentless backup using snapshot technology
  • Protected using Virtual Server Agent (VSA)
  • Commvault hypervisor type: Microsoft Azure
  • Three auth methods: Managed Identity, Azure App, Roles/Permissions

Azure Terminology — Key Concepts for Backup Admins

Subscription An Azure account with a billing boundary. Contains all Azure resources. Azure provides a free trial with $200 credit. Each subscription has a unique Subscription ID (UUID format — used in Commvault hypervisor config). Account Boundary
Resource Group A logical container for grouping related Azure resources (VMs, storage accounts, network interfaces). All resources in a group share the same lifecycle. Logical Container
Storage Account Azure's object storage service. The name must be globally unique. Contains Containers (≈ folders) and Blobs (≈ files). Two storage accounts are needed in a lab: one for customer data, one for backup library. Object Storage
Container A logical grouping within a Storage Account — equivalent to a folder or directory. Names must be lowercase, alphanumeric, hyphens allowed. No other special characters. ≈ Folder
Blob An individual object (file) stored inside a Container. "Blob" is the Azure term for a stored object — equivalent to a file in S3. ≈ File/Object
Access Tier Hot: frequent access, higher cost. Cool: infrequent access (recommended for backup repositories). Cold/Archive: rarely accessed, lowest cost. Cost Optimization
Access Keys Storage account authentication credentials. Two keys exist (Key1, Key2) — either can be used. The key provides full access to the storage account. Unlike AWS, there is no separate secret key — the key itself is the credential. Full Access
Network Security Group (NSG) Azure's firewall equivalent. Controls inbound and outbound traffic to/from VMs. By default, all inbound ports are blocked. Outbound traffic is allowed by default. Ports Blocked by Default
⭐ Azure Cost Management for Labs

Azure charges per hour of VM runtime. Even if you stop (not delete) a VM, the OS disk cost still accrues. Always delete VMs and storage accounts after completing your lab. A standard access node VM costs ~$0.13/hour; running for a full month costs ~$107. For lab VMs, select the smallest eligible size (B1S: 1 vCPU, 1 GB RAM = ~$15/month). Enable "Delete public IP when VM is deleted" to avoid orphaned public IP charges.

For Azure Blob Storage backup, Commvault has certified only specific operating systems as access nodes: Oracle Linux, Red Hat Enterprise Linux, and Windows Server. Azure Blob Storage Backup — Configuration
Cloud library setup, access node deployment, and virtual client creation for Azure Blob

Step 1: Create the Azure Cloud Library (Backup Destination)

Before backing up anything, you need a destination for the backup data. Create an Azure-based cloud library in Commvault using the backup Storage Account's access keys.

Getting the Storage Account Access Keys

In the Azure Portal: Storage Account → Security + Networking → Access Keys → Show Keys. Copy both the Storage Account Name and Key1 (or Key2). These are the credentials Commvault needs to write to the library.

⚠ Access Key Security

Azure Storage Access Keys provide full control over the storage account — read, write, delete all data. Treat them like a root password. Never share them in emails, tickets, or chat messages. If compromised, rotate (regenerate) them immediately in the Azure Portal.

Creating the Cloud Library in Commvault Command Center

Azure Cloud Library Configuration
1
Storage → Cloud → Add

In Command Center, navigate to Storage → Cloud → Add. By default, Azure Storage is listed — select it.

2
Provide Library Name, Media Agent, and Region

Give the library a descriptive name (e.g., Azure-Cloud-Lib). Select the CommServe or Media Agent. Choose the region matching where you created the backup Storage Account (e.g., South India).

3
Select Authentication: Access Key

Choose Account Name + Access Key as the authentication method. Create a new credential: give it a name, paste the Storage Account name, and paste the Access Key. Service Host defaults to blob.core.windows.net — leave as default.

4
Create and Specify a Container

A Container must exist inside the Storage Account before saving. Create one in the Azure Portal (e.g., cv-container, lowercase, no special characters). Paste the container name in Commvault. Save — the library should appear as Ready/Online.

Step 2: Deploy Access Node and Install Packages

The access node must be deployed as a VM in the Azure cloud. Packages are pushed from the CommServe to the access node using its Public IP (since CommServe is on-premises).

Required Ports — Azure NSG Inbound Rules

Port 22 — SSH (permanent) Port 8400 — Commvault (permanent) Port 8403 — Commvault (permanent)

In Azure, all inbound ports are blocked by default in the Network Security Group. You must explicitly add rules for ports 8400 and 8403. Port 22 (SSH) is typically opened at VM creation time.

Required Packages on Oracle Linux Access Node

✓ File System ✓ File System Core ✓ Cloud Apps ✓ Storage Accelerator
⚠ Decoupled Installation — When CommServe Has No Direct Route to Access Node

In many setups, the CommServe (on-premises, private IP) cannot directly receive a callback from the Azure access node (cloud, public IP) because the CommServe has no public-facing IP or port. This is called a decoupled mode scenario. Symptoms: package installation fails during the registration phase — packages are installed but the access node cannot register with CommServe. Fix: in CommCell Console, right-click Client Computers → New Client → Unix File System → enter the access node's public IP and enable "Fetch the configuration information from the client already installed in decoupled mode". This pulls the configuration from the already-installed (but unregistered) client, completing the registration.

Azure Blob Virtual Client — Authentication Methods
Three ways to authenticate Commvault to Azure Blob Storage — Access Key, IAM AD, IAM VM Role

Three Authentication Methods for Azure Blob Virtual Client

Method 1 — Access Key Use the Storage Account name + Access Key. Simplest method. Keys provide full storage account access. Must be manually rotated when expired or compromised. Simplest but Less Secure
Method 2 — IAM AD (Azure AD Application) Register an Azure AD application, assign permissions, and use its Application ID, Tenant ID, and Client Secret in Commvault. Supports fine-grained RBAC. Enterprise/AAD
Method 3 — IAM VM Role Assignment Assign the Storage Blob Data Owner role to the access node VM using Managed Identity. No credentials to manage — authentication is identity-based. Recommended. Recommended

IAM VM Role Assignment — Configuring Access Node Identity for Blob Backup

Think of this method like delegating authority: you are telling Azure "this specific VM (the access node) has permission to read and write data in this specific Storage Account — without needing any username or password." It is analogous to assigning a role rather than sharing a key.

IAM VM Role Assignment — Step-by-Step Azure Portal Configuration
1
Enable System-Assigned Managed Identity on the Access Node VM

In the Azure Portal: Virtual Machines → Access Node VM → Identity → System Assigned → Status: On → Save. This creates a unique Azure identity for this VM, tied to its lifecycle. When the VM is deleted, the identity is automatically deleted too.

2
Assign "Storage Blob Data Owner" Role to the Access Node VM

Navigate to the Customer Data Storage Account (the one being backed up): Storage Account → Access Control (IAM) → Add Role Assignment. Search for and select Storage Blob Data Owner. In "Assign access to", choose Managed Identity. Select the access node VM from the list. Review and assign.

📘 What "Storage Blob Data Owner" Means

This role grants full read/write/delete access to the blob containers and data in the storage account. It is the minimum role needed for Commvault to back up and restore blob data. Without this, the access node cannot read the blobs to back them up.

3
Create the Blob Virtual Client in CommCell Console

In CommCell Console: Client Computers → New Client → Cloud Storage → Azure Blob. Set the client name (user-defined), select the access node, choose a storage plan, and go to the Connection Details tab. Leave the host URL as default (blob.core.windows.net). Set authentication to IAM VM Role Assignment. Enter the Storage Account name. Click OK.

4
Configure Subclient Content and Run Backup

Navigate to the new virtual client → Cloud Apps → Default Subclient → Properties → Content. Browse the storage account — all containers are listed. Select the specific container to back up (e.g., data). If no container is specified, everything in the storage account is backed up. Run an ad hoc backup.

Azure Blob Storage — In-Place Restore

Restoring a Deleted Container/Blob — In-Place
1
Simulate Data Loss

In the Azure Portal, delete the target container from the customer data Storage Account to simulate an accidental deletion scenario.

2
Browse and Restore from Backup

In CommCell Console: right-click the default subclient → Browse and Restore. Select the full backup point. Browse the backed-up containers and blobs. Select the deleted container/blob(s).

Adding VSA package to the existing access node and understanding hypervisor authentication methods

Reusing the Blob Access Node for VM Backup

The same access node VM used for Blob Storage backup can also be used for Azure VM backup, provided you install the additional VSA (Virtual Server Agent) package. In smaller environments, this single access node approach is practical and cost-effective.

⭐ Large vs. Small Environments

Large environments: Use separate dedicated access nodes — one for Azure Blob Storage backup and one for Azure VM backup. This provides isolation, better performance, and easier troubleshooting. Small environments: A single access node can serve both roles. Install both Cloud Apps and VSA packages on the same VM.

Already Installed File System, File System Core, Cloud Apps, Storage Accelerator (from Blob Storage setup) From Blob Setup

Azure VM Backup — Three Authentication Methods

Like Blob Storage backup, Azure VM backup supports three methods of authenticating Commvault to the Azure subscription. The authentication method for VM backup is independent of the one used for Blob Storage.

Method 1 — Managed Identity System-assigned managed identity + Custom Role with VM management permissions. No credentials stored — recommended by Microsoft and Commvault. Used in this lab demo. Recommended

Managed Identity + Custom Role — Azure VM Backup Auth
Creating a Commvault custom role in Azure and configuring Managed Identity for VM-level authentication

Why Custom Role vs. Built-In Roles?

Azure has built-in roles like Virtual Machine Contributor that grant broad permissions. A Custom Role allows you to define exactly the minimum permissions Commvault needs — no more, no less. This is important for:

  • Security audits: Your security team may question why a backup tool has Virtual Machine Contributor rights (which include things Commvault does not need). A custom role with only backup-required permissions passes audit scrutiny.
  • Principle of least privilege: Best practice in cloud security — grant only what is required.
  • Clarity: A role named "CV Backup" makes it immediately obvious what this role is for and who created it.
⭐ Always Use Commvault-Provided JSON for Custom Role

Commvault publishes a JSON permissions file in its documentation that defines the exact permissions required for Azure VM backup. Use this JSON as the basis for your custom role — do not use a generic AI-generated JSON, as it may include insufficient or overly broad permissions. The Commvault JSON has been tested and validated against all backup and restore operations.

Step-by-Step: Custom Role + Managed Identity Configuration

Azure VM Backup — Managed Identity + Custom Role Setup
1
Obtain the Subscription ID

In the Azure Portal, search for Subscriptions. Copy the Subscription ID (UUID format, e.g., xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx). This is different from the Subscription Name — do not confuse them. The Subscription ID is required in the Commvault JSON and for hypervisor configuration.

2
Create a Custom Role from Commvault's JSON

In the Azure Portal: Subscriptions → Select Subscription → Access Control (IAM) → Add → Add Custom Role → Start from scratch. Click Edit, delete the default template, and paste the Commvault-provided JSON. Replace "subscriptionId" placeholder with your actual Subscription ID. Save, then click Review + Create → Create.

JSON — Commvault Custom Role structure (abbreviated)
{ "Name": "CV Backup", "Description": "Lets you manage virtual machines, not access to them, and not the virtual network or storage account.", "AssignableScopes": [ "/subscriptions/YOUR-SUBSCRIPTION-ID" ], "permissions": [ { "actions": [ /* VM management actions required by Commvault */ ], "notActions": [], "dataActions": [], "notDataActions": [] } ] }
⚠ Subscription ID vs. Subscription Name

A common mistake: pasting the Subscription Name (human-readable text) instead of the Subscription ID (UUID format). This causes the role creation to fail with "subscription identifier is invalid." Always verify you are copying the ID column, not the Name column, from the Subscriptions page.

3
Assign the Custom Role to the Access Node's Managed Identity

In the Azure Portal: Subscriptions → Access Control (IAM) → Add Role Assignment → Select "CV Backup" role → Assign access to: Managed Identity → Select the Access Node VM. Assign at subscription scope — this means the role automatically applies to all VMs in the subscription, including any new VMs created later. You do NOT need to assign the role to each VM individually.

📘 Scope Inheritance in Azure

When you assign a role at the subscription level, all resources within that subscription automatically inherit the role assignment. This is why the CV Backup role appears on the Production VM even though you assigned it only to the Access Node at subscription scope — the Production VM is inside the same subscription and inherits the permission.

4
Configure Azure Hypervisor in Commvault Command Center

In Command Center: Protect → Virtualization → Add Hypervisor → Microsoft Azure → Next. Provide the Subscription ID. In credentials: select Connect using Managed Identity for Azure resources (do not enter any keys or tenant IDs). Select the Access Node VM. Click Next — Commvault will validate the connection.

💡 Guided Setup is Simpler

You can also use Command Center's Guided Setup → Configure → Microsoft Azure workflow instead of manually navigating to Protect → Virtualization. Both reach the same configuration screen — Guided Setup may be easier for first-time configuration.

5
Add VM Group — Select VMs to Protect

After successful hypervisor connection, Commvault prompts to add a VM Group. Click Browse — all VMs in the connected subscription are listed. Select only the production VM (not the access node — it is the proxy, not the workload). Assign a plan (e.g., "Azure VM Backup" plan pointing to the Azure cloud library). Give the VM Group a name and finish.

Azure VM — Backup and In-Place Restore
Running backups, validating agentless operation, and full VM in-place restore

Running the Azure VM Backup

With the VM Group configured, trigger a manual backup. The agentless nature can be confirmed before and after the backup.

Simulate Data Loss (Root Partition Deletion)

SSH into the production VM and delete root partition files to simulate corruption: cd / && rm -rf * (with appropriate warnings — this is destructive, lab only!). After deletion, sudo commands return "no such file or directory" — the VM is corrupted and unusable. A full VM restore is the only option.

2
Navigate to Restore: Virtualization → Select Hypervisor → VM → Restore

In Command Center: Protect → Virtualization → Select Azure hypervisor → Select the production VM → Restore. Select Full Instance restore type.

3
Select Restore Point and In-Place Options

Select the latest backup point. Choose In-Place restore. Enable Power On after Restore and Unconditionally Overwrite. Optionally specify the Access Node. The storage account field shown in the restore dialog refers to the staging storage — leave as the associated cloud library storage account.

4
What In-Place Restore Does in Azure

An Azure VM in-place restore deletes the existing VM and recreates it from the backup. This means the Public IP address will change after restore — always retrieve the new Public IP from the Azure Portal after an in-place restore before trying to SSH back in.

⚠ Public IP Changes After In-Place Restore

Because the VM is deleted and recreated during in-place restore, Azure assigns a new public IP address to the restored VM. Do not use the old IP to reconnect. After restore completion, go to the Azure Portal → Virtual Machines → Select the restored VM → copy the new Public IP from the Overview page.

5
Verify Successful Restore

SSH into the restored VM using the new Public IP. Run sudo ls — you should have sudo access again. Navigate to /usr/customer-data/ — the test files created before backup should be present. The VM is fully restored and operational.

⭐ Lab Cost Reminder

After completing Azure lab exercises, always stop or delete Azure VMs. Stop the VM if you plan to return; delete it if you are done. Even a stopped VM incurs OS disk charges. Storage accounts also incur charges based on stored data volume. For lab environments, delete all resources when done to avoid unexpected billing.

Interview & Exam — Scenario-Based Q&A
Real-world questions covering Application-Aware Backup, Azure Blob, and Azure VM protection
A customer has 200 VMware virtual machines, all running SQL Server. They want application-consistent backups but don't want to install SQL agents in every VM. What do you recommend, and what are the trade-offs?
Recommend Application-Aware Backup. By enabling this option in the VM Group properties (with guest OS credentials provided), Commvault automatically pushes the required plugins to each guest VM on the first backup run — no manual agent installation required. The backup captures both the VM snapshot (streaming) and the SQL Server in an application-consistent state using IntelliSnap. Trade-offs: (1) No transaction log backups — only full/incremental VM-level recovery is possible, so RPO is limited by your VM backup schedule. (2) No table-level granular restore — only full database or full VM restore. (3) No additional SQL agent license is consumed — only the VM license applies, which is a significant cost saving for 200 VMs. If the customer needs log-level backups or granular table-level recovery, then agent-based SQL Server backup would be the correct choice instead.
When an Application-Aware backup runs for the first time, you notice three jobs appearing in the Job Controller instead of one. What are they, and is this expected?
Yes, this is fully expected and by design. The three jobs are: (1) Parent VM backup job — the main orchestrator job you triggered. (2) Install Software child job — automatically pushes required packages (File System Core, Media Agent binaries, application plugins like MSSQL, Storage Accelerator) into the guest VM. This job appears only on the first run — never again after initial installation. (3) VSA App-Aware Workflow child job + Snap backup — handles the application-consistent snapshot using IntelliSnap. On all subsequent backup runs, only 2 jobs appear (parent VM backup and the snap/workflow job) since the Install Software step is skipped.
What is the difference between Agent-Based SQL backup and Application-Aware backup in terms of license consumption?
With Agent-Based backup, you install a Commvault SQL Server agent inside each VM. Each installed agent consumes an agent-based license. In a 200-VM environment, this means 200 SQL agent licenses on top of the VM licenses — significant additional cost. With Application-Aware backup, Commvault does not install a licensed SQL agent — it uses a lightweight plugin pushed automatically. Only the VM-level license is consumed. No additional agent license is deducted, even though SQL Server is being backed up in an application-consistent manner. This is one of the primary reasons customers with large VMware deployments prefer Application-Aware backup — it protects applications without consuming per-application agent licenses.
You are configuring Azure Blob Storage backup. The package installation on the Oracle Linux access node fails during the registration phase, even though the packages are installed successfully. What is the cause and how do you fix it?
This is a decoupled installation scenario. The CommServe (on-premises, private IP 192.168.x.x) cannot receive a direct callback from the Azure access node (cloud, public IP) because CommServe does not have a public-facing IP. The packages install fine on the access node, but the registration — where the access node registers with CommServe — fails because CommServe is unreachable from the public internet. Fix: after the failed installation, go to CommCell Console → right-click Client Computers → New Client → Unix File System → enter the access node's public IP as the hostname → enable "Fetch the configuration information from the client already installed in decoupled mode" → Finish. This tells CommServe to pull the registration details from the already-installed (but unregistered) access node, completing the connection. After this, push any missing packages (Cloud Apps, Storage Accelerator) via Add/Remove Software.
Why can't you use Ubuntu as an access node for Azure Blob Storage backup, and what OS should you use instead?
Ubuntu is not on Commvault's certified operating system list for Azure Blob Storage access nodes. Commvault has certified only Oracle Linux, Red Hat Enterprise Linux (RHEL), and Windows Server as supported access node operating systems for Azure storage account backup. This is a Commvault product certification decision — even though Ubuntu runs fine as a general Linux VM in Azure, Commvault's Cloud Apps agent for Azure Blob has only been tested and validated on these specific distributions. Always consult the Commvault documentation's system requirements before deploying an access node. The production VM being backed up has no such restriction — it can be any OS since the backup is agentless.
What is a Managed Identity in Azure, and why is it the recommended authentication method for Commvault Azure VM backup?
A Managed Identity is an automatically managed identity in Azure Active Directory assigned to an Azure resource (in this case, the access node VM). It allows the resource to authenticate to other Azure services without storing any credentials — no keys, no passwords, no client secrets that can expire or be compromised. For Commvault Azure VM backup, the access node VM's Managed Identity is assigned the custom "CV Backup" role at the subscription level. When Commvault connects using this method, the access node presents its managed identity token to Azure — Azure verifies the identity and grants access. It is recommended because: (1) No credentials to rotate or manage, (2) Tied to the VM lifecycle — deleted when the VM is deleted, (3) Satisfies enterprise security audits, (4) Supported by both Microsoft and Commvault as the preferred approach. In Commvault Command Center, this is configured by simply enabling "Connect using Managed Identity for Azure resources" — no tenant ID, application ID, or secrets need to be entered.
After an Azure VM in-place restore completes, a junior engineer tries to SSH into the restored VM using the same IP address as before and cannot connect. What is the likely cause?
During an Azure VM in-place restore, Commvault deletes the original VM and recreates it from the backup. When the VM is recreated, Azure assigns a new public IP address — the old IP no longer exists or points to the restored VM. The fix is to go to the Azure Portal → Virtual Machines → select the restored VM → copy the new Public IP from the Overview page → use this new IP to SSH. This is expected behavior and should be communicated to the team before initiating any in-place restore. If IP address stability is a requirement, consider using a Static public IP or Azure DNS instead of relying on a dynamic public IP.
A customer's Azure Blob Storage backup completes but when they browse and restore, they can see the container in the backup but the blob files inside appear empty. What might have happened?
This is a timing issue: the backup job ran before any files were uploaded to the container. Commvault backs up whatever is present in the container at the time the backup job runs. If the backup ran first and the files were uploaded afterward, the backup captures an empty container. The fix is: (1) Ensure files are uploaded to the container before initiating backup, or (2) Run a new full backup after the files are present, then restore from that backup. Going forward, this highlights the importance of backup scheduling — the backup window should be after expected data upload windows, not before. In production, incremental backups after an initial full backup would capture newly added blobs as they are created.
