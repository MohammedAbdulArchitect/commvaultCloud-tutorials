The Enterprise Backup Landscape

Why enterprise backup matters and who the major players are

Why Enterprise Backup?

Major banks, large healthcare organisations, and Fortune 500 companies have critical data that cannot be lost ‚Äî customer transactions, patient records, financial data. Their infrastructure spans on-premise data centres and cloud environments, subject to strict compliance and disaster recovery requirements.

Unlike personal backups (copying files to an external hard drive), enterprise backup tools must handle:

  • Petabytes of data across thousands of servers simultaneously
  • Automated, scheduled operations with minimal human intervention
  • Deduplication ‚Äî reducing storage requirements by 50‚Äì90% by eliminating redundant data
  • Complex recovery scenarios including granular file-level and application-level restores
üè¢ Real-World Example ‚Äî XYZ Bank

XYZ Bank operates globally with data centres in India, China, Europe, and the United States. Each region may have its own Commvault environment (CommCell), managing tens of thousands of servers and protecting petabytes of critical banking data. One hour of unplanned downtime could cost millions and permanently damage customer trust.

Commvault Architecture

The three-tier model ‚Äî the most important concept in every Commvault interview

üíº #1 Interview Question

Q: "What is the architecture of Commvault?" ‚Äî This is the most common opening question in any Commvault interview. You must be able to explain it clearly, confidently, and from memory.

The Three-Tier Architecture

Commvault is built on a three-tier architecture. Each tier has a distinct, non-overlapping role. Understanding this separation is key to understanding how the product scales and how to troubleshoot it.

Tier 1

CommServe (CommCell Manager)

Role: Master controller ‚Äî the brain of the operation.

  • Coordinates all backup and restore jobs
  • Stores metadata and job history in SQL Server database
  • Provides the management consoles (Command Center + CommCell Console)
  • Schedules jobs, manages licenses, users, and policies
  • Recommended OS: Windows Server
‚úàÔ∏è Think of it as the air traffic controller ‚Äî directing all operations but never touching the actual data.
Tier 2

Media Agent

Role: Data mover ‚Äî physically moves all backup data.

  • Transfers data streams from clients to storage
  • Performs deduplication, compression, and encryption
  • Manages disk libraries, cloud storage, and tape
  • Retrieves and rehydrates data during restores
  • Supported OS: Windows or Linux
üöö Think of it as the cargo truck ‚Äî physically moving data from source to destination.
Tier 3

Client (iDataAgent)

Role: Source of backup data ‚Äî the protected machine.

  • Any server or workstation requiring backup protection
  • Runs agent software specific to the data type
  • Sends data stream to the Media Agent during backup
  • Can be agent-based or agentless (VMs)
  • Examples: File servers, Oracle DB, SQL Server, VMs
üè≠ Think of it as the warehouse ‚Äî the source of all goods being shipped.

The CommCell Environment

Understanding the complete Commvault deployment unit

When you combine all three components ‚Äî CommServe + Media Agents + Clients + Storage ‚Äî the resulting environment is called a CommCell. This is the fundamental deployment unit of Commvault.

üî¢ CommCell Components at a Glance

  • CommServe ‚Äî exactly one per CommCell (single management point)
  • Media Agents ‚Äî 2 to 100+ depending on scale and throughput needs
  • Clients ‚Äî 100 to 50,000+ servers and workstations
  • Storage ‚Äî disk arrays, cloud buckets (AWS/Azure/GCP), tape libraries

üåê Storage Types Supported

  • Disk Library ‚Äî most common; local or NAS-attached disk on the Media Agent
  • Cloud Library ‚Äî Azure Blob, AWS S3, GCP Storage; elastic capacity
  • Tape Library ‚Äî preferred in high-security environments (government, compliance)

Deployment Scenarios

Single CommCell vs. Multi-CommCell for global organisations

Single CommCell Approach

One CommServe managing all global locations.

‚úÖ Advantages

  • Centralised management ‚Äî single console for all regions
  • Lower licensing costs (one CommServe license)
  • Unified reporting and administration

‚ö†Ô∏è Considerations

  • Single point of failure risk
  • WAN latency for geographically dispersed deployments
  • All regions dependent on one control point

Multiple CommCell Approach

A separate CommServe deployed per region or location.

‚úÖ Advantages

  • Regional independence and resilience
  • Better local performance ‚Äî no WAN dependency
  • Compliance with data sovereignty laws
  • Isolated failure domains ‚Äî one region's outage doesn't affect others

‚ö†Ô∏è Considerations

  • Higher licensing costs (multiple CommServe licenses)
  • Separate management consoles per region
  • Greater administrative overhead

  • CommCell 1 ‚Äî India: Managing Asia-Pacific backup operations
  • CommCell 2 ‚Äî China: Managing China HQ and regional offices
  • CommCell 3 ‚Äî Europe: Managing European data centres (GDPR compliance)
  • CommCell 4 ‚Äî USA: Managing North American operations

Each CommCell operates independently, potentially managing 50,000+ clients with hundreds of Media Agents in that region.

üí° Why Agentless Matters

Agentless backup for virtual machines means no Commvault agent is installed inside individual VMs. Instead, Commvault uses hypervisor APIs (Hyper-V VSS / VMware VADP) to snapshot and capture entire VMs. This means:

  • No per-VM agent to install, patch, or maintain ‚Äî critical at scale (hundreds of VMs)
  • Application-consistent backups through native hypervisor integration
  • Instant VM recovery and granular file restoration from the same backup
  • Once you understand Hyper-V agentless backup, VMware operates very similarly

Interview Q&A

Scenario-based answers for common Commvault architecture questions

What is the architecture of Commvault? Explain each component.
Commvault uses a three-tier architecture. The CommServe is the master controller ‚Äî it coordinates all backup and restore operations, stores metadata in a SQL Server database, and provides centralised management through the Command Center web console. The Media Agent is the data mover ‚Äî it physically transfers data from clients to storage, performs deduplication, compression, and encryption, and retrieves data during restores. The Client is the protected machine ‚Äî it runs an iDataAgent appropriate for its data type (file system, database, VM) and sends data to the Media Agent. Together, CommServe + Media Agents + Clients + Storage form a CommCell environment.
Does the CommServe handle any backup data directly?
No ‚Äî the CommServe never touches the actual backup data. Its sole responsibility is metadata management and job orchestration. All data movement happens directly between the Client and the Media Agent. This architectural decision is what allows CommCells to scale to tens of thousands of clients without the CommServe becoming a performance bottleneck.
What is a CommCell? How large can it get?
A CommCell is the complete Commvault deployment unit ‚Äî one CommServe managing all associated Media Agents, Clients, and Storage targets. In small environments, a CommCell might have 2 Media Agents and 100‚Äì500 clients. In large enterprise deployments, a single CommCell can have 50‚Äì100+ Media Agents and 10,000‚Äì50,000+ clients protecting petabytes of data. Very large organisations (global banks, telcos) often deploy multiple CommCells ‚Äî one per region ‚Äî and may use Commvault's CommCell Router or Orchestrate for unified reporting across all regions.
When would you recommend a multi-CommCell deployment vs. a single CommCell?
A single CommCell suits organisations where all infrastructure is geographically co-located or where centralised management is the priority ‚Äî it reduces licensing costs and simplifies administration. A multi-CommCell approach is better when: (1) data sovereignty or regulations require data to remain within a specific country, (2) the organisation has multiple large data centres and WAN latency would affect backup windows, or (3) fault isolation is required so one region's CommServe failure doesn't impact all other regions. The most common pattern in global enterprises is a CommCell per region.
What is agentless backup and when is it used in Commvault?
Agentless backup refers to protecting virtual machines without installing a Commvault iDataAgent inside the guest VM. For VMware, Commvault uses the VMware VADP (vStorage API for Data Protection) through a dedicated proxy server running the Virtual Server Agent (VSA). For Hyper-V, the VSA is installed on the Hyper-V node itself. The hypervisor API provides application-consistent snapshots. Agentless backup is preferred for VM environments because: it eliminates the overhead of managing hundreds of in-guest agents, supports instant VM recovery, and enables granular file-level restores from a single backup image.
Name the enterprise backup market leaders and explain why Commvault is the focus for L2 roles.
The Gartner Magic Quadrant leaders are Commvault, Veeam, Rubrik, Cohesity, and Veritas. Commvault is the most commonly required skill at the L2 level for several reasons: it has the broadest agent coverage (Oracle, SAP, Exchange, file systems, VMs, cloud), a 30+ year installed base in enterprise banking and healthcare, and the most complex configuration model ‚Äî meaning L2 engineers need deeper knowledge than with simpler products. Many large global enterprises have significant legacy Commvault deployments that require experienced administrators for day-to-day operations, incident management, and upgrades.

When asked to explain Commvault's architecture, structure your answer as: "Commvault uses a three-tier architecture. Tier 1 is the CommServe ‚Äî the master controller that orchestrates all jobs and stores metadata in SQL Server, but never moves data itself. Tier 2 is the Media Agent ‚Äî the data mover that transfers backup streams from clients to storage, performing deduplication and encryption. Tier 3 is the Client ‚Äî any server running a Commvault iDataAgent appropriate for the data type being protected. Together they form a CommCell, which scales from hundreds to tens of thousands of clients in enterprise environments."
