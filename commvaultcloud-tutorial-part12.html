CommServe Migration — Hardware Refresh Overview
Use cases, planning considerations, and why this is an offline-only process

What is CommServe Hardware Refresh?

CommServe hardware refresh (also called CommServe migration) is the process of moving the Commvault CommServe database and configuration from one physical or virtual server to another. It is an infrequent but critical operation that requires careful planning. While you may only perform it once in several years for a given customer, the impact of doing it incorrectly is severe — potentially losing all backup job records and configuration.

Common Use Cases

  • End-of-life hardware: The physical server hosting the CommServe is reaching end-of-support and cannot be extended further with CPU, RAM, or disk.
  • OS upgrade with no direct path: For example, CommServe is on Windows Server 2012. Microsoft says there is no direct upgrade path to Windows Server 2022. A new server with the target OS must be built and a hardware refresh performed.
  • Physical to virtual migration: Moving the CommServe from a physical server to a VM, or between cloud platforms.
  • Performance upgrade: Replacing an undersized server with a higher-spec one.
  • OS crash recovery: After a complete OS failure when the system cannot be recovered by the OS team — use CS Recovery Assistant to restore configuration from DR backup onto a rebuilt server.
  • Post-patch instability: After applying CommServe patches that cause unexpected behavior, Commvault Support may instruct recovery from the last known-good DR backup.
📘 Standalone vs LiveSync

This procedure applies to a standalone CommServe. For CommServe with LiveSync (high-availability standby), the core procedure is the same but with additional steps. LiveSync significantly reduces data loss risk during a CommServe crash — from potential hours of lost job history (standalone) down to minutes.

⚠ Local Drive DR Backup — Real-World Risk

In the lab demonstration, the DR backup was stored on a local drive (E:\DR) purely for convenience. In production this creates a critical risk: if the OS crashes and the server cannot be booted, the DR dump files are trapped on an inaccessible disk. Best practice: always configure a network share path for CommServe DR backup. If a network share is genuinely unavailable, upload DR dumps to Metallic Cloud or a cloud library as the next best option. Tape-based recovery is technically possible but very complex — it requires knowing which specific tape contains the latest DR backup.

🔵 Method 1 — Same IP & Hostname
  • New server will use the exact same IP and hostname as the old CommServe
  • At installation wizard: Select "Use Existing Database" when asked
  • Provide the path to the DR dump folder (SET_<number>) when prompted
  • CommServe installs and imports the database in a single operation
  • All clients reconnect automatically — no reconfiguration needed on any client
  • Most common scenario for pure hardware refresh (server identity stays the same)
🟣 Method 2 — Different IP or Hostname
  • New server will have a different IP address or hostname
  • At installation wizard: Select "Create New Database" to install cleanly
  • After installation completes, run the CS Recovery Assistant tool to import the old database configuration
  • Handles IP/hostname remapping of the old configuration to the new server identity
  • Used for: IP change, data centre migration, OS platform change to a new server name
  • Clients may need their CommServe IP updated after the migration
💡 CS Recovery Assistant — Used by Most Customers Regardless of Method

Even for Method 1 (same IP/hostname), many customers prefer to use the CS Recovery Assistant tool because it is a well-tested, documented recovery path. Commvault's official recommendation differentiates the two methods by IP/hostname change, but in practice the CS Recovery Assistant is used for both. The key principle: always confirm the old CommServe is fully decommissioned and powered off before running the Recovery Assistant on the new server — the tool explicitly requires you to confirm this before proceeding.

What is the CS Recovery Assistant?

The CommServe Recovery Assistant (CSRecoveryAssistant.exe) is a Commvault-provided tool located in the Base folder of the CommServe installation. It imports a DR backup dump into a freshly installed CommServe SQL database, restoring the complete environment — all clients, storage policies, plans, schedules, libraries, and job history.

CommServe Hardware Refresh — Recovery Walkthrough
1
Take a Final DR Backup on the Old CommServe

CommCell Console → Control Panel → CommServe DR Backup → Run Now. Wait for completion. Navigate to the DR path (e.g., E:\DR) and identify the latest SET_<timestamp> folder (e.g., SET_030). Copy this entire folder to an external server (network share, jump server) accessible from the new CommServe.

2
Shut Down and Decommission the Old CommServe

Stop all Commvault services on the old CommServe. Power it off or isolate it from the network. This is a mandatory prerequisite — two CommServe machines must never be active simultaneously with the same configuration, as it causes split-brain conflicts and data corruption. The Recovery Assistant explicitly asks you to confirm decommissioning before proceeding.

3
Install Commvault Software on the New Server

Run the Commvault installer on the new hardware. At the database setup screen, choose appropriately: Use Existing Database and provide the dump path (Method 1 — same IP), or Create New Database for a fresh install (Method 2 — different IP, then use Recovery Assistant afterwards). Complete the installation and verify Process Manager shows CommServe services running.

4
Open CS Recovery Assistant

Navigate to the Commvault Base folder on the new server. Locate CSRecoveryAssistant.exe. Right-click → Run as Administrator. The tool opens with two options: Staging or Recovery (Production). Select Perform Recovery.

5
Confirm Decommission — Type the Exact Phrase

The tool displays a statement that must be typed exactly as shown — including all punctuation, capitalisation, and spacing. It confirms that the original CommServe has been decommissioned and this machine will take over as production CommServe. Even a single extra period or missing character will prevent the wizard from advancing to the next page. Type carefully and proceed.

6
Provide the DR Dump Path

Enter the full path to the SET_<number> folder. Ensure the target drive has at least 2× the folder size in free space. The tool validates the path and extracts the database dump files.

7
Start Recovery — Wait for Completion

Click Start Recovery. The tool imports all CommServe database tables into the SQL Server instance. This restores the complete environment: all clients, storage policies, plans, schedules, libraries, and job history. Wait for the success message: "Recovery completed successfully."

8
Apply License, Start Services, Verify

Apply the license file when prompted (or later via CommCell Console). Start all Commvault services via the Process Manager. Open CommCell Console and Command Center — verify both load without errors. Proceed to post-migration verification (next section).

Three Use Cases for CS Recovery Assistant

Use CaseTriggerData Loss?Notes
Hardware Refresh Planned migration to new hardware or OS None — take a fresh DR backup immediately before migration Most common use case. Full planned procedure with maintenance window.
Post-Patch / Post-Change Recovery CommServe misbehaves after applying patches or configuration changes. Commvault Support recommends recovery from last known-good DR backup. Possible — jobs run after the last DR backup are lost from CommServe DB metadata. Actual backup data on media is unaffected. Always use the Repair option first (preserves caches and settings). Only proceed to CS Recovery Assistant if Repair does not resolve the issue.
OS Crash Recovery CommServe OS crashed, cannot be recovered. OS team reinstalls the OS. Yes — all job records between the last DR backup and the crash are lost from CommServe DB. Data on media unaffected. Data loss = time between last DR backup and crash (e.g., DR at 10 AM, crash at 6 PM = 8 hours of lost job records). LiveSync reduces this to minutes.
📘 Repair vs Manual Reinstall — Critical Difference

When CommServe behaves unexpectedly, Commvault Support often recommends the Repair option before recovery. Repair uninstalls and reinstalls CommServe packages while preserving: index cache, job results folder, MongoDB cache, and additional settings applied by Support. A manual uninstall + reinstall does NOT preserve these — they are wiped. Always follow Support's recommendation on which path to take.

DDB Re-synchronization — When and Why

The Deduplication Database (DDB) is hosted on a Media Agent and maintains its own records of deduplicated block fingerprints independent of the CommServe SQL database. After CommServe recovery from an older DR backup, the two databases can become out of sync, causing DDB partitions to go Offline or enter Maintenance Mode.

🔍 Understanding the Out-of-Sync Scenario

Scenario: DR backup taken at 10 AM. OS crashes at 6 PM. CommServe recovered at 7 PM using the 10 AM dump. Result: CommServe SQL database only knows about jobs up to 10 AM. The Media Agent DDB has job records up to 6 PM — 8 hours of additional backup activity. The two are now out of sync → DDB partition shows Offline / Maintenance Mode. Fix: DDB Re-synchronization, which reconciles the CommServe database with the DDB state on the media agent.

How to Trigger Re-sync CommCell Console → Storage Resources → Deduplication Engines → right-click the DDB store that is offline. ⭐ DDB Best Practice — Keep DDB on Media Agent, Not CommServe

Always host DDB partitions on a dedicated Media Agent, not on the CommServe machine. Hosting DDB on the CommServe means a CommServe failure simultaneously takes down both CommServe metadata and DDB — making re-synchronization impossible and recovery far more complex. Disk library mount paths should ideally be on the same media agent as the DDB for that storage pool.

SAP HANA — Overview & Commvault Integration
In-memory database architecture, multi-tenant containers, Backint interface, and key terminology

What is SAP HANA?

SAP HANA (High-Performance Analytic Appliance) is an enterprise in-memory database platform designed for processing massive volumes of real-time data. Unlike traditional disk-based databases, HANA stores and processes data in RAM, enabling extremely fast analytics and transactional processing. Due to its high licensing and hardware cost, it is primarily deployed by large enterprises.

Commvault integrates with SAP HANA via the Backint interface — a SAP-defined API that allows third-party backup tools to communicate with HANA and perform backup and restore operations. This integration is analogous to Oracle RMAN for Oracle databases — not identical, but a similar concept of a vendor-defined interface for backup integration.

🎯 Why Use Commvault for SAP HANA Backup?

SAP HANA has its own native backup capability. Customers use Commvault instead (or in addition) to get: unified backup management and reporting across all workloads (SAP HANA, VMs, file servers), cross-system restore capabilities, enterprise-grade retention and compliance policies, and integration with their existing Commvault infrastructure. The HANA admin retains control and can also trigger ad-hoc backups — those jobs appear in the Commvault Job Controller automatically.

📘 Who Does What — Admin Responsibilities

The SAP HANA database server is managed by the SAP HANA admin. The backup admin installs the Commvault agent and configures the backup/restore policies. The HANA admin provides to the backup admin: OS user credentials (hxeadm), HANA system user DB credentials, HDB SQL binary path, and (for keystore auth) the keystore key name. The backup admin does not control or administer the HANA database itself.

SAP HANA Lab Setup & Commvault Agent Installation
Deploying HANA Express on GCP and pushing the SAP HANA agent from Command Center

Lab Environment — Why GCP?

SAP HANA requires a minimum of 32 GB RAM because it is an in-memory database. This makes local laptop deployment impractical. The recommended lab approach: deploy a GCP VM from the Marketplace using a SUSE Linux 15 for SAP image, which includes most SAP prerequisites pre-installed.

Supported Operating Systems SAP HANA 2.0 is supported on SUSE Linux Enterprise Server (SLES) 15 and Red Hat Enterprise Linux (RHEL) 8/9. Not supported on Ubuntu, Debian, or other distributions. SLES 15 or RHEL Only

⚠ HANA Commands are Case-Sensitive

All Linux commands and HANA-specific commands (HDB, HDB info, hdblcm) are case-sensitive. HDB is uppercase; hdblcm is lowercase. A wrong case gives "command not found" errors. The HANA command prompt is also case-sensitive for SQL statements.

🔵 User Credentials — Simpler
  • Provide the HANA DB username (SYSTEM) and password directly in Commvault
  • Commvault stores credentials in the Credential Vault (encrypted)
  • Simpler to configure — no keystore preparation needed
  • Suitable for lab and non-sensitive environments

🟣 HDB Keystore — More Secure (Recommended for Production)
  • HANA admin creates an HDB user keystore entry on the HANA server using hdbuserstore SET
  • Keystore maps a key name to credentials (username, password, host, port)
  • Backup admin provides only the key name to Commvault — no raw password
  • No password stored in Commvault configuration
  • Recommended for all production environments
  • HANA admin must create the keystore before Commvault can be configured with this method
📘 Information Required from HANA Admin Before Configuration

Regardless of authentication method, the backup admin needs the following from the HANA admin: (1) HANA OS admin username — format <SID>adm, e.g., hxeadm. (2) HANA SID — three-character system ID (e.g., HXE). (3) HANA DB credentials — DB username (SYSTEM) and password for User Credentials method, or key name for Keystore method. (4) HDB SQL binary path — run find /usr/sap -name "hdbsql" 2>/dev/null on the HANA server to locate it. This path is required by Commvault regardless of authentication method.

📘 Keystore User Must Be in System DB, Not Tenant DB

A common mistake when creating the HDB keystore user: creating it in the tenant DB (HXE) instead of System DB (SYSTEMDB). The Commvault SAP HANA integration requires the backup user to exist in the System DB with appropriate privileges. Creating the user in the tenant DB only results in authentication failures when Commvault tries to connect to System DB for backup coordination. Always connect to SYSTEMDB when running the CREATE USER and GRANT commands.

One Touch Recovery — Overview & Concepts
Bare-metal recovery for physical servers, terminology, and backup consistency types

What is One Touch Recovery?

One Touch Recovery (OTR) is Commvault's feature for complete backup and recovery of physical servers — including the operating system, drivers, installed applications, configuration files, registry (Windows), and all file system data. It is Commvault's term for what other vendors call Bare Metal Recovery (BMR) or System State Backup.

🎯 Why One Touch Recovery Is Critical

Without OTR, if a physical server crashes the server team must: reinstall the OS from scratch, reinstall all applications and databases, reconfigure all settings (many undocumented), restore data files from backup. This is extremely time-consuming. With One Touch Recovery, the entire server — OS + applications + data — is restored in one operation, bringing it back to exactly the state it was at backup time.

Vendor Terminology

Commvault One Touch Recovery — superset of both System State backup (Windows terminology) and Bare Metal Recovery. This is the official Commvault term. One Touch Recovery
Veritas NetBackup Bare Metal Recovery (BMR) — equivalent feature, different product name. Same concept of full physical server restore. BMR
Microsoft / Windows System State Backup — Windows-specific term. Also used as the toggle label in Commvault's subclient content tab for Windows OTR. System State

Key Constraints — What OTR Does and Does Not Cover

  • Physical servers only — designed for bare-metal physical hardware. Not recommended for virtual machines (VMs use hypervisor-based snapshot/image backup instead).
  • Hypervisors NOT supported — ESXi, Hyper-V, OpenStack hypervisor layer cannot be protected by OTR. You cannot install a Commvault agent on a hypervisor OS layer directly.
  • File System agent required — unlike agentless VM backup, OTR requires the Commvault File System agent installed on the physical server.
  • Restore requires console access to the client — unlike VM backup restore (triggered entirely from CommCell Console), OTR restore requires direct console access to the physical server to mount the Live CD and boot from it. This is the key operational difference.
  • Supported OS (Linux): Red Hat, SUSE, Ubuntu, Oracle Linux, Rocky Linux, AlmaLinux. AIX and Solaris also supported (critical for banking customers on legacy physical hardware).
  • Supported OS (Windows): Most Windows Server versions. WinPE 10 for modern hardware; WinPE 5.1 for older hardware.

Recommended answer: "Ask the customer what application is running in the VM. If it is an I/O-intensive database — Oracle, SAP HANA, SQL Server, DB2 — recommend an agent-based backup. Enabling One Touch Recovery — Windows Configuration

Enabling OTR for a Windows physical server requires a single checkbox change in the file system subclient. Prerequisites: the server is already configured as a Commvault client with the Windows File System agent installed.

Windows One Touch Backup — Configuration Steps
1
Ensure File System Agent is Installed

CommServe Migration Best Practices

DR backup on a network share Always configure CommServe DR backup to write to a network share, not a local drive. Local drive = inaccessible if OS crashes. Network share, Metallic Cloud, or cloud library are the acceptable alternatives. Network Share Only
Take a fresh DR backup before any migration Do not rely on the scheduled backup. Trigger a manual DR backup immediately before migration and copy the SET folder off the server before proceeding. Manual DR Before Migration
Repair before CS Recovery Assistant For CommServe instability, try the Repair option first — it preserves index cache, job results folder, MongoDB cache, and additional settings. Only escalate to CS Recovery Assistant if Repair does not resolve the issue. Repair First
Use Staging mode for DR testing Periodically use CS Recovery Assistant in Staging mode (separate server, different IP) to validate DR backup integrity and test the recovery process. Never discover issues with your DR process during a real disaster. Test DR Regularly
Keep DDB on Media Agent, not CommServe DDB partitions should always be on a dedicated Media Agent, not on the CommServe machine. Co-locating CommServe and DDB means a CommServe failure takes down both simultaneously, making recovery harder. Separate DDB from CommServe

SAP HANA Backup Best Practices

Use HDB Keystore in production Use the HDB Keystore authentication method in all production environments. User Credentials method is acceptable only in lab/development. Keystore avoids raw passwords in the CommServe configuration and is the recommended security posture. Keystore for Production

Do not kill HANA admin command-line jobs HANA backup jobs appearing in Job Controller that you did not initiate are legitimate operations by the HANA admin (scheduled or ad-hoc HDB SQL backups). Always confirm with the HANA team before terminating any HANA-related jobs. Coordinate with HANA Admin

Interview Q&A
Scenario-based questions on CommServe migration, SAP HANA, and One Touch Recovery

CommServe Migration Questions

A customer needs to upgrade their CommServe from Windows Server 2012 to Windows Server 2022. Microsoft says there is no direct in-place upgrade path. What do you recommend?
Perform a CommServe hardware refresh. Steps: (1) Verify Windows Server 2022 is in Commvault's supported OS list for the current CommServe version. (2) Build a new server with Windows Server 2022 and the required disk/RAM/IOPS configuration. (3) Take a fresh DR backup on the existing CommServe and copy the SET folder off the server to a network share. (4) Shut down the old CommServe. (5) Install Commvault on the new server — using "Use Existing Database" if the IP/hostname stays the same, or "Create New Database" followed by the CS Recovery Assistant if the IP/hostname changes. (6) After recovery, verify services, clients, media agents, libraries, and DDB partitions. (7) Run a test backup to confirm end-to-end functionality.
A CommServe OS has crashed and cannot be recovered. The last scheduled DR backup ran at 8 AM. It is now 5 PM. If we recover using the 8 AM DR backup, what data is lost and what is preserved?
The CommServe SQL database (CommServe metadata) is restored to its state at 8 AM. This means: all backup job records from 8 AM to 5 PM are lost from the CommServe database — the CommServe has no record of those 9 hours of jobs. However, the actual backup data written to media (disk libraries, cloud libraries, tape) during that 9-hour window is NOT deleted — it still physically exists on the media. The problem is the CommServe no longer has the index/catalog records to browse and restore that data. If the Media Agents are still running and the DDB is not out of sync, jobs can sometimes be re-catalogued. For the DDB — if media agents ran jobs during those 9 hours, the DDB will be out of sync with the restored CommServe DB, and DDB re-synchronization will be needed to bring DDB partitions back online.
What is the difference between the Staging and Recovery modes in the CS Recovery Assistant? When would you use each?
Staging mode imports a DR backup database onto a separate machine (different IP/hostname) to create a test or staging environment that mirrors the production CommServe. It does NOT affect production — the staged CommServe is used for testing, demo, or validation only. Recovery (Production) mode is used when a machine is taking over as the actual production CommServe — it replaces the old CommServe permanently. Never use Recovery mode just for testing because it sets the machine as the production CommServe and the tool requires you to confirm the old CommServe has been decommissioned. Best practice: use Staging mode regularly to test DR backup integrity and practice the recovery procedure before a real disaster forces you to use it.
After a CommServe recovery from a DR backup, some DDB partitions are showing as Offline in Maintenance Mode. What is the cause and how do you fix it?
The cause is a timing mismatch between the CommServe database and the DDB on the Media Agent. The CommServe was recovered from a DR backup taken at a specific point in time (e.g., 10 AM). The Media Agent DDB continued running and accumulated job records after that time (say, up to 6 PM when the crash occurred). When the CommServe came back with the 10 AM database, it no longer has records of jobs 10 AM to 6 PM — but the DDB has those records. This inconsistency triggers the DDB to go into Maintenance Mode. Fix: CommCell Console → Storage Resources → Deduplication Engines → right-click the offline DDB store → select Re-synchronize. This reconciles the CommServe database with the DDB state. The option appears only when inconsistency is detected — it may be hidden or greyed out when DDB and CommServe are in sync.

SAP HANA Questions

A customer is asking whether they should use User Credentials or HDB Keystore for SAP HANA authentication in Commvault. What do you recommend and why?
For any production environment, recommend HDB Keystore. Reasons: (1) No raw password is stored in the Commvault configuration — the keystore on the HANA server holds the credentials and Commvault only knows the key name. (2) If the HANA admin changes the database password, they update the keystore entry on the HANA server without requiring any changes to the Commvault configuration (as long as the key name stays the same). (3) It follows the principle of least privilege and separation of duties — the backup admin never needs to know the HANA DB password. User Credentials method is acceptable for lab and dev environments where security requirements are relaxed and simplicity is prioritised.
You see SAP HANA backup jobs appearing in the Commvault Job Controller that you did not schedule. What do they indicate and what should you do?
These are legitimate backup operations triggered by the HANA admin from the HANA server command line using hdbsql with the BACKUP DATA USING BACKINT command. Commvault's Backint interface captures all HANA backup events regardless of their source, so they appear in Job Controller. This is expected and correct behavior — it means Commvault is properly integrated with HANA. Do NOT kill these jobs without first confirming with the HANA admin team. They may be scheduled by the HANA admin, ad-hoc emergency backups, or part of a maintenance procedure. If they run regularly and unexpectedly, ask the HANA admin to document their backup schedule so you can distinguish planned jobs from anomalies.
During SAP HANA configuration in Command Center, the HANA host does not appear in the dropdown. What are the possible causes and how do you resolve them?
The most common cause is that the Commvault agent on the HANA server is not in a Ready state — either the service is not running, the external IP changed after a VM restart, or there is a network connectivity issue on ports 8400/8403. Resolution steps: (1) In CommCell Console, go to the HANA client properties and check the registered IP address. If the VM was restarted and the IP changed, update it to the new IP. (2) Run Check Readiness — this actively tests connectivity and reports why it may be failing. (3) On the HANA server, run commvault status to verify services are running. (4) Once Check Readiness shows Ready, go back to Command Center and the host will appear in the dropdown. Always run Check Readiness before attempting any database configuration.
A HANA admin has run command-line backups using HDB SQL but the jobs fail with authentication errors. What is the likely cause?
The most likely cause is an incorrect syntax — specifically using BACKUP DATA TO FILES instead of BACKUP DATA USING BACKINT. The TO FILES syntax writes to a local path and bypasses Commvault entirely, causing the backup to fail when it cannot find the Backint handler. The correct syntax for Commvault-integrated command-line backup is: BACKUP DATA USING BACKINT (PREFIX 'COMPLETE_DATA_BACKUP');. Another possible cause: special characters in the HANA system user password not being properly escaped — for example a $ in the password must be escaped with a backslash (\$) or the entire password must be wrapped in double quotes when used in the HDB SQL command.

One Touch Recovery Questions

A customer asks if they can configure One Touch Recovery for their virtual machines to get the same protection as their physical servers. How do you respond?
One Touch Recovery is designed specifically for physical servers and is not the right approach for virtual machines. For VMs, the correct solution is hypervisor-based agentless image backup (VSA backup via VMware/Hyper-V snapshots), which gives similar full-server protection with important advantages: the restore is triggered entirely from CommCell Console or Command Center with no need for console access to the VM, no Live CD is required, restore is faster, and backup is agentless (no agent inside each VM). OTR for VMs has a key practical problem: restore requires direct console access to the VM (to mount the bootable recovery media and go through the recovery wizard), which is operationally more complex than simply triggering a restore from the central management console. For VMs without databases, use agentless image backup. For VMs with I/O-intensive databases (Oracle, SAP HANA), add the appropriate database agent inside the VM for application-consistent backup.
What are the six stages of the Linux One Touch Recovery wizard? Why is knowing these stages important?
The six stages are: (1) Network Configuration — assign IP address, netmask, gateway, and DNS to the server being recovered so it can communicate with CommServe. (2) Client Configuration — provide the Commvault client name (hostname) and confirm the machine identity. (3) CommServe Configuration — provide CommServe IP/hostname and CommCell Console login credentials. (4) Firewall Configuration — optional; provide firewall details if the client is behind a firewall relative to the CommServe. (5) Silent Install of Commvault Packages — Commvault agent packages are installed into the temporary Live CD environment (requires 8 GB RAM minimum). (6) Job Selection and Recovery — select the backup job to restore from; confirm disk formatting; data is restored from CommServe backup storage; disk layout and network persistence are configured. Knowing these stages demonstrates you understand the complete recovery workflow and can guide a server team through the process under pressure during a real disaster.
A physical Linux server has crashed. The HANA team says they need it restored urgently from the Commvault One Touch backup taken yesterday. Walk through what you would do.
Step 1: Confirm the OTR backup from yesterday is available and not expired — check in CommCell Console that the backup job exists and its retention has not elapsed. Step 2: Download the latest Linux Live CD ISO from store.commvault.com if not already available. Step 3: Coordinate with the server team to get ILOM/iDRAC console access to the crashed server. Step 4: Mount the Linux Live CD ISO as a virtual CD/DVD drive in the ILOM/iDRAC. Step 5: Change the BIOS boot order to CD-ROM first (F2 at POST → Boot section). Step 6: Boot from the Live CD — at the boot prompt, type default and press Enter. Step 7: Walk through all six stages of the wizard — providing the server's original IP (Stage 1), client name (Stage 2), CommServe IP and credentials (Stage 3), skipping firewall if not applicable (Stage 4), waiting for package install (Stage 5), selecting yesterday's backup job and confirming disk format (Stage 6). Step 8: Wait for restore completion. Step 9: Reboot, remove Live CD ISO, and the server boots from the restored hard disk. Step 10: Verify OS, applications, and data are intact. Notify the HANA team the server is available. The entire process is coordinated with the server team since they manage the physical server and ILOM access.
What is the difference between Crash Consistent, Application Consistent, and Agent-Based backup? When would you recommend each?
Crash Consistent: A hypervisor snapshot taken via API, same as a manual snapshot. The file system is consistent post-restore (no chkdsk needed), but in-memory application state is not captured. Suitable for VMs without I/O-intensive databases — web servers, lightweight apps, file shares. Application Consistent: A lightweight plugin pushed by Commvault into the VM that quiesces the application before the snapshot, then thaws after. Both file system and application are consistent post-restore. Suitable for VMs running supported applications (e.g., PostgreSQL) where full agent install is not feasible. Agent-Based: A full Commvault database agent installed inside the VM or on the physical server, communicating directly with the application via its native API (RMAN for Oracle, VDI for SQL Server, Backint for SAP HANA). Provides granular backup, point-in-time recovery, and guaranteed application consistency. Mandatory for I/O-intensive databases: Oracle, SAP HANA, SQL Server, DB2. Recommendation guideline: ask the customer what application is running in the VM. If it is a production database — recommend agent-based. If it is a lightweight application — agentless is sufficient. Always document the customer's decision if they decline an agent for a database VM.
